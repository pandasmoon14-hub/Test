# Four-Donor Promotions v0.1

Purpose: authoritative cross-donor promotion ledger after comparison of:
- RHBF
- Shadowrun
- Lancer
- Starfinder

This file records what is now strong enough to promote into Astra-facing consolidation, what should remain as optional recurring profiles, and what should still not be promoted.

---

## Promote now as Astra baseline grammar

### Batch A

1. **Layered build grammar**
   Astra must support multiple lawful build owners rather than one sovereign package layer.

2. **Path as one lawful build owner, not the sole owner**
   Paths remain valid and important, but later donors prove that themes/background-like identity, overlays, access states, licenses, operator/platform splits, and resource-backed identity layers must coexist.

3. **Theme/background-like identity as a real build layer**
   Theme/background-like identity is now strong enough to promote clearly as a stable build owner.

4. **Access law as real doctrine**
   Domain access, license-backed access, special channels, permissions, credentials, and gated acquisition all validate access as a distinct and durable Astra rules layer.

5. **Capability doctrine distinct from codified active expression**
   Competence, triggers, skills, and broad capability statements must remain distinct from Techniques, spells, class features, or comparable active objects.

6. **Ranked specialization overlays**
   Ranked overlays are now stable grammar.

7. **Contacts/resources as lawful build pressure**
   Social and material starting pressure can be part of a build without collapsing into later world-procedure doctrine.

8. **Operator / platform separation**
   This is now clearly baseline Astra grammar.

### Batch B

9. **World-operation loop grammar**
   Astra must support structured operation cycles without universalizing one donor cadence.

10. **Contract / opportunity structures**
    Multiple donors now prove that opportunity structures are a stable owner, whether they appear as faction jobs, brokered work, missions, expeditions, or exploratory claims.

11. **Preparation / legwork / setup procedure**
    Preparation is a real procedure family, not filler.

12. **Faction pressure and leverage logic**
    Factions and institutions are stable procedural pressure owners across local, civic, exploratory, and interstellar scales.

13. **Conditional / technical / authorization-linked access procedure**
    Access may depend on institutional, technical, situational, credential, or authorization-driven conditions.

14. **Remote / delegated / crewed operation procedure**
    Astra should preserve operator action routed through delegated assets, support entities, or shared platforms.

15. **Standing / scrutiny / trace separation**
    Consequence layers should not collapse into one undifferentiated reputation number.

16. **Tactical shell doctrine**
    Structured tactical objective procedure is now stable grammar.

17. **Downtime as structured bridge**
    Between-operation procedure is a real procedural layer when donor pressure supports it.

18. **Broadened Operational Anchor doctrine**
    Astra operational anchors may include cities, capitals, strongholds, ships, stations, ports, frontier nodes, and exploratory bases.

19. **Exploration as structured campaign engine**
    Exploration/site-chain/world-chain procedure is now clearly a lawful campaign engine.

20. **Operational nodes at multiple scales**
    Stations, ports, colonies, frontier worlds, settlements, and outposts are real procedural nodes.

21. **Travel-access interplay**
    Travel is a real procedure family shaped by access, danger, and infrastructure.

### Batch C

22. **Classification/category/family as mechanical field**
    Classification must do real work in Astra.

23. **Multiple content ecologies**
    Astra must support organic, manufactured, licensed, bonded, salvage, magical, technological, hybrid, and mixed ecologies.

24. **Broad resource-source ontology**
    Resource sources are broader than slain actors.

25. **Platform record grammar**
    Platform-side records are now baseline doctrine.

26. **Control compatibility / operation handoff**
    Batch C must preserve compatibility fields while leaving live procedure in Batch B.

27. **Host-cost profile for installed modification**
    Installed-modification cost/support pressure is now baseline, though exact implementations remain open.

28. **Deployable / support-entity middle zone**
    Astra should preserve an intermediate zone between ordinary gear, subsystems, and full actors.

29. **Bound-growth object grammar**
    Bound-growth objects remain Astra-stable.

30. **Natural Treasure as progression-significant class**
    Natural Treasure remains a valid Astra-facing resource class.

31. **Starship as a major platform family**
    Starships are now clearly major platform-side schema owners.

32. **Vehicle-family schema breadth**
    Platform doctrine must remain broader than mechs only and broader than starships only.

33. **Mixed object ecology**
    Science-fantasy mixed item ecologies are now clearly baseline-supportable.

34. **Rated system/computer object families**
    Objects whose significance lies in system/rating/state logic now have strong donor support.

35. **Broadened entity schema**
    Entity doctrine must survive organic, sapient, technological, platform-adjacent, and science-fantasy mixed threats.
