# Astra Mixed Object Ecology v0.1

Derived from cross-donor consolidation after RHBF, Shadowrun, Lancer, and Starfinder.

## Scope of this file

This file owns Astra's **mixed object ecology doctrine**.

It defines:
- what an object ecology is,
- how multiple object-source and object-function families may coexist,
- how magical, technological, hybrid, bonded, installed, harvested, rated, and platform-side objects should remain distinguishable,
- how source family differs from usage family, host family, and progression significance,
- what coexistence rules Astra currently recognizes,
- and what remains optional-profile support or donor-local retention.

It does **not** own:
- full item catalogs,
- exact rarity tables,
- exact donor chapter taxonomies,
- exact economic price curves,
- exact crafting recipes,
- exact augmentation slot charts,
- exact relic growth math,
- exact starship component lists,
- or exact subsystem numbers.

This is a doctrine owner for cross-family object coexistence, not a universal item list.

---

## Why this file exists

The donor set now proves that Astra cannot safely assume one dominant object-source model.

RHBF proves that Astra must support:
- natural treasures,
- harvested components,
- wondrous or relic-bearing objects,
- bound-growth object families,
- and progression-significant materials.

Shadowrun proves that Astra must also support:
- installed modifications,
- technical devices,
- rated systems,
- regulated or credential-shaped objects,
- and manufactured object ecologies with compatibility and host-cost pressure.

Lancer proves that Astra must support:
- chassis/platform-side components,
- mount-bearing and install-bearing distinctions,
- deployables and support entities,
- and manufactured combat/support ecologies that are not reducible to ordinary gear.

Starfinder then proves that Astra must also support:
- technological items,
- magical items,
- hybrid items,
- augmentations,
- vehicles and starships,
- rated systems/computers,
- and broader science-fantasy mixed item ecologies.

The stable conclusion is:

> Astra needs one doctrine owner that explains how unlike object families can coexist without being flattened into one generic “item” mass or one donor-shaped chapter order.

## Core ecology law

An **object ecology** is the structured relationship between multiple object families that coexist in the same rules environment.

An object ecology is mixed when the same doctrine layer must host several distinct object families whose:
- source logic,
- host logic,
- acquisition logic,
- progression significance,
- and operational roles

do not collapse cleanly into one category.

The stable rule is:

> Astra should classify objects by what they are doing doctrinally, not only by what donor chapter they appeared in.

This means Astra must be able to distinguish, without confusion, between:
- source family,
- host relationship,
- access relationship,
- progression significance,
- platform relationship,
- and scene/operation role.

## Mixed ecology is not the same as other doctrine owners

### Mixed ecology is not one item catalog
This file does not list everything that exists.
It explains how unlike things are allowed to coexist.

### Mixed ecology is not platform grammar
Platform doctrine owns platform-side records and install-bearing object families.
This file explains how platform-side families coexist with other object ecologies.

### Mixed ecology is not bound-growth doctrine
Bound-growth objects remain their own family.
This file explains how that family coexists with installed, rated, magical, hybrid, or platform-side objects.

### Mixed ecology is not economy doctrine
Objects may have value, scarcity, legality, or market pressure.
This file does not define price law or full economy logic.

### Mixed ecology is not access law
Some objects matter because they shape access.
That does not mean all object ecology becomes access doctrine.

## Core classification axes

Astra should not classify mixed object ecologies on only one axis.

The current cross-donor-stable axes are:

### 1. Source family
Where the object’s logic primarily comes from.

Common source families currently supported:
- natural/harvested
- crafted/manufactured
- magical
- technological
- hybrid
- bonded/attuned
- credential/authorization-bearing
- platform-derived
- anomalous
- mixed-origin

### 2. Host family
What kind of thing the object belongs to or integrates with.

Common host families currently supported:
- carried personal use
- body-facing installed
- platform-facing installed
- route/infrastructure-facing
- bonded bearer-linked
- free object / independent asset
- support-entity linked
- cargo/containerized
- environmental/site-linked

### 3. Operational role
What the object actually does in play.

Common operational roles currently supported:
- offensive or active projection
- defensive/supportive protection
- mobility/travel support
- access mediation
- rated/system mediation
- progression-significant growth support
- salvage/resource output
- scene support / deployable support
- logistics/storage/cargo support
- ritual/research/interface support

### 4. Progression significance
How much the object matters for growth or advancement.

Some objects are:
- ordinary utility,
- support-significant,
- build-shaping,
- progression-significant,
- or identity-defining.

### 5. Persistence state
Whether the object is:
- consumable,
- temporary,
- replaceable,
- upgradeable,
- bonded,
- persistent,
- or campaign-anchor level.

No single axis is sufficient by itself.

## Recognized object ecology families

These are the current Astra-facing family clusters that must visibly coexist.

### 1. Natural and harvested ecology
This family includes:
- harvested components,
- natural treasures,
- site yields,
- environmental extractions,
- anomalous resource outputs,
- and similar object families whose importance often comes from discovery or recovery.

RHBF pressures this family most strongly.

This family should remain distinct from purely manufactured or purchased gear.

---

### 2. Manufactured and technical ecology
This family includes:
- devices,
- tools,
- rated systems,
- computers,
- regulated equipment,
- fabricated parts,
- and other objects whose significance comes from engineered or system-driven behavior.

Shadowrun and Starfinder pressure this family strongly.

This family should remain distinct from magical or harvested source assumptions even when the two coexist in one setting.

---

### 3. Installed ecology
This family includes:
- augmentations,
- bodily installs,
- platform-side installs,
- capacity-bound systems,
- host-cost-bearing modifications,
- and comparable integration-heavy objects.

Shadowrun, Lancer, and Starfinder jointly justify this family strongly.

This family should remain distinct from:
- ordinary carried gear,
- bonded objects,
- and free independent devices.

---

### 4. Bonded and attuned ecology
This family includes:
- bound relics,
- attuned implements,
- bearer-linked growth objects,
- and similar relationship-dependent items.

RHBF pressures this family most strongly.

This family should not be collapsed into general magic-item doctrine or installable doctrine by default.

---

### 5. Platform-side ecology
This family includes:
- starship components,
- vehicle-side systems,
- frame/chassis systems,
- mounted weapon-bearing assets,
- support modules,
- deployables,
- and related platform-bound object classes.

Lancer and Starfinder strongly pressure this family.

This family should remain visible rather than being reduced to “gear for vehicles.”

---

### 6. Credential and access-shaped ecology
This family includes:
- passes,
- permits,
- seals,
- route keys,
- manifests,
- docking rights,
- access authorizations,
- and similar objects whose main importance is permission mediation.

RHBF, Shadowrun, and Starfinder all pressure this family differently.

This family should remain distinct from pure technology and from pure actor-side access state.

---

### 7. Hybrid object ecology
This family includes objects that are doctrinally mixed, such as:
- tech-magic hybrids,
- bonded technical assets,
- anomalous installs,
- magical systems with rated or access-sensitive behavior,
- or platform-side objects whose source logic is not singular.

Starfinder pressures this family most explicitly.

This family should not be used as a lazy collapse category.
It exists because some donors really do create objects that lawfully span multiple source families.

## Coexistence rules

Astra now needs explicit coexistence rules so one ecology does not overwrite another.

### Rule 1: Source family does not erase host family
A magical item and a technological item may both be body-facing installs.
Their source families differ even if their host relationship matches.

### Rule 2: Host relationship does not erase progression significance
A body-facing install may be progression-significant or ordinary utility.
A platform-side module may be identity-defining or replaceable.
Host family alone does not determine importance.

### Rule 3: Access-shaped objects remain visible
Objects whose main purpose is mediating permission should not disappear into general equipment doctrine.

### Rule 4: Bonded objects remain distinct from installs
An object linked by bearer growth/attunement is not automatically the same as a host-cost-bearing install.

### Rule 5: Platform-bound objects remain distinct from carried gear
An object meant for a ship, vehicle, frame, or support platform is not just a larger weapon or tool.

### Rule 6: Hybrid families should be explicit when they matter
When a donor truly mixes source logics, Astra should preserve that mix rather than forcing false purity.

### Rule 7: Donor chapter order is not doctrine
A donor placing technology, magic, hybrid items, and vehicles in separate chapters does not by itself define Astra’s taxonomy.
A donor combining them in one chapter does not erase meaningful distinctions either.

## Acquisition and recovery across ecologies

Different ecologies often imply different acquisition pathways.

Astra should currently recognize that objects may be:
- found,
- harvested,
- salvaged,
- crafted,
- purchased,
- requisitioned,
- licensed,
- bonded,
- awarded,
- claimed,
- installed,
- or recovered from sites/platforms/entities.

The stable rule is:

> Acquisition pathway is one ecology signal, but it is not the only ecology signal.

A harvested component is not the same as a salvaged subsystem.  
A bonded relic is not the same as a purchased device.  
A permit-bearing access asset is not the same as a magic item, even if both are rare and valuable.

## Cross-ecology interactions

### Natural/harvested with installed
Harvested or natural resources may become inputs to installed objects or upgrades.

### Manufactured with bonded
A manufactured object may become bonded or attuned without losing the fact that it began as a manufactured object.

### Rated system with access asset
A rated system may also mediate access, but not all rated systems are credential assets.

### Platform-side with mixed ecology
A starship or vehicle component may be purely technological, magical, hybrid, bonded, or access-shaped.
Platform-side location does not settle source family alone.

### Hybrid with progression significance
A hybrid object may be progression-significant, but hybrid status alone does not guarantee it.

### Deployables/support entities
A support entity may be:
- manufactured,
- hybrid,
- magical,
- platform-linked,
- or access-gated.

Its bounded middle-zone status should remain visible regardless of source family.

## Required ecology questions

When classifying an Astra object in a mixed ecology, ask:

1. What is its primary source family?
2. What host relationship does it depend on?
3. What operational role does it serve?
4. Is it progression-significant, identity-defining, or ordinary support?
5. Is it bonded, installed, carried, platform-bound, credential-bearing, or independent?
6. Does it change access, travel, operation, combat, salvage, or growth?
7. Does it lawfully span more than one ecology family?

If more than one family applies, preserve the mixed relationship explicitly instead of forcing false simplicity.

The following are lawful recurring profiles, but not universal Astra object law:

- exact hybrid-item chapter prominence,
- exact rarity ladders for different object ecologies,
- exact donor tech-level or magic-tier labels,

- exact donor installation caps,
- exact donor rarity/economy curves,
- exact subsystem chapter taxonomies,
- exact “magic item” or “tech item” universal category names.

Astra should support these as donor-local or profile-level patterns, not as global law.

## Design rules for Astra-facing mixed ecologies

1. Do not assume one dominant source family.
2. Keep source family, host family, and operational role distinct.
3. Keep bonded, installed, access-shaped, and platform-bound objects visibly separate when those distinctions matter.
4. Allow mixed-origin objects without flattening them into vague “special items.”
5. Do not let donor chapter order define Astra taxonomy.
6. Do not let one donor’s magic-tech balance become Astra baseline law.
7. Preserve progression-significant objects as a special concern rather than treating everything as ordinary loot.
8. Preserve object ecology breadth so later donors do not feel like exceptions.

- natural/harvested objects,
- manufactured/technical objects,
- installed objects,
- bonded/attuned objects,
- platform-side objects,
- credential/access assets,
