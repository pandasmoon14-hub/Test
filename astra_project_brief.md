# Astra Ascension — Project Brief and Standing Reference

## Project state

This project is building **Astra Ascension** as a coherent rules-and-setting ecosystem designed to absorb, translate, and consolidate material from many different TTRPG systems into a stable Astra-native framework.

The current focus is **not** to produce a finished public rulebook immediately.
The current focus is to establish durable doctrine and file architecture so future conversion work does not drift, hallucinate, or quietly inherit donor assumptions as hidden default law.

## High-level mission

Astra Ascension is intended to:
- absorb a large number of donor systems,
- convert them into Astra-native material consistently,
- reconcile mechanical variation without collapsing into contradiction,
- and ultimately support a strong canonical sourcebook plus a separate live-play adapter.

The target is not “make RHBF but renamed.”
The target is not “make a vague mashup universe.”
The target is a **translation-stable Astra framework** that can confidently ingest 80+ sourcebooks across multiple genres and subsystem styles.

## Current long-term pipeline

### 1. Build Astra doctrine
Create the core framework files that define the system architecture.
This stage exists to stop future conversion drift.

### 2. Convert 80+ donor books into Astra-native form
Once doctrine is stable enough, use the cloud conversion workflow to generate Astra-native equivalents of the donor material.
This is a conversion stage, not a final play-model stage.

### 3. Consolidate into a canonical Core Astra Ascension sourcebook
After substantial converted content exists, formalize a unified Astra sourcebook derived from:
- the doctrine stack,
- the converted corpus,
- and deliberate canon decisions.

This sourcebook becomes the main Astra-native reference.

### 4. Train a separate play-facing adapter
Only after Astra canon is mature should the project train a separate adapter focused on:
- live play,
- rules application,
- character creation guidance,
- scene handling,
- adjudication,
- and reliable Astra-native interaction.

That play adapter should learn from:
- the canonical sourcebook,
- the doctrine files,
- selected converted Astra-native content,
- and curated examples.

## Project discipline

The project depends on maintaining separation between four different layers:

1. **Doctrine / framework design**
2. **Conversion work**
3. **Canonical sourcebook consolidation**
4. **Live-play / GM behavior**

These layers should not be casually merged.
A great conversion corpus is not automatically a good play corpus.
A donor book converted into Astra form is not automatically final canon.
A training example is not automatically a rules authority.

## Canon hierarchy

When deciding conflicts, the authority order is:

1. Current Astra doctrine files
2. Canonical Core Astra Ascension sourcebook
3. Converted Astra-native donor content
4. Example packs and demonstrations
5. Original donor assumptions

This hierarchy exists to prevent donor systems from quietly becoming hidden law.

## RHBF’s role

**Ruthless Heavens Boundless Fate** is currently the most important donor pressure source because it provides strong mechanical and structural material.
Its value includes:
- useful character-creation pressure,
- domain and skill structure pressure,
- progression pressure,
- and strong donor evidence for what must be separated into doctrine layers.

But RHBF must not be treated as Astra’s automatic ceiling or universal metaphysical default.
Its cosmology is too specific to remain hidden in the walls unchanged.
RHBF-specific assumptions must only enter Astra when they are explicitly adopted into doctrine.

## Why the current Batch A focus matters

Batch A is no longer just “the RHBF-derived player-core packet.”
It is the foundational **player/chassis doctrine layer** for Astra Ascension.

Its job is to survive later conversion of many system types, including:
- fantasy systems,
- sci-fi systems,
- hybrid systems,
- cultivation systems,
- class/archetype systems,
- profession/background systems,
- proficiency/skill systems,
- spell, power, technique, maneuver, cyberware, biotech, psionic, relic, and tool-based systems,
- and unusual or esoteric subsystems.

That means Batch A must define durable architecture, not merely summarize one donor game.

## Accepted Batch A doctrine logic so far

The current accepted direction includes these design commitments:

- “Becoming an Ascendant” must be generalized beyond a single planetary assimilation story.
- Core resolution must be broad enough to absorb different system styles.
- Skills must function as translation architecture, not merely a short skill list.
- Character creation needs real framework files so Astra does not invent decorative nonsense.
- Paths, Domains, Techniques, progression, backgrounds, ancestry/kinform/culture, and starting kits all require enough structure to survive future donor pressure.
- Damage families deserve their own expansion logic rather than being buried as a tiny subsection.
- Resource economies, cost structures, recharge models, and backlash logic deserve their own doctrine layer.

## Why file 02 still comes first

The first doctrine file to draft should be:
- `batchA_02_player_chassis_doctrine.md`

This file must answer, in universal Astra terms:
- what an Astra player-grade actor is,
- what routes can produce one,
- what assumptions are mandatory,
- what assumptions are merely overlays,
- and what all downstream files are allowed to assume about a valid Astra character chassis.

Without that file, later files will quietly invent incompatible versions of the player entity.

## Main risks to watch

### 1. Donor cosmology leakage
Astra must not inherit RHBF’s metaphysics as hidden universal law by accident.

### 2. Skill / access / ability collapse
Competencies, permissions, and active abilities must stay distinct.

### 3. Resource-economy underdesign
Mana, stamina, ammo, heat, charges, cooldowns, corruption, overcast, backlash, and similar systems need formal architecture.

### 4. Damage underdesign
Damage families, resistance logic, and cross-family interactions can sprawl quickly.

### 5. Progression-axis collapse
Single-axis and multi-axis donor systems must not be forced into one muddy progression model.

### 6. Identity-layer collapse
Biology, kinform, heritage, culture, faction imprint, profession, and background must not be merged carelessly.

### 7. Canon hierarchy failure
Converted books are not automatically equal to the canonical sourcebook.

### 8. Conversion/play contamination
The conversion adapter and the play-facing adapter serve different roles and should remain distinct.

## Current training philosophy

The project’s current training logic is:

- first teach Astra doctrine and conversion discipline,
- then convert donor books into Astra-native content,
- then synthesize a formal sourcebook from that normalized content,
- then train a separate play-facing adapter on canon plus curated examples.

This is more realistic for an 8B-class model than trying to teach one adapter to invent Astra, reconcile donor conflicts, preserve all converted books equally, and run live play at the same time.

## Working principle going forward

Whenever a design choice risks short-term convenience at the cost of long-term conversion stability, choose the long-term stable option.

The project should always prefer:
- frameworks over vibes,
- explicit structure over implication,
- modular files over mega-documents,
- canon hierarchy over equal-weight aggregation,
- and disciplined architecture over donor mimicry.
