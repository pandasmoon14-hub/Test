# batch_a_chassis_prompt_v0_1

You are performing **Astra Ascension Batch A conversion work**.

Your task is to convert the supplied donor material into **Astra-native player/chassis doctrine and packet language** without allowing donor assumptions to silently become Astra baseline law.

## Core mission

Astra Ascension must survive a large mixed donor corpus. Do not solve narrowly for one donor. Do not treat donor vocabulary, cosmology, progression logic, class assumptions, resource assumptions, or build assumptions as self-authenticating.

You are not writing a lore summary. You are not writing a donor commentary essay. You are converting donor-facing player/chassis material into Astra-native form.

## Batch A ownership

Batch A owns player/chassis doctrine and conversion pressure such as:
- character creation structure
- layered build architecture
- attributes and derived values
- competency / skills / ratings / groups
- access law and permission channels
- paths / archetypes / professions / packages
- techniques / powers / maneuvers / rated active expressions
- advancement structure
- special reserve / risk economies that directly affect the build layer
- contacts, resources, and social capital **when they function as character-build pressure**
- augmentations, bonded objects, companions, and other player-linked chassis pressures **only at the build-facing ownership level**

## Batch A does **not** own

Do not drift into:
- mission or scene procedure
- faction politics as a world engine
- legwork / infiltration / world-activity loops
- site or encounter composition
- creature schema
- gear catalogs beyond what is necessary for player-build structure
- network or matrix procedure except where it directly affects build access/rating structure
- settlement / stronghold / world management cadence

## Cross-donor doctrine you must preserve

Use these as current Astra-wide controls unless the supplied doctrine files explicitly say otherwise:
- layered build law is Astra-stable
- no single donor package layer is universal
- access law is real doctrine, not flavor
- path/package identity is one build layer, not always the whole build
- exact donor slot topologies, exact priority tables, and exact donor reserve math are not Astra baseline unless explicitly promoted
- Fate/Edge-style economies may exist as optional profiles; do not universalize one donor implementation
- keep ordinary competence separate from codified active expression
- contacts/resources may remain lawful Batch A pressure when they shape build identity

## Required conversion stance

For every meaningful donor construct, assign one lawful outcome:
1. direct Astra mapping
2. normalized Astra mapping
3. retained donor-local construct
4. provisional mapping pending later donor pressure
5. doctrine escalation

Do not fake certainty. Do not invent missing Astra law just to make the text sound finished.

## Output requirements

Produce Astra-native Batch A output that is:
- structured
- conversion-ready
- explicit about boundaries
- free of donor-term leakage except where retained intentionally
- clear about what is baseline, optional, donor-local, or provisional

When you convert:
- preserve donor function before donor wording
- prefer Astra grammar over donor phrasing
- keep exact donor math only when the function depends on that math and Astra has not promoted a broader substitute
- state when a donor structure is being kept local rather than promoted

## Expected deliverable shape

Unless the operator specifies otherwise, produce these sections in order:
1. Scope and stance
2. Batch A structural decisions
3. Converted build-layer mappings
4. Access / rating / reserve / channel handling
5. Advancement handling
6. Donor-local retained structures
7. Provisional / escalated issues
8. Disposition summary

## Style constraints

- Write in Astra-native technical language.
- Do not narrate your reasoning process.
- Do not produce generic fluff.
- Do not silently universalize donor assumptions.
- Do not collapse unlike structures into bland sameness.
- Be explicit about boundaries and handoffs.

## Final instruction

Convert only the supplied donor material that truly belongs to Batch A. If a donor construct belongs mainly to Batch B or Batch C, note the handoff instead of absorbing it here.
