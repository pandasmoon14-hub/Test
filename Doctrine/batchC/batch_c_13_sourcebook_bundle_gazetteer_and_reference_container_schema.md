# batchC_13_sourcebook_bundle_gazetteer_and_reference_container_schema.md

## Purpose and authority

This file defines the **bundle, container, gazetteer, dossier, compendium, guide, kit, and reference-container schema** for Astra Ascension.

Its job is to define how Astra represents **packaged collections of already-defined content families** when those collections are not themselves one actor entry, one site entry, one faction entry, one mission/scenario structure, one reward parcel, or one generator.

This file is doctrine for **container-shaped content packages**.
It does **not** define atomic entry families.
It does **not** define overlay families.
It does **not** define encounter or mission composition families.
It does **not** define reward parcel structure.
It does **not** define generator structure.
It does **not** define canon consolidation.
It does **not** convert donor sourcebooks into canon merely because they can be represented in sourcebook-like container form.

C13 exists because a large donor corpus does not only contain individual entries and play structures.
It also contains **books, guides, dossiers, compendia, reference collections, operational kits, and mixed-family source units** that package many lower schemas together.
Those packages cannot safely be collapsed into the entry families they contain, and they cannot safely be treated as doctrine or canon simply because they resemble finished books.

C13 therefore owns the schema grammar for:
- container identity
- contained-content aggregation
- container family, function, posture, and source-relation distinctions
- section and inclusion topology
- contained-unit role doctrine
- cross-family weighting inside mixed containers
- cross-container reference structure
- donor-publication normalization into Astra container form
- reference-facing, conversion-facing, and hybrid container postures
- authored publication unit vs normalized Astra container distinctions
- linked, federated, companion, and derivative container relations
- container-local supporting prose roles
- container output ecology and access topology

C13 remains **schema doctrine**, not converted output, not canon, not a lore encyclopedia, and not a mini-sourcebook-writing guide.

## Why C13 must exist

A container schema is mandatory for corpus-scale conversion.

Astra must be able to represent donor structures such as:
- gazetteers
- location chapters
- city guides
- world profiles
- bestiary books or bestiary packs
- item compendia
- faction dossiers
- mixed GM-facing reference books
- converted sourcebook bundles
- modular reference packets
- appendicial reference kits
- operator kits and support toolkits
- cross-family guidebooks that package sites, factions, parcels, generators, and scenarios together
- multi-book lines that share indices, appendices, or reference spines

These are not interchangeable.
A city guide is not the same as one site entry.
A bestiary pack is not the same as one actor entry.
A faction dossier is not the same as one faction record.
A converted sourcebook bundle is not the same as one mission/scenario structure.
A world profile is not the same as one gazetteer.
A location chapter is not automatically one locale record.
An operator kit is not the same as a bestiary pack or a world profile just because it also packages supporting content.

Without C13, later conversion work will tend to fail in one of twelve ways:
1. treat packaged donor books as if they were giant atomic entries
2. treat reference collections as if they were scenarios or world profiles by default
3. move lower-schema ownership into C13 because the donor packaged those materials together
4. treat donor-publication chapters as one-to-one with Astra containers
5. mistake sourcebook-shaped bundles for doctrine or canon
6. flatten mixed-family books into one vague “setting file”
7. hide bundle topology, section roles, and cross-container relations in prose without schema ownership
8. treat appendix collections as if they were lower-schema definitions
9. use C13 as an overflow bucket for anything too large to fit elsewhere
10. mistake line toolkits and support kits for one of the cleaner family labels
11. let donor organization of content silently define Astra container topology
12. flatten multi-book lines or federated reference suites into one oversized bundle despite distinct functions

C13 exists to stop those failures.

## Core doctrinal center

The doctrinal center of C13 is:

**C13 owns the schema by which Astra represents packaged content containers.**

In Astra terms, C13 owns the structure of:
- what kind of container this is
- what it is for
- how it groups lower schemas
- what donor-source relation it has
- how sections, indices, and inclusion structures are organized
- what reference posture, access posture, and outcome posture it has
- what donor-publication relation it preserves or rejects
- how it links to other containers
- how it exposes contained content without re-owning that content

C13 is therefore about **container-shaped packaging and reference structure**.
It is not about redefining the entries contained inside the package.

## Scale ladder and container place in Batch C

For Batch C routing discipline, Astra should recognize the following scale relation for container-shaped content:

1. **Underlying entry or lower-schema family**
   One actor, site, faction, item, parcel, generator, packet, mission, or other lower content family that can be contained.

2. **Contained-unit relation**
   One inclusion, section placement, cross-reference, thematic grouping, regional grouping, index link, output-access path, support-note role, or dependency relation within a container.

3. **Container record**
   One gazetteer, dossier, compendium, bundle, guide, world profile, kit, reference packet, or other container owned by C13.

4. **Linked or federated container set**
   Several related containers that remain distinct because they differ in audience posture, function, scope, donor relation, output ecology, or contained-family emphasis, even while sharing one index spine, line identity, or toolkit umbrella.

5. **Library- or canon-scale assembly**
   Larger accumulations of converted output or formal canon consolidation beyond ordinary Batch C container schema ownership.

C13 owns level 3 directly and defines level-2 contained-unit logic where needed.
C13 may acknowledge level 4 as linked or federated container aggregation, may reference level 1 content families, and may later feed level 5 processes, but it does not own those other layers.

## Dominant-function routing rules

All candidate C13 content must be routed by **dominant structural job**, not by donor publication size or prestige.

A record belongs in C13 when its main job is to define:
- a packaged reference assembly
- a bundle of lower-schema content
- a container for later reference, conversion, audit, export, or grouped presentation
- a guide, dossier, compendium, bundle, kit, world profile, or sourcebook container whose main work is packaging rather than becoming one contained entry

A record does **not** belong in C13 merely because it is large, chapter-shaped, prestigious, or printed as a book.

If the dominant job is defining the actual place, route to C06.
If the dominant job is defining the actual faction/institution, route to C07.
If the dominant job is defining the actual mission/scenario structure, route to C10.
If the dominant job is defining reward parcels, route to C11.
If the dominant job is defining generators, route to C12.
If the dominant job is final canon consolidation, route beyond Batch C.

## Container identity, contained content, publication unit, and source-relation distinctions

C13 must preserve several distinct layers that donors often blur together.

### Container identity

A **container** is the packaged assembly itself.
It has its own identity, purpose, scope, posture, and section topology.

### Contained content identity

A **contained entry** retains its own schema family and entry identity even while being included inside a container.

### Publication unit

A **publication unit** is the donor-presented chapter, section, appendix, guidebook, sourcebook, or boxed grouping from which normalization begins.
It is not automatically equivalent to one Astra container.

### Source relation

A **source relation** describes how the Astra container stands in relation to donor-origin packaging or donor-origin source lines.
It may preserve donor structure closely, regroup donor materials heavily, synthesize an operator-facing assembly from several donor units, or stage canon-preparatory aggregation without becoming canon.

### Distinction principles

1. Container identity is not contained-entry identity.
2. Publication unit identity is not automatically container identity.
3. Source relation is not the same as container posture.
4. A donor chapter may normalize into zero, one, or many Astra containers.
5. Several donor chapters may normalize into one Astra container.
6. One container may include many contained schema families without becoming any one of them.

## Container family, function, posture, and source-relation doctrine

C13 must distinguish several different kinds of classification.

### Container family

Container family identifies **what kind of container** the record is.
Examples include:
- gazetteer
- city guide
- world profile
- bestiary pack
- item compendium
- faction dossier
- converted sourcebook bundle
- modular reference container
- appendix-like support container
- mixed-reference container
- operator or reference kit

### Container function

Container function identifies **what the container is for**.
Examples include:
- regional reference
- urban reference
- world orientation
- actor cataloging
- item cataloging
- faction-reference packaging
- donor-book preservation
- conversion staging
- campaign-support reference
- quick-reference support
- cross-family orientation
- operator tooling
- audit trace support

### Container posture

Container posture identifies **how the container behaves** with respect to normalization, audience, and use.
Examples include:
- purely reference-facing
- conversion-staging
- hybrid reference-and-routing
- donor-structure-preserving
- normalized-Astra regrouped
- broad overview
- deep dossier
- pack-like modular
- chapter-like serial
- operator-facing rather than player-facing

### Container source relation

Container source relation identifies **how the container relates to donor-origin material or donor-line packaging**.
Examples include:
- donor-preserving mirror
- normalized Astra regrouping
- synthetic operator-facing assembly
- canon-preparatory reference set
- hybrid preserved-and-regrouped container
- line-derived companion assembly

Family, function, posture, and source relation must not be collapsed into one label.

## Core C13 container families

C13 should support the following major container families.
These are not all interchangeable and should not be flattened into one universal “sourcebook bundle” schema.

### 1. Gazetteer

A container packaging multiple place-related references, regional clusters, routes, factions, service notes, and related content into a reference-facing package.
A gazetteer is not one site entry.

### 2. City guide

A container packaging urban districts, major sites, local factions, services, access notes, operational neighborhoods, and supporting references into a city-scale package.
A city guide is not one settlement entry.

### 3. World profile

A container packaging world-scale orientation, regions, global pressures, major powers, broad travel or cosmological references, and cross-region index structure.
A world profile is not automatically a gazetteer and is not one site record.

### 4. Bestiary pack

A container packaging actor entries, taxonomic groupings, habitats/regions, challenge links, encounter-use notes, and related actor reference materials.
A bestiary pack is not one actor schema.

### 5. Item compendium

A container packaging item entries, resource families, relic groupings, usage classes, acquisition notes, and item-reference structures.
An item compendium is not one item entry.

### 6. Faction dossier

A container packaging one or more faction entries, relation maps, supporting documents, operation notes, access notes, parcel hooks, generator hooks, or dossier-style grouping.
A faction dossier is not automatically one faction record.

### 7. Converted sourcebook bundle

A container representing a normalized donor book or donor-book subset as an Astra package while preserving traceability, grouping, and section relations without elevating the bundle to doctrine or canon.

### 8. Modular reference container

A reusable package organized for practical reference use, operator use, conversion staging, or subsystem-local grouping without pretending to be one donor book.

### 9. Appendix-like support container

A container packaging support materials such as tables, indices, maps, quick references, glossaries, side tools, or cross-family support collections while preserving that such collections do not redefine the lower schemas they include.

### 10. Mixed-reference container

A container packaging multiple unlike families together when donor structure or Astra normalization requires a lawful grouped assembly rather than a single-family compendium.

### 11. Operator or reference kit

A mixed operational kit packaging reference materials, support appendices, tools, rosters, route aids, quick-reference structures, procedural supports, or cross-family working sets for operator-facing use without pretending to be one cleaner family such as gazetteer or compendium.

## Boundary hardening for world profiles, gazetteers, city guides, and locale packets

These donor forms blur together frequently and need harsher boundaries.

### World profile
A world profile is **world-scale orientation and macro reference packaging**.
It organizes broad world-level structure, not one localized place unit.

### Gazetteer
A gazetteer is **multi-place or regionalized reference packaging**.
It organizes many place-linked materials into one reference structure, not one place record.

### City guide
A city guide is **urbanized, district-sensitive, locality packaging**.
It organizes city-scale or settlement-scale grouped material, not one locale entry.

### Locale packet or locale-heavy composition
A richly written locale packet remains **C06 or C09 depending on dominant function**.
It does not become C13 simply because it is elaborate or book-like.

### Boundary principles

1. Rich writing density does not create container identity by itself.
2. Urban packaging is not the same thing as one settlement entry.
3. Multi-place grouping is not the same thing as one place record.
4. A locale packet belongs in C06 or C09 when dominant function remains placehood or local composition rather than packaging.

## Container granularity doctrine

C13 must preserve container granularity explicitly.

### Granularity classes
- atomic container
- sectioned container
- linked container set
- federated container set
- bundle of bundles
- library-facing grouping beyond ordinary container identity

### Granularity principles

1. One donor “book” may normalize into several containers.
2. A sectioned container remains one container when its internal sections share one structural identity line.
3. A linked container set should remain plural when components differ materially in function, audience posture, donor relation, or contained-family emphasis.
4. A federated set may share one index spine, line identity, regional cluster, campaign line, or toolkit umbrella without collapsing into one supersized bundle.
5. Library-facing or canon-facing assemblies should not be silently treated as ordinary container records.
6. Granularity is structural, not presentational.

## Containment and inclusion topology doctrine

C13 must preserve how lower content is grouped into containers.

### Inclusion topology classes

- flat inclusion list
- sectioned inclusion
- hierarchical inclusion
- multi-family grouped inclusion
- regional grouping
- thematic grouping
- dossier grouping
- index-driven inclusion
- hub-and-spoke reference topology
- companion appendix relation

### Inclusion principles

1. Inclusion topology is not the same as contained-entry identity.
2. A container may group unlike families without collapsing them into one schema family.
3. Strong grouping at the donor level does not justify moving lower-schema ownership into C13.
4. Inclusion topology should preserve both grouping logic and cross-reference logic when donor or Astra use posture depends on them.

## Contained-unit role doctrine

Contained entries do not all perform the same job inside a container.
C13 must preserve their roles without transferring ownership.

### Contained-unit role classes
When relevant, Astra should distinguish:
- primary entry
- secondary support reference
- sidebar or callout support
- index anchor
- appendix support entry
- cross-family context note
- audit or traceability anchor
- quick-reference anchor

### Contained-unit role principles

1. Contained-unit role is not contained-entry family.
2. A primary entry remains owned by its lower schema even when container prominence is high.
3. Support references and audit anchors should not be mistaken for secondary ownership by C13.
4. Donor books often embed one family inside another in structurally meaningful ways; role doctrine prevents accidental flattening.

## Cross-family weighting doctrine for mixed containers

Mixed containers often have unequal family emphasis.
C13 should preserve that explicitly rather than pretending all included families matter equally.

### Cross-family weighting classes
When relevant, Astra should distinguish:
- primary inclusion family
- secondary support family
- tertiary appendix family
- trace-only family

### Weighting principles

1. Weighting affects container identity and use posture without transferring ownership.
2. A site-led container with faction support is not the same as a faction-led dossier with site support.
3. Generator-heavy support kits and mission-heavy reference bundles need explicit weighting to normalize cleanly.
4. Trace-only families should not be promoted to primary families merely because the donor placed them nearby.

## Section-role doctrine

C13 must preserve that sections inside a container do different jobs.

### Section-role classes
When relevant, Astra should distinguish:
- orientation or overview section
- indexed entry section
- thematic grouping section
- regional grouping section
- cross-reference section
- donor-traceability section
- appendix-like support section
- operator note section
- usage-routing section
- glossary or terminology section

### Section-role principles

1. Section role is not contained-entry family.
2. A section may organize content without owning the content it organizes.
3. Section roles matter because donor books often hide several distinct container jobs inside one chapter sequence.

## Structural density and composition-mode doctrine

Some containers are sparse wrappers around a small set of linked entries.
Others are dense structured packages with many sections, indices, cross-family groupings, or support tools.
Density alone is not enough; C13 must also preserve what kind of density the container has.

### Density classes
- sparse wrapper container
- medium-density reference container
- dense authored container

### Composition-mode classes
When relevant, Astra should distinguish:
- mostly index and cross-reference
- mostly grouped inclusions
- mostly section-driven exposition
- mostly appendix or support tooling
- heavily mixed

### Density principles

1. Sparse containers should not be artificially inflated.
2. Dense containers should not be flattened into one vague bundle label if section roles and inclusion topology matter.
3. Density is structural, not stylistic.
4. Composition mode helps distinguish equally dense but differently organized containers.
5. A dense container may still remain narrowly scoped if its function is specific.

## Index and cross-reference topology doctrine

Indices are one of the main ways containers expose lower content.
C13 should preserve whether an index belongs to one container or to a larger set.

### Index topology classes
When relevant, Astra should distinguish:
- internal container-bound index
- external shared index
- family-specific index
- federated multi-container index
- donor-preserving table-of-contents spine
- route-oriented reading path
- dossier-style investigative path

### Index principles

1. Index topology is not the same as inclusion topology.
2. An index may expose content differently from its section placement.
3. A federated index does not collapse several containers into one supersized bundle.
4. Donor-preserving table-of-contents structure may coexist with a normalized Astra access spine.

3. Reference-facing containers must not be mistaken for canon merely because they look complete.
4. Operator-facing posture should be explicit when a bundle exists to support later conversion work rather than direct play reference.
5. Access posture should interact with C01 visibility discipline when one container needs public-safe and restricted forms.

## Container-local supporting prose doctrine

A large donor corpus will include explanatory matter, legends, usage notes, historical framing, editorial commentary, sidebars, conversion notes, and other prose that is not itself a contained entry.
C13 must preserve this as packaging-support material rather than force every paragraph into a lower entry family.

### Supporting prose classes
When relevant, Astra should distinguish:
- pure packaging prose
- orientation prose
- conversion-operator prose
- traceability commentary
- non-authoritative flavor support
- doctrine-summary references

### Supporting prose principles

1. Supporting prose is not automatically a contained entry.
2. Supporting prose may matter structurally without owning lower-schema content.
3. Supporting prose should not be mistaken for canon authority simply because it sits near a polished bundle.
4. Packaging context should not be dropped merely because it is not one of the lower entry families.

## Container output ecology doctrine

Containers do not own lower-family outputs, but they do expose them in structured ways.
C13 should preserve how a container expects users to move through its content.

### Output ecology classes
When relevant, Astra should distinguish:
- searchable reference exposure
- grouped quick-reference access
- thematic entry paths
- route-oriented reading order
- dossier-style investigative access
- appendix support access
- audit or tracing paths

### Output ecology principles

1. Output ecology is not the same as section order.
2. Containers with similar included content may still differ materially in how they expose that content.
3. This distinction is especially useful for differentiating world profiles, city dossiers, bestiary packs, and operator kits.

## Cross-container relation and lineage doctrine

- appendix-attached container
- regional expansion container
- bestiary companion to site/region guide
- item companion to sourcebook bundle
- dossier attached to broader world profile

### Lineage classes
When relevant, Astra should distinguish:

- alternate regrouping

- partial replacement
- superseding revision

- split derivative line

### Cross-container principles

2. Companion containers should not be flattened into one larger bundle unless that larger bundle has its own lawful container identity.
3. Alternate regroupings of the same donor material should preserve derivative or pinned-history discipline.
4. A supplement relation does not transfer ownership of the contained family to the container.
5. Lineage language should be used whenever container change is more than editorial drift.

## Federated container-set doctrine

Some donor lines and converted suites are best represented as several distinct containers sharing one broader reference spine.
C13 must support this explicitly.

### Federation classes
When relevant, Astra should distinguish federated sets sharing:
- one index spine
- one donor source line
- one thematic umbrella
- one campaign line
- one regional cluster
- one operator-facing toolkit identity

### Federation principles

1. Federation is not the same as bundle-of-bundles.
2. Federated containers may share access paths, indices, and tool identities without losing distinct container identities.
3. Multi-book settings, supplement suites, campaign lines, and box-set style donors especially pressure this distinction.
4. Federation should prevent flattening a whole line into one blob while still preserving line-level coherence.

## Donor-publication normalization doctrine

C13 must explicitly govern how donor publication structure is normalized into Astra container form.

### Publication normalization classes
When relevant, Astra should distinguish:
- one publication unit to one container
- one publication unit to several containers
- several publication units to one container
- mixed C13 plus lower-schema normalization
- donor-structure-preserving bundle
- normalized regrouped Astra container

### Publication normalization principles

1. Donor publication layout is not container identity.
2. One donor chapter may normalize into zero, one, or many containers.
3. Several donor chapters may normalize into one container.
4. Publication headings, appendices, and sidebars do not prove container family by themselves.
5. Converted sourcebook bundles should preserve donor traceability without letting donor table-of-contents structure become Astra law.
6. One donor “setting chapter” may normalize partly into C13 containers and partly into C06/C07/C10/C12 structures.
7. One donor appendix may contain several distinct container roles.
8. One “GM tools” section may contain both C12 generators and C13 packaging logic.

## Traceability and donor-preservation posture doctrine

Some containers exist partly to preserve donor-origin structure for later audit, operator use, or comparison.
This must be explicit rather than silently assumed.

### Traceability posture classes
When relevant, Astra should distinguish:
- high donor-preservation posture
- moderate donor-trace posture
- normalized-Astra regrouped posture
- audit-facing posture
- conversion-work staging posture

### Traceability principles

1. Traceability posture is not the same as canon posture.
2. High donor-preservation posture does not raise donor structure to doctrine.
3. Highly normalized regrouping should preserve enough traceability to support later audit and revision.
4. Containers that preserve donor sequence should still be reviewed through C01 derivative and pinned-history logic when revised.

## Container revision doctrine

Containers are especially likely to look “editorially changed” when they are actually structurally different.
C13 must set a higher threshold for when revision preserves identity and when it should mint a derivative line.

### Revision classes
When relevant, Astra should distinguish:
- editorial revision preserving identity
- regrouping still preserving identity
- derivative regrouping requiring new line
- pinned historical preservation of old shape

### Revision principles

1. Editorial cleanup does not always require a new container line.
2. Structural regrouping that materially alters inclusion topology, audience posture, source relation, or output ecology may require a derivative line.
3. Silent container drift is especially dangerous because it often looks like mere editing.
4. Historical bundle shapes should remain pin-able when operator traceability or audit needs them.

## Template, archetype, instance, and container-set doctrine

C13 must support both reusable container patterns and specific normalized container instances.

### Reusable container templates
Examples of template-level container patterns include:
- city guide with district sections, faction references, service appendices, and site index
- bestiary pack with actor families, habitats, and cross-reference appendices
- sourcebook bundle preserving donor section sequence with normalized Astra cross-links
- world profile with region sections, faction overviews, and route index
- dossier pack linking multiple organizations and one operational theater
- operator kit with mixed appendices, procedural aids, maps, and reference rosters

These are templates, not converted container instances.

### Normalized archetypal containers
A normalized archetypal container is Astra’s stable doctrinal skeleton for a recurring donor publication family, narrower than a pure template and broader than one converted instance.

### Instantiated containers
An instantiated container is a specific gazetteer, dossier, bundle, compendium, guide, or reference packet created during donor conversion.

### Linked and federated container-set doctrine
Linked or federated container sets may appear in C13 in two distinct ways:
- as several containers with explicit companion, sequel, supplement, federation, or lineage relations
- as one higher-level container that lawfully includes subordinate container sections without erasing their lower identities

These are not the same schema situation and must not be conflated.

### Normalization rules

1. Preserve donor container distinctions whenever Astra can represent them lawfully.
2. Normalize donor publication structure into Astra container structure only as far as needed for cross-corpus stability.
3. Do not invent canon authority merely because a normalized container looks polished.
4. Do not erase genuine donor packaging complexity simply to make the bundle look simpler.
5. Mark authored publication structure and normalized container structure separately when they differ materially.

## Donor-presentation normalization warnings

Donor presentation frequently disguises container structure.
C13 must explicitly warn against flattening presentation into schema identity.

### Common misleading presentation forms
When relevant, Astra should watch for:
- “setting chapter” headings that hide several lower-schema families plus one or more containers
- “GM section” labels that mix C10, C12, and C13 work
- appendix blocks that combine support tooling, reference packaging, and lower-schema material
- prestige or hardcover presentation that tempts converters to over-promote container authority
- boxed side materials that are still lower-schema entries rather than container records

### Warning principles

1. Presentation format is not structural identity.
2. One page or one chapter may contain multiple container roles.
3. Book prestige is not schema ownership.
4. Donor organization of content must not silently define Astra container topology.

## Publication-unit warning

A donor publication unit may normalize into:
- one container instance
- several linked container instances
- one container template plus one or more instances
- mixed lower-schema entries plus one or more containers
- mixed C13 and non-C13 records without one-to-one correspondence

One donor chapter may normalize into zero, one, or many C13 records, and may also normalize partly into C06, C07, C10, C11, or C12 structures without one-to-one correspondence.
Location chapters, bestiary sections, setting chapters, appendices, GM toolkits, and support suites are especially misleading publication forms and must not be mistaken for container family granularity.

Publication layout is not container identity.

## Cross-file composition discipline

### C02 actor handoff

C13 may package actor entries in bestiary packs, world profiles, or mixed-reference containers.
C13 must not redefine actor ontology.

### C04 item/resource handoff

C13 may package item entries in compendia, bundles, appendices, or mixed-reference containers.
C13 must not redefine item ontology.

### C06 site handoff

C13 may package site entries in gazetteers, city guides, world profiles, and mixed-reference containers.
C13 must not absorb place ontology.
A gazetteer is not one site entry.
A city guide is not one locale record.

### C07 faction handoff

C13 may package faction entries in dossiers, sourcebook bundles, world profiles, or mixed-reference containers.
C13 must not absorb faction ontology.
A faction dossier is not automatically one faction record.

### C09 packet handoff

C13 may package encounter packets or scene packets inside guides, bundles, or supporting containers.
C13 must not redefine packet structure.

### C10 mission/scenario handoff

C10 owns the playable macro-composition itself.
C13 owns only the container that may package such structures for reference, audit, staging, or grouped presentation.
If the dominant job is to package several missions, scenarios, arcs, or path segments for reference, it stays in C13.
If the dominant job is the macro-playable structure itself, it belongs in C10 even if the donor presented it as a book.
An adventure module chapter may normalize partly into C10 structures and partly into C13 packaging.
A sourcebook bundle is not itself a mission/scenario schema.

### C11 parcel handoff

C13 may package parcel templates, parcel instances, reward appendices, or claim-support materials.
C13 must not redefine parcel structure.
A reward appendix is not C11 doctrine.

### C12 generator handoff

C13 may package tables, oracles, toolkits, rumor kits, encounter-table chapters, or procedural support sections inside containers.
C13 must not redefine generator structure.
A generator appendix or oracle section belongs in C13 only insofar as it is packaged as a container section; the generator identity itself still belongs to C12.
A table appendix is not the same thing as C12 schema ownership.

### Doctrine and canon handoff

C13 is still Batch C doctrine.
It defines container schemas.
It does not elevate converted bundles to doctrine.
It does not elevate sourcebook-shaped bundles to canon.
Formal canon consolidation remains outside ordinary Batch C container ownership.

## Anti-collapse rules

C13 must not:
- treat containers as if they were atomic entries
- move lower-schema ownership into C13 because the donor packaged materials together
- treat a city guide as one site record
- treat a faction dossier as one faction record
- treat a bestiary pack as one actor schema
- treat a converted sourcebook bundle as canon
- treat donor-publication chapters as one-to-one with Astra containers by default
- flatten mixed-family books into one vague “setting file” when several containers are warranted
- become a lore encyclopedia, donor conversion repository, or mini-sourcebook under another name
- use container size or prestige as proof of schema ownership
- mistake mixed GM supplements or operator kits for one cleaner family without structural justification
