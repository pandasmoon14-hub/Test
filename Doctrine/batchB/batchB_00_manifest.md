# batchB_00_manifest

## 1. Purpose and authority

This file is the canonical map for **Batch B** of Astra Ascension.

Batch B is the **operational / world-interaction doctrine shell**.
Its job is to define the procedural and systemic layer that surrounds the Batch A player/chassis framework once characters begin acting inside scenes, conflicts, environments, institutions, economies, and ongoing campaign structures.

This file is authoritative for:
- the Batch B file roster
- Batch B scope boundaries
- Batch B build order
- Batch B dependency logic
- authoritative ownership of Batch B subject areas
- status flags for Batch B files
- Batch B relationship to Batch A and reserved Batch C work

This file is **not** authoritative for:
- player/chassis doctrine already owned by Batch A
- conversion examples or edge-case worked mappings
- canon-sourcebook prose
- live-play guidance
- bestiary, adventure-path, or random-table schema content reserved for Batch C

## 2. Batch B mission

Batch B exists to define the **operational doctrine shell** for Astra Ascension.

Where Batch A answered:
- what a player-grade actor is
- how a character is structured
- how abilities, access, damage, conditions, progression, and starting identity/loadout are architected

Batch B answers:
- how scenes and encounters are procedurally run
- how conflict, survival, environment, and discovery operate
- how gear, value, crafting, and services function
- how factions, institutions, downtime, companions, vehicles, and bases operate
- how post-encounter inflows and logistical continuity are handled

Batch B must remain:
- modular
- doctrine-first
- conversion-stable
- operational rather than encyclopedic

## 3. Scope boundaries

### 3.1 Batch B covers

Batch B covers:
- scene, conflict, action, and encounter procedure
- health, injury, healing, death, and recovery procedure
- the full item / gear object model
- wealth, value, requisition, barter, and trade doctrine
- crafting, salvage, repair, modification, and installation
- exploration, travel, navigation, time, and distance
- hazards, weather, regions, exposures, and afflictions
- stealth, detection, visibility, and sensory procedure
- investigation, research, clue flow, and discovery procedure
- social conflict, influence, reputation, and negotiation procedure
- downtime, training, projects, services, and rest-cycle procedure
- factions, institutions, clearance, and service-access procedure
- companions, auxiliaries, summons, drones, and similar support entities
- mounts, vehicles, mechs, ships, and platform operations
- settlements, bases, capitals, and infrastructure operations
- loot, rewards, salvage, and post-encounter inflows

### 3.2 Batch B does not cover

Batch B does not cover:
- player/chassis ontology
- core stat architecture
- capability law
- access law
- ability schema
- resource economy in the ability-side sense
- condition and damage-family taxonomy
- progression-axis doctrine
- identity-layer doctrine
- package grammar
- chargen procedure
- starter-loadout entry doctrine

Those remain in Batch A.

Batch B also does **not** own:
- creature / NPC / adversary conversion schema
- traps / puzzles / trials schema
- scenario / mission / adventure-path schema
- random-table and generator schema

Those are reserved for Batch C.

## 4. Relationship to Batch A

Batch B is downstream of Batch A.

Batch B assumes Batch A already defines:
- legal actor/chassis structure
- resolution grammar
- stat architecture
- capability architecture
- access architecture
- ability object architecture
- resource-cost / recharge / backlash architecture
- condition/effect architecture
- damage-family taxonomy
- progression architecture
- identity-layer architecture
- package grammar
- character-assembly procedure
- starter-loadout entry protocol
- conversion invariants

Batch B may reference those doctrines.
It may not silently overwrite them.

## 5. Relationship to Batch C

Batch C is explicitly reserved as the later **content / schema conversion shell**.

Batch C will exist because Batch B alone is not sufficient to absorb:
- creature and adversary catalogs
- traps, puzzles, and trials
- adventures and adventure paths
- random tables and generators
- location/module content schemas

Batch B defines the operational rules shell.
Batch C will define the content/object schemas that populate that shell.

## 9. Dependency logic

### 9.1 Batch B depends on Batch A

All Batch B files assume Batch A doctrine remains authoritative for core actor, ability, damage, progression, and conversion law.

### 9.2 Internal Batch B dependency expectations

- `B01` should precede most other operational procedure files because it defines the general scene/conflict shell.
- `B02` depends on `B01` for structured operational sequencing.
- `B03–B05` form a tightly related material/economy/crafting cluster.
- `B06–B09` form an exploration/environment/sensory/investigation cluster.
- `B10–B12` form a social/institutional/downtime cluster.
- `B13–B15` handle nonstandard holders, platforms, and site-scale operations after the general shell is established.
- `B16` should follow the encounter/economy/infrastructure files so inflow logic has lawful anchors.

### 9.3 Anti-collapse rule

No Batch B file may silently absorb:
- Batch A player/chassis law
- later Batch B file ownership
- reserved Batch C schema ownership

If a design pressure cannot be solved cleanly within the owning file, it must be escalated rather than smeared across the stack.

## 13. Final doctrine note

Batch B is not “more character files.”
It is the operational shell that characters, entities, institutions, environments, and material systems collide with after Batch A is in place.

If a proposed file is primarily about:
- what a character is,
- how a character is built,
- or how an actor-side rule object is defined,

it probably belongs in Batch A.

If a proposed file is primarily about:
- how the world, encounter space, environment, institution, economy, or operational procedure behaves around those actors,

it probably belongs in Batch B.

If a proposed file is primarily about:
- monsters,
- traps,
- adventures,
- random tables,
- generators,
- or content-object schema,

it probably belongs in Batch C.
