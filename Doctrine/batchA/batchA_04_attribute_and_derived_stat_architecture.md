# batchA_04_attribute_and_derived_stat_architecture.md

## Status
Draft — Core Now

## Authority
This file is the authoritative doctrine file for **Astra's character-side numerical architecture**.

It defines:
- what qualifies as a primary attribute,
- what qualifies as a derived stat,
- how defensive values are classified,
- how tempo, mobility, and awareness values are handled,
- how resistance and save families are structured,
- how supplemental and exceptional metrics are admitted,
- how multi-frame actors and temporary overlays are handled,
- and how donor-side stat structures map into Astra without polluting core doctrine.

It does **not** define:
- chargen assignment procedures,
- point-buy or rolled-stat methods,
- package-specific bonuses,
- ancestry or kinform grants,
- equipment tables,
- ability formulas,
- resource-spend rules,
- progression tables,
- or full donor conversion examples.

If any later Batch A file appears to contradict this file on the **classification, ownership, or admissibility of character-side quantitative structures**, this file governs unless deliberately revised.

---

## 1. Purpose

Astra requires a stable numerical skeleton.

That skeleton must be strong enough to support later files that reference persistent actor-side numbers, but flexible enough to survive donor systems that do **not** present a traditional six-stat sheet.

This file therefore exists to prevent three opposite failures:
- **donor body-plan capture**, where one donor's attribute package becomes hidden Astra law,
- **stat mush**, where attributes, defenses, tracks, resources, and oddball metrics are thrown into one bag labeled “numbers,”
- and **mapping collapse**, where later conversions import new donor stats directly into core doctrine without classification discipline.

Astra does not treat every donor stat list as sacred. It classifies what a number *is doing* before deciding where it belongs.

## 3. Core Stat Architecture Principles

### 3.1 Astra uses a stable internal scaffold
Astra maintains a stable internal numerical scaffold even when a donor system does not expose equivalent named attributes.

A donor may express capability through:
- attributes,
- approaches,
- roles,
- drives,
- themes,
- skill arrays,
- tags,
- or other nontraditional structures.

That does not remove Astra's need for a coherent internal architecture. It only changes how the donor maps into that architecture.

### 3.2 Function matters more than donor naming
A donor-side number, die size, tier, grade, or ladder position is classified by **job**, not by surface label.

Before Astra admits any donor-side metric, it must determine whether the metric is functioning as:
- a primary attribute,
- a derived stat,
- a defense,
- a resistance or save value,
- a resource,
- a track,
- a supplemental metric,
- or a donor-local special structure.

### 3.3 Not every number is the same kind of thing
Astra does not treat all quantified values on a character sheet as one flat category.

A broad attribute, a passive defense, a corruption meter, a fuel pool, a shield segment track, a social standing metric, and a clock are not identical merely because each can be written down.

### 3.4 Primary attributes are broad foundations
Primary attributes describe broad, persistent actor-side capacities that can lawfully inform many later systems.

They are not intended to replace:
- capabilities,
- access rules,
- active abilities,
- conditions,
- resources,
- or progression structures.

### 3.5 Derived stats must not collapse into monoculture
Derived values may draw from primary attributes, but no single primary attribute should silently become the owner of an unreasonable number of unrelated outputs unless doctrine explicitly authorizes that concentration.

This rule exists to prevent hidden overloading, where one broad stat quietly becomes the king of defense, initiative, perception, ranged offense, stealth, reflex, and half the rest of the game while everyone pretends that was elegant on purpose.

### 3.6 Defensive architecture must be cross-domain
Astra defensive architecture must be able to represent resistance and protection across multiple domains, including:
- physical pressure,
- cognitive pressure,
- social pressure,
- environmental pressure,
- technological pressure,
- and metaphysical or spiritual pressure.

Astra must not assume that one bodily-only or classic three-save structure is always sufficient.

### 3.7 Value forms may vary
Persistent actor-side quantified metrics may be represented as:
- numeric,
- stepped,
- banded,
- categorical,
- or other lawful donor-native value forms.

Astra cares about the function of the metric first. Representation form matters, but it does not decide category by itself.

### 3.8 Full player expression is canonical, not universal
The full player-reference stat architecture is Astra's canonical expression model.

Not every actor must expose the entire model in full. NPCs, hazards, summons, drones, vehicles, minions, abstract obstacles, and reduced-support entities may use compressed, reduced, or specialized stat expressions so long as they still map lawfully into the architecture.

### 3.9 Actors may have multiple stat-bearing frames
An actor may possess one or more stat-bearing frames or overlays.

Examples include:
- a character and a mech,
- a pilot and a vehicle,
- a caster and a bonded spirit,
- a biological body and a cybernetic shell,
- a main actor and an attached proxy,
- or a baseline form and a transformed combat frame.

This file therefore governs both single-frame and multi-frame numerical expressions.

### 3.10 Baseline and active profiles may differ
Some actors have a stable baseline profile but use temporary overlays, transformations, substitutions, possession states, stances, battle shells, form changes, or linked platforms that replace or modify part of the active stat profile.

Astra therefore distinguishes:
- **baseline stat profile** — the actor's ordinary standing numerical expression,
- **active stat profile** — the currently operative expression during play,
- **overlay** — an attached modifier, replacement, or partial substitute that changes some portion of the active profile.

## 4. Primary Attribute Architecture

### 4.1 What a primary attribute is
A **primary attribute** is a broad, persistent quantitative measure of actor-side capacity that:
- represents a foundational dimension of the actor,
- can inform multiple downstream systems,
- is not tied to one narrow action family only,
- and is stable enough to serve as part of Astra's general reference skeleton.

A primary attribute is not merely:
- one skill by another name,
- one resource pool,
- one condition meter,
- one equipment modifier,
- or one donor-specific gimmick with a dramatic title.

### 4.2 What primary attributes must do
Astra's primary attribute layer must be able to support broad functional coverage across at least these job families:
- physical force and applied power,
- endurance, resilience, or bodily stability,
- agility, coordination, or motion control,
- intellect, reasoning, memory, or analytical processing,
- awareness, inner force, resolve, or cognitive-spiritual steadiness,
- and social presence, influence, or outward force of personality.

This file establishes the required job families. Final naming discipline belongs to Astra doctrine and must remain conversion-stable rather than donor-imitative.

### 4.3 Donor systems may be attribute-light
A donor system does not need to expose a classical broad-attribute list in order to map into Astra.

When a donor uses:
- approaches,
- roles,
- drives,
- themes,
- skill pyramids,
- descriptor ladders,
- or other non-classical capability structures,
Astra may still map those structures into a lawful internal attribute scaffold where needed.

The existence of Astra attributes does not require donor-visible one-to-one equivalents.

### 4.4 Primary attributes are not universal action buttons
Primary attributes may inform action resolution, but they do not replace:
- capability architecture,
- access architecture,
- active ability schema,
- or subsystem-specific effect logic.

A broad attribute may help explain *how much* force, resilience, clarity, influence, or control an actor can bring to bear. It does not by itself define every lawful action the actor can perform.

### 4.5 Primary attribute admission rule
A new core primary attribute should only be admitted if all three are true:
- it represents a truly broad and persistent dimension of actor identity or capacity,
- it cannot be handled more cleanly as a derived stat, defense, resource, track, or supplemental metric,
- and multiple future donor systems are likely to pressure the same gap.

If those conditions are not met, the metric should remain supplemental, donor-local, or mapped elsewhere.

## 5. Derived Stat Architecture

### 5.1 What a derived stat is
A **derived stat** is a persistent or semi-persistent quantitative value produced from other parts of the actor architecture.

A derived stat may be shaped by:
- primary attributes,
- actor type,
- frame type,
- equipment state,
- advancement state,
- active overlays,
- or other lawful inputs.

### 5.2 Derived stats are outputs, not substitute foundations
Derived stats exist to express downstream consequences of the actor's structure. They are not a replacement for the primary attribute layer.

### 5.3 Common derived-stat families
Derived stats commonly include values such as:
- vitality or survivability metrics,
- defense values,
- initiative or reaction values,
- perception-like or detection values,
- speed or movement values,
- carrying or operating limits,
- signature or aura-style influence values,
- and other lawful outputs that are too specific to be primary attributes but too persistent to be temporary effects.

### 5.4 Source discipline
When defining a derived stat, Astra should identify:
- what source families can influence it,
- whether it is always-on or context-dependent,
- whether it belongs to the actor, a frame, or an overlay,
- and whether it is persistent, recalculated, or temporarily replaced.

### 5.5 Anti-overload rule
No derived stat should become a dumping ground for unrelated functions merely because it already exists.

If one value is being asked to cover:
- defense,
- initiative,
- stealth resistance,
- perception,
- social poise,
- vehicle handling,
- and ritual control,
then the architecture is no longer disciplined. It is hiding a design problem in a trench coat.

## 6. Defensive Stat Architecture

### 6.1 What a defensive stat is
A **defensive stat** is any persistent or semi-persistent quantified value used to resist, reduce, avoid, absorb, deflect, mitigate, or endure incoming pressure.

A defensive stat may be:
- passive,
- reactive,
- comparative,
- layered,
- or frame-specific.

### 6.2 Defensive values are not all the same family
Astra defensive architecture must be able to represent different defensive modes, including but not limited to:
- evasion or avoidance,
- armor, guard, or soak,
- shielding or barrier values,
- social resistance,
- cognitive resistance,
- environmental resilience,
- technological hardening,
- metaphysical protection,
- and subsystem-specific protective values.

### 6.3 Defense versus save/resist distinction
A defensive stat is not automatically the same thing as a save or resist event.

A defense may be:
- a standing target to overcome,
- a passive benchmark,
- or a persistent mitigating value.

A save or resist procedure is a resolution event governed by `batchA_03_resolution_framework.md`.

This file governs the actor-side numerical architecture that such procedures may reference.

### 6.4 Cross-domain requirement
Astra must not assume that physical threat, toxin threat, social pressure, hacking intrusion, psychic assault, corruption pressure, and reality distortion all route through one identical defense model.

Different domains may require different defensive values, resistance families, or mapping logic.

## 7. Tempo, Mobility, and Awareness Stats

### 7.1 Tempo values
Astra recognizes a class of stat structures that govern action order, readiness, response timing, or operational tempo.

These may include:
- initiative-like values,
- reaction-related values,
- turn-order influence,
- processing speed,
- pilot sync speed,
- or comparable donor-native tempo expressions.

### 7.2 Mobility values
Astra recognizes a class of stat structures that govern movement, repositioning, travel pace, maneuverability, chase viability, locomotion mode, or comparable action-space control.

These values may belong to:
- the actor,
- a vehicle or platform frame,
- a transformed form,
- or a temporary mobility overlay.

### 7.3 Awareness values
Astra recognizes a class of stat structures that govern perception, detection, notice, signal interpretation, situational reading, sensory acuity, or comparable awareness functions.

Awareness values may exist in:
- active forms,
- passive forms,
- or subsystem-specific derivatives.

### 7.4 Distinction rule
Tempo, mobility, and awareness values may be influenced by primary attributes, but they are not themselves automatically primary attributes.

They should be admitted as primary attributes only if they meet the broad-foundation standard defined in Section 4.

## 8. Save and Resistance Family Architecture

### 8.1 Save and resistance values are architecture-level families
Astra requires a lawful way to represent resistance and save families across different threat types.

These families may correspond to pressures such as:
- bodily trauma or bodily corruption,
- agility or positional danger,
- mental or emotional pressure,
- social coercion or influence,
- environmental hostility,
- technological intrusion,
- spiritual or metaphysical disruption,
- or other donor-native pressure families.

### 8.2 Family doctrine over fixed donor ritual
This file defines the need for resistance/save families and their classification discipline. It does not permanently freeze one donor's exact trio, quartet, or ladder as universal law.

### 8.3 Family admission rule
A resistance/save family should be recognized at architecture level only if:
- it corresponds to a recurring pressure class across multiple donor types,
- it is not more cleanly handled as one isolated defense or subsystem-specific metric,
- and it improves clarity rather than multiplying redundant categories.

### 8.4 Cross-domain support rule
Astra resistance architecture must be able to support physical, cognitive, social, environmental, technological, and metaphysical forms of resistance without assuming all such pressures reduce to one bodily-only scheme.

## 9. Supplemental and Exceptional Metrics

### 9.1 What a supplemental metric is
A **supplemental metric** is a persistent or semi-persistent quantified value that matters to the actor but does not qualify as a core primary attribute.

Examples may include values analogous to:
- luck,
- aura or presence signatures,
- sanity or stability,
- corruption load,
- reputation pressure,
- operating heat,
- trace level,
- signal integrity,
- notoriety,
- fate pressure,
- or other cross-system oddities.

### 9.2 Exceptional metrics are lawful, but not automatically core
Astra allows exceptional metrics where donor pressure warrants them. It does not automatically promote them into universal core doctrine.

### 9.3 Supplemental metrics must declare their class
Any supplemental metric admitted into doctrine must explicitly state whether it behaves primarily as:
- a persistent stat,
- a defense-like value,
- a resource-linked value,
- a state-linked value,
- a track-like value,
- a signature metric,
- or a donor-local special structure.

### 9.4 Promotion restraint rule
A supplemental metric should only become a universal core value if repeated donor pressure proves that it is broadly necessary and cleanly distinct from existing core architecture.

Otherwise it remains supplemental or donor-local.

## 10. Stats, Tracks, Resources, and State Separation

### 10.1 Not every persistent quantified value is a stat
Astra distinguishes between:
- **stats** — broad persistent capacities or structured derived values,
- **resources** — quantities intended to be spent, recovered, consumed, charged, depleted, or replenished,
- **tracks** — progression, damage, corruption, countdown, shield-segment, trace, stress, or state ladders that record ongoing change,
- **states** — current conditions or effect-bearing statuses,
- and **supplemental metrics** — lawful persistent quantities that do not fit the above cleanly.

### 10.2 Misclassification rule
If a value primarily exists to be:
- spent,
- filled,
- emptied,
- advanced along a sequence,
- or marked across changing segments,
then it may not belong in the same class as a primary attribute even if it is written on the same sheet.

### 10.3 Ownership rule
Tracks, resources, and states may interact with the stat architecture, but they are not owned by this file unless the question is specifically about classification boundaries.

## 11. Multi-Frame Actors and Overlay Doctrine

### 11.1 Actor-level versus frame-level metrics
An actor may possess metrics that belong to:
- the actor as a whole,
- one specific frame,
- an attached platform,
- a companion frame,
- a summoned proxy,
- a vehicle shell,
- or a temporary combat or transformation overlay.

The file owning a later subsystem may decide how such structures are used, but this file governs the lawfulness of their classification.

### 11.2 Multiple stat-bearing frames are legal
Astra recognizes that one playable entity may lawfully involve multiple stat-bearing frames.

Examples include:
- pilot plus vehicle,
- character plus mech,
- summoner plus bound construct,
- caster plus bonded spirit,
- baseline body plus transformed battle shell,
- or biological frame plus cybernetic combat overlay.

### 11.3 Overlay behavior
An overlay may:
- add to the baseline profile,
- modify the baseline profile,
- suppress part of the baseline profile,
- replace a subset of values,
- or temporarily substitute a different active profile.

### 11.4 Replacement and substitution discipline
When a temporary form, shell, possession state, combat mode, or linked platform changes active stats, Astra should record whether the change is:
- additive,
- substitutive,
- partial replacement,
- or total profile swap.

This prevents later conversion from treating every transformation as the same kind of numerical event.

## 12. Reduced and Specialized Actor Expressions

### 12.1 Full expression is not mandatory for every entity
The full player-reference stat architecture is canonical for doctrine. It is not mandatory for every non-player entity to expose every layer in full.

### 12.2 Reduced expressions are lawful
NPCs, minions, hazards, drones, vehicles, summoned entities, scene structures, and abstract obstacles may use reduced or compressed expressions if they still map lawfully into Astra's broader architecture.

### 12.3 Specialized expressions are lawful
Some entities may expose highly specialized stat structures because their role is narrow.

Examples may include:
- vehicles with mobility and integrity emphasis,
- hazards with threshold and consequence emphasis,
- drones with reduced social expression,
- abstract scenes with pressure metrics but no full actor skeleton.

### 12.4 Reduction does not erase mapping discipline
Even when an entity uses a reduced or specialized expression, its values should still be classifiable by Astra architecture rather than invented ad hoc every time.

## 13. Donor-Stat Mapping Rules

### 13.1 Classify before admitting
When a donor introduces a new quantified value, Astra must classify it before deciding where it belongs.

The first question is not “what cool name does it have?”
The first question is “what job is it doing?”

### 13.2 Donor-metric classification sequence
A donor-side metric should be tested against this order:
1. Is it a broad primary attribute?
2. Is it a derived stat?
3. Is it a defense?
4. Is it part of a resistance/save family?
5. Is it a resource?
6. Is it a track?
7. Is it a supplemental metric?
8. Is it a donor-local special structure that should not be promoted?

### 13.3 Representation-form neutrality
A donor-side metric may arrive as:
- a flat number,
- a step die,
- a grade,
- a tier,
- a band,
- a rank label,
- or another categorized value form.

Representation form does not determine its category by itself.

### 13.4 Admission outcomes
After classification, a donor metric should be assigned one of four outcomes:
- **core-mapped** — maps cleanly into existing Astra architecture,
- **supplemental-admitted** — recognized but not core,
- **specialized-mapped** — lawful only within a specific subsystem or frame type,
- **donor-local** — retained only for donor integrity and not promoted into Astra core doctrine.

### 13.5 Pollution guardrail
A donor stat must not be promoted into Astra core architecture merely because:
- the donor names it dramatically,
- it appears in one popular system,
- it feels thematic,
- or it is convenient in the moment.

Promotion requires repeat pressure and clean classification.

## 15. Operational Summary

Astra uses a stable internal stat architecture.

That architecture:
- recognizes broad primary attributes without requiring donors to expose identical labels,
- distinguishes primary attributes from derived stats, defenses, resources, tracks, and supplemental metrics,
- supports multiple value forms,
- supports multi-frame actors and temporary overlays,
- supports reduced and specialized entity expressions,
- and requires donor metrics to be classified before admission.

The purpose of this file is not to make every sheet identical.
The purpose is to make every lawful sheet classifiable.
