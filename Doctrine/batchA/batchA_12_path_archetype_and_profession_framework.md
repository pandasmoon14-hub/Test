# batchA_12_path_archetype_and_profession_framework

This file defines Astra doctrine for the **current package layer**: path, archetype, school/tradition, profession, and bond/platform package architecture.

Its purpose is to prevent donor systems from collapsing current role, training method, vocation package, bonded platform logic, identity layers, and progression law into one bundled package object.

This file is authoritative for:
- package grammar

- package coordination law

- multiclass / hybrid package law
- package grant boundaries

- kinform, heritage, culture, homeland, upbringing, or formative identity layers
- chargen order
- core access law
- core capability law
- core ability schema

## 3. Core package principles

### 3.1 Packages are coordinators, not sovereign owners

Packages may coordinate grants across other Astra doctrine layers, but they do not own or redefine those layers.

A package may reference or coordinate:
- access structure from `batchA_06_access_tags_permissions_and_proficiency_gates.md`
- capability emphasis from `batchA_05_competency_and_skill_translation_architecture.md`
- ability selections from `batchA_07_ability_object_model.md`
- resource consequences from `batchA_07b_resource_cost_recharge_and_backlash_architecture.md`
- progression hooks from `batchA_10_progression_axes.md`
- identity prerequisites or synergies from `batchA_11a` and `batchA_11b`

A package may not silently replace those files as their owner.

### 3.2 Package function outranks donor label

A donor may call a package a class, playbook, archetype, doctrine, occupation, sect art, discipline, role, office, or path.

Astra classifies the package by **function**, not donor noun.

### 3.3 Current package layers are not formative identity layers

A package may build on identity layers from `11a` and `11b`, but it may not silently absorb:
- kinform
- heritage
- culture
- homeland
- upbringing
- event history
- formative affiliation

### 3.4 Packages shape play without becoming total identity

Packages matter.
They may strongly influence playstyle, grant emphasis, and define structured current-role expression.
They do not become the whole person by default.

### 3.5 Package porosity is lawful, but must be explicit

Some packages are exclusive.
Some are porous.
Some allow grafts.
Some are transitional.
Some are parallel.
Some are commitment-gated.

Astra must define this as law rather than leaving it to tone.

## 4. Package function grammar

Astra recognizes at least five distinct package functions.

### 4.1 Primary role package

A **primary role package** is the actor’s current operational play spine.

It answers:
- what structured role the actor presently inhabits
- what the actor broadly does in play
- what coordinated grant pattern defines the actor’s main current package identity

Examples of role pressure include battlefield controller, bonded hunter, plague manipulator, shadow infiltrator, ritual duelist, void surveyor, beast commander, field healer, and similar structured roles.

### 4.2 Archetype graft

An **archetype graft** is a secondary modifier, branch, overlay, or specialization layer attached to another package structure.

It answers:
- how a current package is bent, branched, specialized, or expanded
- what secondary emphasis is grafted onto a primary role or profession package
- what commitment or completion rules govern the graft

### 4.3 School / tradition package

A **school / tradition package** is a method package.
It represents a structured training doctrine, school, sect art, martial stream, ritual discipline, academy track, clan method, or comparable approach layer.

It answers:
- what doctrine or method the actor uses
- what practice stream or training logic shapes the package
- what method-bound grants or progression hooks exist

### 4.4 Profession package

A **profession package** is the current-role package for vocation-centered donors.

It is **not** the same as profession-before-adventure in `batchA_11b`.

It answers:
- what current practical vocation-structured package the actor occupies now
- what organized field role, license-bearing trade, or structured work identity defines current play

### 4.5 Bond / platform package

A **bond / platform package** is the package type for characters whose mechanical center is relational, bonded, companion-based, frame-based, or platform-based.

This may include:
- bonded beasts
- mechs
- drones
- relic symbiosis
- summoned frameworks
- living weapons
- ship or vehicle frames
- companion-command packages
- platform-mediated doctrines

This package type exists so bond- or platform-centered donors do not get flattened into ordinary role packages or leak into identity / progression / ability files.

## 5. Role layer vs school/tradition layer vs profession layer vs bond/platform layer

These layers may coexist, but they do not do the same job.

- **Role layer** defines current operational play spine.
- **School / tradition layer** defines method or doctrinal approach.
- **Profession layer** defines current vocation-shaped package where applicable.
- **Bond / platform layer** defines relational or platform-centered package logic.
- **Archetype graft** modifies or branches one or more of the above.

A donor package that bundles multiple such jobs must be split or normalized rather than preserved intact out of convenience.

## 6. Package coordination model: grants by reference, not ownership

Packages coordinate grants across other doctrine layers.

### 6.1 Access coordination

A package may grant curated access bundles, access tags, licenses, permissions, schools, armament permissions, or attunement hooks.
Those grants are still governed by `batchA_06`.

### 6.2 Capability coordination

A package may emphasize or bundle capability selections, specialties, profession-linked know-how, or curated capability growth.
Those grants are still governed by `batchA_05`.

### 6.3 Ability coordination

A package may grant starting abilities, ability access, upgrade branches, prepared selections, doctrine lists, or curated ability families.
Those grants are still governed by `batchA_07`.

### 6.4 Resource-economy coordination

A package may alter cadence, upkeep burden, reserved capacity, substitution rules, or package-specific cost hooks.
Those consequences are still governed by `batchA_07b`.

### 6.5 Progression coordination

A package may define progression hooks, unlock cadence references, package-bound branches, or eligibility conditions.
Those relations are still governed by `batchA_10`.

### 6.6 Identity interaction

A package may require, reference, or synergize with kinform, heritage, culture, upbringing, or formative imprint.
Those identity layers remain governed by `11a` and `11b`.

## 7. Package grant law

Packages may lawfully grant combinations of:
- structured access bundles
- curated ability selections
- curated capability emphasis
- defensive profile emphasis
- loadout affinities
- package-bound presentation markers
- progression hooks
- bond / platform relations
- package-specific upgrade branches
- lawful compatibility notes with other package layers

Packages may also define:
- package entry conditions
- package maintenance expectations
- package conflict rules
- package coexistence rules
- package-specific narrative identity markers

## 8. Package grant prohibitions

Packages may **not** grant by default:
- total biological identity
- full culture package
- homeland or upbringing replacement
- entire event-history identity
- unrestricted universal access
- whole progression architecture by themselves
- total replacement of supporting progression axes
- unrestricted command authority
- full doctrine-layer ownership over access, capability, abilities, resources, conditions, or damage
- silent replacement of `11a`, `11b`, `05`, `06`, `07`, `07b`, or `10`

A package may coordinate those layers.
It may not redefine their owning doctrine.

## 9. Package rigidity, acquisition timing, and commitment

Astra recognizes that packages differ by how rigidly and when they are acquired.

### 9.1 Acquisition timing

A package may be:
- start-only
- early-eligible
- milestone-eligible
- progression-threshold gated
- access-gated
- narrative-event gated
- bond / platform acquisition gated
- doctrine-school entry gated

### 9.2 Rigidity states

A package may be:
- exclusive
- porous
- graftable
- transitional
- parallel-compatible
- commitment-bound
- temporary
- replaceable
- partially layered

### 9.3 Commitment law

Some packages require commitment before another of the same type may be added, replaced, or exited.
Astra recognizes dedication-like and completion-like commitment structures where appropriate.

## 10. Package relationships

Astra recognizes at least the following lawful package relationships:
- exclusive package
- primary package with secondary graft
- layered parallel packages
- transitional package progression
- package family branching
- package replacement with continuity rule
- package coexistence with conflict restrictions

Package relationships must be explicit.
They may not be inferred by vibes.

## 11. Multiclass / archetype grafting / hybrid package law

Astra allows package hybridization, but not as mush.

A hybrid or multi-package character must still classify grants into proper Astra layers.
Adding a second package does not authorize silent collapse of:
- access into package identity
- capability into package flavor
- ability families into unnamed “class features”
- progression into one blob

Hybrid package handling must answer:
- what is primary
- what is secondary
- what is grafted
- what conflicts
- what is delayed
- what remains source-local

## 12. Package-bound progression hooks

Packages may lawfully reference:
- primary ascension ladder thresholds
- supporting progression axes
- package-specific unlock milestones
- role mastery steps
- school / doctrine deepening
- profession certification stages
- bond / platform maturation
- package-specific branch decisions

Packages do not own progression law.
They own only their hooks into it.

## 13. Companion, platform, and nonstandard package holders

Packages are not restricted to ordinary humanoid holders.

Astra explicitly supports packages centered on:
- companions
- bonded beasts
- constructs
- AI shells
- drones
- relic symbiosis
- living weapons
- mechs
- ships
- vehicles
- summoned frameworks
- distributed or shared holders

A package holder, package source, and delivery platform may differ.

## 14. Donor mapping rules

For every donor package, the converter must ask:

1. Is its primary job current role, method / tradition, profession, or bond / platform?  
   - Classify it accordingly.

2. Does it include secondary grants in access, capability, ability, defense, loadout, or progression?  
   - Route those grants through the owning Astra layers.

3. Is the donor package exclusive, porous, graftable, transitional, or parallel?  
   - Record that explicitly.

4. Is any part of the donor package actually identity/background material?  
   - If yes, route that portion to `11a` or `11b`.

5. Is any part of the donor package actually source-local presentation only?  
   - If yes, preserve it as presentation marker rather than doctrine law.

6. Is the donor package relational or platform-centered?  
   - If yes, consider bond / platform package classification instead of flattening it into a general role package.

7. Is a donor package doing more than one package-function job?  
   - If yes, split or normalize it rather than preserving the fusion intact.
