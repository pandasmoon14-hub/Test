## Authority
This file is the authoritative doctrine file for **Astra conversion invariants and non-negotiable mapping law**.

It defines:
- the non-overwrite law by which Astra doctrine outranks donor assumptions,
- the mandatory classification discipline for all donor constructs before mapping,
- the lawful difference between direct mapping, normalized mapping, source-local retention, quarantine, and rejection,
- the split-versus-merge rules that prevent donor constructs from collapsing multiple Astra layers,
- the preservation rules that determine what a conversion should preserve and what it must not preserve intact,
- the source-local containment and doctrine-promotion rules,
- the mandatory uncertainty response when classification would require invention,
- the escalation triggers for doctrine refinement,
- and the conversion checklist that all later donor-conversion work must obey.

It does **not** provide:
- a giant worked example library,
- full donor-specific conversion case studies,
- live-play adjudication advice,
- or downstream subsystem design that properly belongs to later files.

If any later Batch A or donor-conversion file appears to contradict this file on the **lawful handling of donor material**, this file governs unless deliberately revised by Astra doctrine.

---

## 1. Purpose

Astra is being built in a staged order:
- doctrine and framework first,
- donor conversion second,
- canon consolidation third,
- live-play behavior later.

That order only remains stable if conversion work is constrained by hard law.

This file exists to prevent five recurring failure modes:
- **donor overwrite**, where repeated donor assumptions quietly become Astra default law,
- **layer collapse**, where one donor construct smuggles capability, access, ability, progression, or identity into one fused blob,
- **false preservation**, where donor flavor is preserved by importing donor structure wholesale,
- **decorative improvisation**, where missing doctrine is patched with Astra-sounding filler instead of lawful escalation,
- and **silent promotion**, where source-local donor behavior becomes framework doctrine without an explicit adoption decision.

This file is therefore not advisory etiquette.
It is the enforcement layer for all later conversion work.

## 3. Core Conversion Principles

### 3.1 Convert into Astra
Convert donor material into Astra-native form.
Do not import donor metaphysics, donor terminology, donor ontology, donor structure, or donor social assumptions wholesale unless Astra doctrine explicitly adopts them.

### 3.2 Doctrine outranks donor material
Astra doctrine is the governing authority.
Donor texts are pressure sources and conversion inputs, not peer legislators.

### 3.3 Preserve function before packaging
When converting, preserve donor **function** and **play intent** where possible, but express them through Astra architecture rather than donor form.

What Astra should preferentially preserve, in descending priority, is:
1. lawful mechanical function,
2. lawful subsystem role,
3. lawful progression role,
4. lawful access role,
5. lawful narrative role,
6. source-local flavor,
7. donor metaphysical packaging.

If donor metaphysical packaging conflicts with Astra doctrine or distorts Astra architecture, packaging loses.

### 3.4 Classification before naming
A donor label is never the final truth.
A donor construct must be classified by Astra job before its donor name can influence conversion language.

### 3.5 No silent promotion
A donor-local pattern may not become Astra doctrine merely because:
- it appears repeatedly in one donor corpus,
- it feels reusable,
- it sounds elegant,
- or it is mechanically convenient.

Promotion requires explicit Astra doctrine adoption.

### 3.6 Uncertainty is not permission to improvise
If a donor construct cannot be classified without inventing new doctrine, the correct action is escalation or quarantine, not decorative Astra-flavored improvisation.

## 4. Canon and Override Invariants

### 4.1 Authority hierarchy
For conversion purposes, authority is applied in this order:
1. current Astra doctrine files,
2. canonical Astra sourcebook material once formalized,
3. converted Astra-native donor content,
4. example packs or demonstrations,
5. original donor assumptions.

### 4.2 Override law
If a donor assumption conflicts with current Astra doctrine, Astra doctrine wins.

If a converted source conflicts with canonical Astra sourcebook material once formalized, canon wins.

If an example conflicts with doctrine or canon, the example loses.

### 4.3 Converted content is not self-canonizing
A converted donor construct does not become Astra doctrine merely because it has been converted.

### 4.4 Repetition does not create authority
Repeated donor patterns do not become Astra law by repetition alone.
This is especially important when one donor corpus is currently the strongest pressure source.

## 5. Mandatory Classification Invariants

### 5.1 Every donor construct must be classified by Astra job
Before mapping, every donor construct must be classified into one or more Astra-owned jobs such as:
- player/entity ontology,
- identity/origin layer,
- stat architecture,
- capability,
- access,
- ability object,
- resource economy,
- condition/effect,
- damage family,
- progression axis,
- loadout/platform/infrastructure,
- social/institutional state,
- or source-local remainder.

### 5.2 Donor names are provisional until classification completes
Terms such as class, skill, feat, discipline, domain, path, lineage, essence, soul, level, corruption, reputation, or similar donor labels remain provisional until Astra classification determines their lawful function.

### 5.3 One donor construct may require multiple classifications
If a donor construct performs more than one job, it must be decomposed and classified accordingly.

## 6. Split-Versus-Merge Invariants

### 6.1 Multi-job donor constructs must be split
If a donor construct performs multiple jobs that Astra assigns to different layers, split it.
Do not import the fused donor object intact.

### 6.2 Separate Astra layers must not be collapsed for donor convenience
Do not merge:
- capability with access,
- capability with ability,
- access with ability,
- progression axes into one false ladder,
- identity layers into one ancestry-culture-background blob,
- condition meaning into damage-family meaning,
- or cost/recharge law into effect payload.

### 6.3 Source-local fusion is allowed only when lawful
A donor may keep a source-local fused presentation if:
- Astra classification has already been completed,
- each underlying job is still lawfully represented,
- and the source-local fused presentation does not overwrite doctrine ownership.

## 7. Lawful Conversion Output States

Every converted construct must end in one of the following lawful output states.

### 7.1 Direct Astra mapping
The donor construct maps cleanly into an existing Astra doctrine slot with no major normalization required.

### 7.2 Normalized Astra mapping
The donor construct is preserved functionally but must be redistributed across Astra layers, axes, or schema fields in order to fit Astra architecture.

### 7.3 Source-local retained construct
The donor construct remains source-local after conversion.
It is expressed in Astra-compatible form but is not promoted to general doctrine.

### 7.4 Quarantined pending doctrine
The donor construct cannot be mapped lawfully without new doctrine or without risking contradiction, so it is held pending refinement.

### 7.5 Rejected as incompatible or out-of-scope
The donor construct is explicitly rejected because it conflicts too strongly with doctrine, violates project scope, or cannot be made lawful without unacceptable distortion.

### 7.6 Output-state rule
No donor construct may remain in an undefined “sort of converted” state.

---

## 8. Preservation Invariants

### 8.1 Preserve job, not costume
The converter should preserve the donor construct's job and play intent before preserving its donor presentation form.

### 8.2 Mechanical compatibility outranks donor metaphysical packaging
If the donor's metaphysical explanation conflicts with Astra doctrine but the donor mechanic can be lawfully expressed through Astra architecture, preserve the mechanic and rewrite the packaging.

### 8.3 Flavor may remain source-local
Source flavor may be preserved where useful, but source flavor does not authorize doctrinal overwrite.

### 8.4 Preservation is selective, not total
A lawful conversion may preserve:
- function,
- pacing,
- access role,
- progression pressure,
- risk profile,
- or scene role,
without preserving the donor's exact ontology, terminology, cosmology, or social assumptions.

---

## 9. Source-Local Containment and Promotion Rules

### 9.1 Source-local is the default for donor-specific behavior
If a donor-specific mechanic, assumption, or flavor pattern is useful but not broadly doctrinal, retain it source-locally.

### 9.2 Promotion requires explicit adoption
A donor-local behavior may only be promoted into Astra doctrine if it is:
- repeated across relevant donor pressure,
- functionally useful for future conversion,
- compatible with current doctrine,
- and explicitly adopted by doctrine revision.

### 9.3 No promotion by convenience
A construct is not promoted merely because promotion would make the current conversion easier.

### 9.4 No promotion by repetition inside one donor corpus
A pattern repeated many times inside one donor corpus is still donor-local until Astra says otherwise.

## 10. Mandatory Uncertainty Response and Escalation Triggers

### 10.1 Mandatory uncertainty response
If a donor construct cannot be classified without inventing new doctrine, the converter must:
- escalate for doctrine refinement,
- or quarantine the construct.

The converter must not patch the gap with invented Astra-sounding filler.

### 10.2 Escalation triggers
Escalate when any of the following occur:
- the donor construct requires a new core concept not yet owned by Astra doctrine,
- the donor construct would force collapse of distinct Astra layers,
- the donor construct conflicts directly with existing doctrine,
- the donor construct depends on donor cosmology as if it were universal law,
- the donor construct cannot be preserved functionally without redefining existing Astra terms,
- the donor construct produces unresolved contradiction across multiple files,
- or the donor construct pressures a repeated architecture gap rather than a one-off local quirk.

### 10.3 Quarantine rule
Quarantine is lawful when escalation is required but doctrine is not yet revised.
Quarantine is not failure; it is anti-hallucination discipline.

## 11. Prohibited Conversion Shortcuts

The following shortcuts are prohibited.

### 11.1 No donor cosmology leakage
Do not import donor cosmology as hidden Astra baseline.

### 11.2 No donor terminology leakage without classification
Do not preserve donor nouns as if they are self-validating Astra doctrine terms.

### 11.3 No structural donor overwrite
Do not import donor architecture wholesale merely because it is internally elegant inside the donor.

### 11.4 No fake Astra filler
Do not invent Astra-sounding bridge language to cover missing framework structure.

### 11.5 No silent ladder flattening
Do not flatten multi-axis donor progression into one false ladder when Astra architecture supports a better mapping.

### 11.6 No silent object fusion
Do not preserve fused donor objects when their jobs belong to different Astra files unless the fusion is explicitly source-local and fully classified.

### 11.7 No example-to-law drift
Do not let a worked example or converted source become governing doctrine by implication.

## 12. Mandatory Conversion Questions

Every meaningful donor construct should be tested against the following questions.

1. What job does this construct perform in Astra?
2. Is it one job or several?
3. Which Astra file or files own those jobs?
4. What donor assumptions are source-local only?
5. What function or play intent must be preserved?
6. What donor packaging may be discarded or rewritten safely?
7. Does this map directly, require normalization, remain source-local, require quarantine, or require rejection?
8. Does this contradict current Astra doctrine?
9. Would converting this as-is cause layer collapse, false promotion, or hidden cosmology leakage?
10. If doctrine is insufficient, is the correct next action escalation or quarantine?

A conversion that cannot answer these questions cleanly is not yet lawful.

## 13. RHBF as Primary Pressure Example

RHBF is an important donor pressure source precisely because it demonstrates many of the failure modes this file is designed to prevent.

Examples of pressure it creates include:
- cosmology-specific ascension through forced planetary assimilation,
- donor resource and ontology terms treated as universal law,
- activated power objects labeled as “Skills,”
- Domains doing access work under a donor-specific noun,
- progression split across multiple interacting ranks,
- strong donor social assumptions about power and rulership,
- and repeated donor patterns that are easy to mistake for doctrine simply because they are rich and consistent.

This file exists so those pressures can be translated, not inherited blindly.

## 14. Downstream Relationship

This file governs all later donor-conversion work and constrains the interpretation of later Batch A files when they are used in conversion.

In particular, it is intended to constrain the lawful use of:
- origin and identity frameworks,
- path/archetype/profession frameworks,
- character-creation procedure,
- loadout and kit frameworks,
- and the later translation-pattern library.

No downstream conversion file may overrule the invariants here by implication.

---

## 15. Summary Law

A lawful Astra conversion must:
- classify before naming,
- preserve function before donor packaging,
- split multi-job donor constructs when Astra layers differ,
- contain donor-specific assumptions source-locally unless explicitly promoted,
- escalate or quarantine when doctrine is insufficient,
- and obey Astra authority hierarchy at all times.

Anything less is not Astra conversion discipline.
