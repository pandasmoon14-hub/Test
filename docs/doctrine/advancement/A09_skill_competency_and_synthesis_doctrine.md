# A09 Skill, Competency, and Synthesis Doctrine

## 1. Purpose and status

A09 defines doctrine-level skill/competency/proficiency architecture for Astra and explains how competency pressure is classified, retained, routed, quarantined, or escalated.

Status: draft. This file is not current doctrine law.

A09 is a doctrine boundary file. It provides synthesis posture and conversion discipline without importing donor mechanical defaults.

## 2. What this file owns

A09 owns:

- skill construct and competency construct framing;
- proficiency posture framing;
- training vector framing;
- cross-skill synthesis posture;
- multi-domain check posture;
- non-active capability translation posture;
- professional and cultural competency mapping posture.

## 3. What this file must not own

A09 must not define:

- a canon skill list;
- specific skill names;
- proficiency bonus math;
- point-buy costs;
- training time formulas;
- crafting recipes;
- faction reputation systems;
- technique mastery;
- techniques, spells, or powers;
- resource costs or resource pools;
- action economy;
- damage formulas;
- conditions;
- actor stat blocks;
- XP tables or level tables;
- runtime skill/progression state.

## 4. Required definitions

A09 uses the following doctrine terms:

- skill construct;
- competency construct;
- proficiency posture;
- training vector;
- cross-skill synthesis;
- multi-domain check;
- passive capability translation;
- professional competency mapping;
- cultural competency mapping;
- source-local competency construct;
- quarantined competency construct;
- escalated competency contradiction;
- incompatible synthesis;
- constrained synthesis;
- unstable synthesis;
- source-local competency expression only.

## 5. Core doctrine rules

1. Donor skill/proficiency systems can be retained as source-local records but are not Astra default law.
2. Source-local records preserve provenance and can inform pressure tests without becoming canon.
3. Cross-skill synthesis is qualitative at A09 scope and must not be converted into numeric optimization math.
4. Multi-domain checks are routing posture only at A09 scope and must not define executable runtime rules.
5. Professional/cultural mappings remain doctrine framing and must not become faction or economy mechanics.
6. Unclassifiable competency claims are quarantined or escalated, not normalized by invention.
7. Donor proper nouns remain source-local unless later canon promotion accepts them.
8. A09 cannot absorb K01 ownership; lexicon governance remains outside A09.

## 6. Conversion mapping rules

When converting donor material into A09-compatible doctrine:

- retain donor-specific skill lists as source-local competency expressions;
- translate donor proficiency claims into qualitative proficiency posture only;
- map profession/career/archetype competency claims into training vectors and competency mapping posture;
- route mixed competency-plus-technique claims to A08/A10 boundaries without defining technique or resource mechanics;
- quarantine contradictory competency systems when they require forbidden mechanical detail.

## 7. Source-local handling

- Donor skill lists, tags, careers, and proficiencies may be retained source-local for traceability.
- Source-local competency records do not become accepted lexicon terms through A09.
- Repeated donor pressure can produce canon candidates only through later K-layer/canon review.
- Source-local capability translation claims must identify provenance and remain non-authoritative.

## 8. Donor pressure absorbed

A09 is explicitly designed to absorb pressure from:

- class/level skill proficiency packages;
- point-buy competency allocation models;
- aspect/tag competency abstractions;
- lifepath profession and career competency systems;
- cultivation profession and craft competency claims.

A09 refuses silent adoption and retains only doctrine-compatible conversion posture.

## 9. Hard refusals / rejected imports

A09 refuses the following imports as Astra defaults:

- donor skill lists as Astra law;
- donor proficiency bonus ladders;
- donor point-buy cost tables;
- donor training-time arithmetic;
- donor faction-reputation progression tied to skill advancement;
- donor technique/spell/power systems disguised as skill systems.

## 10. Escalation triggers

Escalate instead of normalizing in A09 when:

- a new competency category is required beyond current doctrine framing;
- synthesis claims contradict A07 advancement boundaries;
- synthesis claims require A08 technique mastery definition;
- synthesis claims require A10 resource/cost/action-economy definition;
- competency mapping conflicts with downstream actor/economy doctrine assumptions.

## 11. Dependencies

A09 depends on:

- A07_advancement_axes_and_progression_pressure.md
- A08_path_domain_and_technique_mastery_doctrine.md

A09 does not redefine A07 advancement-axis taxonomy, A08 technique mastery, A08 activation/targeting grammar, resource systems, damage families, condition/status rules, actor stats, runtime event commits, runtime actor state, or hidden-information behavior.

## 12. Handoff to downstream layers

A09 provides doctrine constraints and routing posture for A10, A11, C01, C02, C03, and K01.

A09 hands off:

- competency pressure that requires cost/resource/action-economy treatment to A10;
- competency claims requiring actor ontology integration to A11;
- competency language and doctrine constraints to C-layer consumers;
- lexicon pressure signals to K01 without migrating ownership.

## 13. Test cases / pressure examples

A09 pressure tests should include at least:

1. A donor class-skill package that attempts to import fixed proficiency-bonus math.
2. A point-buy competency model that attempts to import explicit cost tables.
3. A profession/career mapping that attempts to become faction reputation mechanics.
4. A cross-skill synthesis claim that implicitly requires A08 technique or A10 resource mechanics.
5. A mixed multi-domain check that pressures actor/runtime state definition.

Expected posture: retain source-local evidence, refuse forbidden definitions, quarantine/escalate contradictions, and preserve downstream boundaries.

## 14. Versioning and review protocol

- Increment this file only for doctrine boundary clarity, conversion routing precision, and contradiction-handling posture within A09 scope.
- Do not promote A09 to current without satisfying registry promotion requirements.
- Amendments must preserve A09/K01 ownership separation and A07/A08 dependency boundaries.
