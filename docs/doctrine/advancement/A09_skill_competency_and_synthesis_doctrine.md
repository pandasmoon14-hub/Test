# A09 Skill, Competency, and Synthesis Doctrine

## 1. Purpose and status

A09 defines doctrine-level skill/competency architecture for Astra and explains how competency pressure is classified, retained, routed, quarantined, or escalated.

Status: draft. This file is not current doctrine law.

## 2. What this file owns

A09 owns doctrine framing for:

- skill constructs and competency constructs;
- proficiency/expertise/specialization posture;
- training vectors;
- cross-skill synthesis and multi-domain checks;
- non-active capability translation;
- professional and cultural competency mapping.

## 3. What this file must not own

A09 must not define:

- specific canon skill list;
- specific skill names as Astra defaults;
- specific proficiency bonus math;
- specific point-buy costs;
- specific training time formulas;
- specific crafting recipes;
- specific professional rank ladders;
- specific faction reputation systems;
- technique mastery;
- techniques, spells, or powers;
- resource costs;
- resource pool mechanics;
- action economy;
- damage formulas;
- conditions;
- actor stat blocks;
- XP tables or level tables;
- runtime skill instance state;
- runtime progression state;
- runtime event commits;
- hidden information state;
- accepted lexicon terms;
- donor skill systems as Astra default law.

## 5. Core doctrine rules

1. Donor systems can be retained as source-local records but are not Astra default law.
2. Cross-skill synthesis is qualitative at A09 scope; no numeric optimization math is defined.
3. Multi-domain checks are routing posture only at A09 scope; no runtime rule execution is defined.
4. Unclassifiable skill/competency claims are quarantined or escalated, not normalized by invention.
5. A09 cannot absorb K01 ownership; lexicon governance remains outside A09.

## 6. Conversion mapping rules

- Retain donor-specific skill lists as source-local skill constructs.
- Translate donor proficiency claims into qualitative proficiency/expertise posture only.
- Map profession/career/lifepath claims into training vectors and competency mapping posture.
- Route mixed competency-plus-technique claims to A08/A10 boundaries without defining technique or resource mechanics.
- Quarantine contradictory competency systems when they require forbidden mechanical detail.

## 7. Source-local handling

- donor skill/competency/proficiency/profession/career/lifepath/aspect systems can be retained as source-local records.
- source-local records preserve donor provenance but do not become Astra skill canon.
- repeated donor pressure can produce canon candidates only through later K-layer/canon review.
- unclassifiable skill/competency systems are quarantined or escalated, not normalized by invention.
- donor proper nouns remain source-local unless canon promotion accepts them later.
- source-local skill/proficiency/profession terms do not become accepted lexicon terms through A09.

## 8. Donor pressure absorbed

A09 absorbs pressure from class/level skill packages, point-buy competency allocation, aspect/tag abstractions, lifepath careers, and cultivation profession/craft competency claims while refusing silent adoption.

## 9. Hard refusals / rejected imports

A09 refuses donor skill lists, proficiency ladders, point-buy cost tables, training-time arithmetic, faction-reputation progression tied to skill advancement, and technique/spell/power systems disguised as skill systems as Astra defaults.

## 10. Escalation triggers

Escalate rather than normalize when new competency categories are required, synthesis contradicts A07 boundaries, synthesis requires A08 technique mastery definition, synthesis requires A10 resource/cost/action-economy definition, or competency mapping conflicts with downstream actor/economy assumptions.

## 13. Test cases / pressure examples

Pressure tests include donor class-skill imports with fixed bonus math, point-buy tables, profession-to-faction ranking drift, cross-skill synthesis requiring A08/A10 mechanics, and multi-domain checks pressuring runtime state definitions.
