# SM02 Minimum Pilot Conversion Readiness and Packet Validation Controls

## 1. Purpose and status

SM02 is a pilot readiness and packet/output control file only. It defines the minimum controlled state required before Astra runs a small, bounded pilot conversion experiment on real donor material for evidence gathering.

SM02 does not run conversion and does not create converted donor content. It is not a final schema file, not executable JSON Schema, not a Pydantic model, not a validator implementation, not final mechanics, not exact math, not canon, not sourcebook prose, not live-play behavior, not runtime behavior, not backend/database design, and not training or evaluation data.

Status posture:

- SM02 is a readiness gate and packet/output control document only.
- SM02 defines readiness requirements and future validation targets; it does not turn those targets into final JSON fields, code, schemas, or converted records.
- SM02 may name required packet/output evidence, ledgers, queues, owner routes, and pass/fail controls, but it does not make any pilot output canon-ready, runtime-ready, sourcebook-ready, training-ready, or large-scale-corpus-ready.

## 6. Minimum pilot conversion readiness definition

Minimum pilot conversion readiness means the minimum controlled state required to run a small, bounded conversion experiment on selected real donor packets for evidence gathering.

It is not:

- large-scale corpus conversion readiness;
- canon readiness;
- sourcebook readiness;
- runtime readiness;
- final mechanics readiness;
- live-play readiness;
- training readiness.

Minimum pilot conversion readiness exists only when the selected packets can be traced, reviewed, lawfully routed, quarantined on failure, and evaluated for what broke without letting donor mechanics, donor statblocks, donor economies, donor classes, donor dice, donor cosmology, donor proper nouns, or donor legal/IP risk become Astra defaults.

## 7. Pilot scope constraints

The first pilot must be small and bounded. It should use a limited set of donor packets that deliberately cover a few high-pressure C-family routes without trying to cover everything.

The pilot must have:

- a declared packet count or packet list before conversion starts;
- a declared donor source list before conversion starts;
- a declared C-family pressure plan before conversion starts;
- a declared reviewer and quarantine path before conversion starts;
- a declared stop condition for failure, legal/IP risk, source-local leakage, missing page/range truth, missing evidence, or missing lawful outcome accounting.

The pilot must not expand opportunistically into full corpus conversion. Pilot outputs remain evidence artifacts and review targets, not doctrine, canon, mechanics, runtime data, sourcebook prose, live-play behavior, or training data.

## 8. Donor-family selection constraints

Unless later repository evidence justifies a narrower pressure test, the recommended initial pilot packet families to route are:

- C01 creature/NPC;
- C02 item/gear;
- C03 ability/power/technique;
- C10 table/oracle;
- one source-local or setting/cosmology pressure sample for conditional C14 only if legal/IP/source-local routing is ready.

The first pilot should deliberately avoid trying to cover every family. Vehicle/platform, companion/summon, crafting/salvage, adventure/scenario, map/diagram, faction, and complex location packs may be deferred unless selected for a specific pressure test. In C-family terms, C05, C06, C07, C08, C11, C12, and C13 may be deferred unless selected for pressure test.

C04 relic/implant/installable asset and C09 hazard/environment samples may also be deferred unless they are needed to disambiguate C01, C02, C03, or C10 pressure.

## 9. Conversion packet readiness gate

Before pilot conversion starts, every selected packet must have readiness evidence for:

- packet identity;
- donor source identity;
- donor-family classification;
- extraction run identity;
- page/range truth;
- source hash or hash-later policy;
- evidence references;
- construct inventory;
- repair queue status;
- quarantine queue status;
- legal/IP flags;
- reviewer decision points;
- C-family routing targets;
- benchmark/evaluation prerequisites.

These are readiness requirements and future validation targets only. SM02 does not define final JSON field names, final packet schemas, final output schemas, or final validators.

A packet fails this gate if its identity, donor source, page/range truth, evidence references, or extraction run identity cannot be reconstructed without consulting untracked donor material.

## 10. Evidence and provenance readiness gate

Pilot evidence must preserve the chain from donor source to extraction output to conversion packet to pilot output review.

Minimum evidence/provenance controls require:

- donor source identity tied to packet identity;
- extraction run identity tied to the packet build or handoff artifacts;
- page/range truth for every visible donor construct considered in the pilot;
- source hash or hash-later policy when a source hash is not yet available;
- evidence references sufficient for a reviewer to find why a construct was mapped, rejected, quarantined, or retained source-local;
- confidence/review-routing notes for uncertain records;
- pilot output review status before any pilot evidence is summarized.

Evidence failure must route to repair, quarantine, human review, or deferred gap handling. It must not be patched by inventing facts, fields, math, canon, or runtime state.

## 11. Lawful outcome completeness gate

Every visible donor construct in the pilot packet must have exactly one lawful outcome.

The lawful outcome ledger must account for direct Astra mapping, normalized Astra mapping, source-local retained construct, quarantined construct pending later doctrine, escalated doctrine problem, rejected import, or other lawful outcome taxonomy route already recognized by conversion intake controls.

The mapping ledger must connect each donor construct to its lawful outcome, rationale, evidence references, owner route, confidence/review-routing notes, and any must-not-import posture.

A pilot cannot pass if constructs disappear, if one construct receives contradictory outcomes, if rejected imports become Astra defaults, or if donor constructs are converted without lawful outcome accounting.

## 12. C00-C14 routing readiness gate

Every converted/output record or pilot output candidate must have an owner route among C01-C14 or `pending_schema`.

C00 remains the shared content record base posture owner. C01-C14 remain conversion-stage/canon-review family grammar owners:

- C01 creature/NPC;
- C02 item/gear;
- C03 ability/power/technique;
- C04 relic/implant/installable asset;
- C05 faction/institution;
- C06 location/site/region;
- C07 mission/scenario/adventure;
- C08 vehicle/ship/platform;
- C09 hazard/environment;
- C10 table/oracle;
- C11 companion/summon;
- C12 crafting/salvage/recipe;
- C13 map/diagram annotation;
- C14 source-local setting/cosmology.

C-family routing readiness does not promote any C-family record. It confirms only that the pilot can report which C-family owner or missing-owner fallback saw each output pressure.

## 13. `pending_schema` and missing-owner readiness gate

`pending_schema` is a lawful missing-owner route only. It is not permission to invent fields, import donor structures, bypass C00, bypass C01-C14, draft C15, create runtime state, write sourcebook prose, promote canon, or create mechanics.

The pilot must maintain a pending_schema ledger for every missing-owner pressure item. Each pending_schema ledger entry must preserve evidence references, donor-family classification, C-family routing targets considered and rejected, confidence/review-routing notes, reviewer decision points, lawful fallback, and whether the item routes to quarantine or deferred gap ledger.

Missing schema pressure may pass the pilot only if it routes to `pending_schema`, quarantine, or deferred gap ledger. It fails if the pilot silently creates unowned fields, final schemas, runtime contracts, canon, sourcebook prose, or final mechanics.

## 14. Rejected-import, source-local, and legal/IP readiness gate

The pilot must maintain:

- a rejected-import ledger;
- a source-local retention ledger;
- legal/IP flags;
- reviewer decision points for legal/IP, source-local, protected expression, proper nouns, setting-specific cosmology, donor statblocks, donor economies, donor classes, donor dice, and donor system identity.

Rejected imports must remain rejected and must not become Astra defaults. Source-local content must remain source-local and may route to C14, review, quarantine, or deferred gap handling only. Legal/IP flags must route to review or quarantine.

A pilot fails this gate if donor proper nouns, protected setting content, source-local cosmology, donor math, donor statblocks, donor economies, donor classes, donor dice, or legal/IP-risk content leaks into Astra baseline language.

## 15. Confidence, review-routing, and human-review readiness gate

Every uncertain pilot record must include confidence/review-routing notes and reviewer decision points.

Minimum review controls require:

- confidence notes for extraction ambiguity, donor-family ambiguity, C-family ambiguity, legal/IP risk, source-local retention, missing schema pressure, mechanics pressure, runtime pressure, canon pressure, and evidence gaps;
- explicit routing to human review when confidence is below the pilot threshold selected by the future pilot conversion owner;
- no automatic promotion from pilot output review status to canon, sourcebook, runtime, final mechanics, live-play, training, or large-scale readiness.

Human review may accept the pilot evidence as useful pressure. Human review may not use SM02 to promote final content.

## 16. Repair queue, quarantine, and failure-reporting readiness gate

Pilot conversion may begin only if repair queue status and quarantine queue status are known for each packet.

Failure reporting must be able to identify:

- failure report shape;
- packet identity;
- donor source identity;
- extraction run identity;
- page/range truth status;
- failed readiness gate;
- impacted evidence references;
- impacted construct inventory entries;
- lawful outcome ledger impact;
- mapping ledger impact;
- rejected-import ledger impact;
- source-local retention ledger impact;
- pending_schema ledger impact;
- repair queue status;
- quarantine queue status;
- legal/IP flags;
- reviewer decision points;
- pilot output review status;
- benchmark/evaluation prerequisites impacted.

The pilot must fail closed: repairable failures route to repair queues, legal/IP or doctrine-risk failures route to quarantine or review, and schema/missing-owner failures route to `pending_schema`, quarantine, or deferred gap ledger.

## 17. Pilot fixtures and test-data readiness gate

Pilot fixtures must be minimal, auditable, and non-corpus-forming.

Fixture controls:

- use only the selected pilot packet artifacts;
- keep fixture purpose limited to readiness validation, review, and failure reproduction;
- label fixture material as pilot evidence, not training data, not evaluation corpus, not sourcebook prose, not canon, and not live-play behavior;
- avoid embedding donor prose or converted donor content except as lawful, minimal, provenance-preserving evidence required for review;
- preserve source-local, rejected-import, legal/IP, confidence, quarantine, and repair posture.

SM02 does not create pilot fixtures. It states the readiness gate that a future pilot fixture owner must satisfy.

## 18. Benchmark and evaluation prerequisites

Benchmark/evaluation prerequisites must exist before pilot outputs are used to argue for broader readiness.

Minimum prerequisites:

- a declared benchmark/evaluation question for the pilot;
- a declared pass/fail interpretation rule;
- a declared owner for benchmark/evaluation review;
- a rule that pilot outputs are evidence samples, not evaluation corpora;
- a rule that pilot fixture material is not training data;
- a rule that mechanics I/O pressure routes to a future mechanics requirements owner only;
- a rule that runtime pressure routes to a future runtime/Gate B owner only;
- a rule that canon/sourcebook/live-play/training pressure routes to appropriate later phase owners only.

If these prerequisites are missing, pilot output can still identify failures, but it cannot be used to claim benchmark success, large-scale conversion readiness, final mechanics readiness, runtime readiness, canon readiness, sourcebook readiness, live-play readiness, or training readiness.

## 19. Minimum pilot pass/fail criteria

Minimum pilot pass criteria:

- Pass only if every visible donor construct in the pilot packet has one lawful outcome.
- Pass only if every converted/output record has an owner route among C01-C14 or `pending_schema`.
- Pass only if rejected imports remain rejected and do not become Astra defaults.
- Pass only if source-local content remains source-local.
- Pass only if legal/IP flags route to review or quarantine.
- Pass only if missing schema pressure routes to `pending_schema`, quarantine, or deferred gap ledger.
- Pass only if confidence/review-routing notes exist for uncertain records.
- Pass only if packet identity, donor source identity, donor-family classification, extraction run identity, page/range truth, source hash or hash-later policy, evidence references, construct inventory, lawful outcome ledger, mapping ledger, rejected-import ledger, source-local retention ledger, pending_schema ledger, repair queue status, quarantine queue status, reviewer decision points, C-family routing targets, pilot output review status, failure report shape, and benchmark/evaluation prerequisites are present at the readiness-control level.

Minimum pilot fail criteria:

- Fail if donor math/statblocks/economies/classes/dice/cosmology leak as Astra defaults.
- Fail if final mechanics, runtime schemas, canon, sourcebook prose, live-play behavior, or training data are created.
- Fail if D-series source packs are treated as authority.
- Fail if pilot outputs are treated as large-scale readiness.
- Fail if C00-C14 registry statuses are promoted by this pilot readiness PR.
- Fail if C00-C14 are rewritten, C15 is created, or RHBF is used as hidden law.
- Fail if executable JSON Schema, Pydantic models, runtime/backend/database contracts, entity/component/event schemas, command lifecycle contracts, context packet contracts, save-state shapes, donor-statblock validators, final math, resolution dice, damage formulas, resource formulas, or progression math are created.

## 20. Large-scale conversion non-readiness boundary

SM02 does not unlock large-scale corpus conversion readiness.

A successful pilot may produce evidence for later controls, but it does not prove:

- full corpus extraction stability;
- full corpus packet validation stability;
- cross-donor deduplication readiness;
- broad C-family schema readiness;
- broad legal/IP review readiness;
- broad canon-review readiness;
- broad evaluation readiness;
- runtime/Gate B readiness;
- training readiness.

Large-scale conversion requires later owner-controlled readiness gates after pilot evidence is reviewed.

## 21. Runtime/canon/sourcebook/live-play/training non-readiness boundary

SM02 is not runtime readiness, canon readiness, sourcebook readiness, live-play readiness, training readiness, or final mechanics readiness.

Runtime pressure routes to future runtime/Gate B owner only. SM02 does not create runtime schemas, backend schemas, database schemas, entity/component/event schemas, command lifecycle contracts, context packet contracts, save-state shapes, runtime kernel behavior, or live-play behavior.

Canon pressure routes to canon/conflict owners only. SM02 does not promote canon, accept lexicon, write sourcebook prose, or make pilot outputs sourcebook-ready.

Mechanics pressure routes to a future mechanics requirements owner only. SM02 does not create final mechanics, exact math, resolution dice, damage formulas, resource formulas, progression math, donor-statblock validators, or donor math defaults.

Training and evaluation pressure routes to later training/evaluation owners only. SM02 does not create training data, evaluation corpora, fine-tuning corpora, benchmark corpora, or model behavior policy.

## 22. Owner map and lawful fallbacks

| Pressure | Future owner route | Lawful fallback if not ready |
| --- | --- | --- |
| Conversion packet readiness | future pilot conversion owner | repair queue, quarantine, or defer pilot packet |
| Evidence/provenance readiness | future conversion/evidence validation owner | repair queue, evidence gap ledger, quarantine |
| Lawful outcome completeness | lawful outcome/conversion intake owner | block pilot output, repair ledger, quarantine |
| C00-C14 routing readiness | C00/C-family schema owner and future validation/schema implementation owner | route to `pending_schema`, quarantine, or deferred gap ledger |
| `pending_schema` handling | C00/future validation/schema implementation owner | keep in pending_schema ledger, quarantine, or defer |
| Rejected import/source-local/legal/IP readiness | C00/C14 plus future review/legal owner | rejected-import ledger, source-local retention ledger, legal/IP review, quarantine |
| Confidence/review routing | future review owner | human review required, no promotion |
| Repair/quarantine/failure report | extraction readiness and pilot conversion owner | repair queue, quarantine queue, failure report |
| Benchmark/evaluation prerequisites | future evaluation/benchmark owner | pilot evidence only, no benchmark claim |
| Mechanics I/O pressure | future mechanics requirements owner only | doctrine escalation, `pending_schema`, quarantine donor statblocks |
| Runtime pressure | future runtime/Gate B owner only | block runtime import, quarantine runtime-shaped pressure |
| Canon/sourcebook/live-play/training pressure | appropriate later phase owner only | no promotion, no sourcebook prose, no live-play behavior, no training data |

Lawful fallbacks must be explicit. Silent import, hidden defaults, donor-law adoption, RHBF hidden law, and ownerless promotion are forbidden.

## 23. Risk register

| Risk | Why it matters | SM02 control |
| --- | --- | --- |
| Pilot scope creep | A small experiment could become unreviewed broad conversion. | Predeclare packet list, donor list, C-family pressure plan, and stop conditions. |
| Donor mechanics leakage | Donor math/statblocks/economies/classes/dice could become Astra defaults. | Require lawful outcomes, rejected-import ledger, source-local retention ledger, C-family owner routes, and fail criteria. |
| Source-local/legal/IP leakage | Protected proper nouns or setting cosmology could enter Astra baseline. | Require C14/legal review, source-local retention, legal/IP flags, and quarantine. |
| Missing schema disguised as implementation | Pilot pressure could invent fields or C15. | Route to `pending_schema`, quarantine, or deferred gap ledger; refuse C00-C14 rewrite and C15 creation. |
| Evidence loss | Pilot outputs without provenance cannot be reviewed. | Require packet identity, donor source identity, extraction run identity, page/range truth, source hash or hash-later policy, and evidence references. |
| Over-trusting pilot outputs | A passed pilot could be mistaken for large-scale readiness. | Separate minimum pilot readiness from large-scale corpus conversion readiness and later phases. |
| Runtime/canon/sourcebook/training creep | Pilot artifacts could be treated as final product surfaces. | Explicit runtime/canon/sourcebook/live-play/training non-readiness boundary. |
| D-series authority creep | D00-D19 draft packs could be treated as current doctrine. | State D-series packs are draft source material only and cannot become pilot conversion authority. |
| Evaluation misuse | Pilot examples could become benchmark/evaluation or training corpora. | Require benchmark/evaluation prerequisites and refuse training/evaluation corpora. |

It should test whether the SM02 readiness gates can be applied to a tiny selected packet set before any real conversion output loop is run.
