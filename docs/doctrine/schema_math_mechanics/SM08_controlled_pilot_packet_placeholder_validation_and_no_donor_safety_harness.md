## 6. Placeholder validation harness definition

A placeholder validation harness is a repo-local test/control layer that checks whether SM07 scaffold fixture files are present, labeled, bounded, non-executing, and contain required placeholder capture slots. It is a safety screen for placeholder posture only, not legal/IP review, not execution approval, not conversion readiness proof, not pilot quality validation, and not benchmark/evaluation success.

The placeholder validation harness checks:

- Every scaffold fixture file exists and is nonempty.
- Every scaffold fixture file contains clear placeholder/synthetic/non-donor labels.
- Every scaffold fixture file contains no obvious donor-content artifact markers.
- Every scaffold fixture file contains no known donor-proper-noun sentinel markers.
- The scaffold collectively contains required SM05 authorization reference placeholder terms.
- The scaffold collectively contains required SM06 output-capture placeholder terms.
- The scaffold represents SM04 review-handoff placeholder targets.

## 7. No-donor safety harness definition

A no-donor safety harness is a repo-local test/control layer that checks for explicit no-donor labels and sentinel donor-content markers. It is a safety screen, not legal review and not proof that all possible protected expression has been detected.

The no-donor safety harness checks:

- Every scaffold fixture file contains the terms `placeholder`, `synthetic`, and `non-donor`.
- Every scaffold fixture file does not contain obvious artifact markers such as `converted output:`, `donor excerpt:`, `statblock:`, `source table:`, `source map:`, `training example:`, `sourcebook-ready example:`, `actual pilot output:`.
- Every scaffold fixture file does not contain known donor-proper-noun sentinel markers (a small sentinel list, not exhaustive).
- The scaffold does not contain implementation artifacts such as JSON Schema object definitions, Pydantic class definitions, database table definitions, command lifecycle contracts, context packet contracts, or save-state shapes.

The no-donor safety harness is not legal/IP review. It is not a complete copyright detector. It is not proof that future donor content is safe. It does not authorize real donor execution. It does not validate actual conversion output. It does not prove pilot quality, canon readiness, runtime readiness, sourcebook readiness, live-play readiness, final mechanics readiness, training readiness, or large-scale conversion readiness.

## 25. Risk register

Risks:

1. **False sense of safety**: Passing SM08 harness checks does not mean donor content is safe. Legal/IP review is still required before any real donor conversion.
2. **Sentinel list incompleteness**: The donor-marker sentinel list is not exhaustive. Some donor proper nouns may slip through.
3. **Scope creep**: Scaffold may accidentally expand beyond authorized boundaries. Harness must block scope expansion.
4. **Implementation creep**: Scaffold may accidentally include JSON Schema, Pydantic models, or database schemas. Harness must block implementation artifacts.
5. **Authority confusion**: Harness pass labels may be mistaken for execution approval. Harness must clearly state it is not execution approval.
6. **Registry pollution**: Harness labels may be mistakenly written into registry fields. Harness must clearly state labels are not registry values.
7. **Fixture drift**: Scaffold fixtures may drift from placeholder/synthetic/non-donor posture over time. Regular re-validation is required.
8. **Upstream dependency risk**: SM08 depends on SM05, SM06, SM07 being properly maintained. Changes to upstream files may break SM08 validation.

Mitigations:

- Clear documentation of harness limitations.
- Regular re-validation of scaffold fixtures.
- Strict blocking of scope expansion and implementation artifacts.
- Clear separation of harness labels from registry values.
- Explicit statement that harness pass is not execution approval.
