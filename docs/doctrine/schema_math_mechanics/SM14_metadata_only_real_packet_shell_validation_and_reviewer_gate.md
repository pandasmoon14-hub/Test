# SM14 Metadata-Only Real Packet Shell Validation and Reviewer Gate

## 1. Purpose and status

SM14 is a metadata-only real packet shell validation and reviewer-gate control file only. It defines how reviewers validate that a future metadata-only real pilot authorization packet shell is complete, internally consistent, no-donor safe, owner-routed, C-family bounded, dependency-complete, and still non-executing before any actual authorization decision or execution proposal.

SM14 does not create a completed real authorization packet. SM14 does not make an actual authorization decision. SM14 does not authorize actual execution. SM14 does not approve execution. SM14 does not run conversion. SM14 does not create actual converted donor content or pilot outputs. SM14 is not pilot evidence, not legal/IP clearance, not source clearance, not runtime readiness, not canon readiness, not sourcebook readiness, not final-mechanics readiness, not live-play readiness, and not training readiness.

## 6. Metadata-only shell validation definition

Metadata-only shell validation means reviewer assessment of whether a future metadata-only real authorization packet shell contains all required sections and slots from SM13 without containing donor content, execution approval, final schemas, runtime contracts, final mechanics, canon/sourcebook/live-play/training content, or registry promotion.

The validation subject is the shell posture only: named sections, non-execution statements, metadata-only reference categories, owner routes, dependency routes, lawful fallback routes, and failure routes. Metadata-only shell validation is not validation of donor facts, source text, converted content, canon truth, playable mechanics, runtime contracts, or implementation readiness.

## 7. Reviewer gate definition

The SM14 reviewer gate is a document-control gate that asks whether a future metadata-only shell is safe to send to a later authorization-decision record process. The gate may return document-local shell validation labels and rationale, but it must not make actual authorization, must not approve execution, and must not create a completed real authorization packet.

A reviewer gate pass means only that obvious shell completeness, owner-routing, no-donor, dependency, and phase-boundary failures were not found at shell-validation time. It does not mean conversion can start.

## 8. Packet shell versus completed packet validation boundary

A packet shell is a metadata-only arrangement of section headings, reference categories, owner slots, dependency slots, decision-label slots, and non-readiness statements. A completed packet would contain actual owner decisions, final authorization posture, verified legal/IP disposition, source-access disposition, and downstream execution-specific readiness. SM14 validates only the packet shell.

SM14 does not create a completed real authorization packet and does not validate real packet quality or real packet completeness. A future completed real authorization packet remains downstream and requires later owner-controlled authorization work.

## 9. Validation eligibility requirements

A future shell is eligible for SM14-style validation only if it is metadata-only, no-donor safe, non-executing, scoped to the SM05-SM13 pressure set, and explicitly downstream of SM00-SM13. The shell must include legal/IP owner routing, future review owner routing, source access posture, source-clearance boundary, and failure routes before it can receive a ready-for-authorization-review label.

If the shell lacks SM02 no-waiver posture, SM05 authorization-slot posture, SM13 assembly posture, legal/IP owner routing, or donor leakage hard-fail criteria, reviewers must block or repair the shell rather than advance it.

## 10. No-donor-in-repository validation rule

SM14 validates that the shell contains no real donor content in the repository. The shell must not select, embed, quote, paraphrase, transform, summarize, or convert real copyrighted donor excerpts, donor statblocks, donor tables, donor maps, donor setting prose, protected expression, or donor proper nouns.

Any embedded real donor excerpt, donor statblock, donor table, donor map, donor setting prose, donor proper noun import as an Astra default, or converted donor content is a hard failure. The correct route is repair, quarantine, rejected import, legal/IP review, or source-local retention according to the relevant owner path.

## 11. Legal/IP and external-reference validation posture

SM14 requires reviewers to confirm that a legal/IP owner slot exists and that external-reference metadata is routed to appropriate review. SM14 does not itself provide legal/IP clearance. Metadata is not legal/IP clearance, external-reference readiness is not source clearance, and no-donor sentinel checks are not legal/IP clearance.

External-reference metadata may show that a future owner knows where to look, who must review, and where unresolved questions should route. It must not be treated as permission to ingest, quote, transform, or convert donor material.

## 12. Source access and source-clearance validation boundary

SM14 may validate whether a shell names source access posture and source-clearance boundary slots. SM14 may not validate source access facts, grant source clearance, or treat external-reference readiness as source clearance.

If source access, page/range reference posture, source hash posture, or source-clearance owner routing is missing or ambiguous, reviewers must block, repair, quarantine, or route the shell to the future owner rather than advance it.

## 13. SM02 no-waiver validation requirement

SM14 requires a visible SM02 no-waiver posture. SM02 minimum pilot readiness gates may not be waived by SM14, by shell validation, by reviewer convenience, by metadata-only references, by dry-run success, or by later execution pressure.

A shell that omits SM02 no-waiver posture receives a blocking label such as `shell_validation_blocked_by_missing_sm02_gate`.

## 14. SM03 dry-run validation requirement

## 30. Rejected-import, source-local, legal/IP, and `pending_schema` validation

## 31. Confidence, review-routing, repair, quarantine, and failure-report validation

SM14 reviewers validate that confidence/review-routing notes, repair route, quarantine route, and failure-report route slots exist. These slots tell future owners how to handle incomplete, unsafe, unclear, or over-scoped shells.

Low confidence, missing owner routing, donor leakage risk, source-clearance gaps, scope expansion, or runtime/mechanics creep must route to repair, quarantine, deferred gap ledger, `pending_schema`, legal/IP review, source-local retention, rejected import, or another appropriate future owner.

## 32. Donor leakage hard-fail validation

SM14 requires donor leakage hard-fail criteria. A shell fails immediately if it embeds donor content, imports donor proper nouns as Astra defaults, uses donor statblocks as Astra defaults, includes donor tables, includes donor maps, includes donor setting prose, or presents converted donor content or pilot outputs.

A donor leakage hard fail cannot be cured by saying the content is metadata. The shell must be repaired, quarantined, rejected, or routed for legal/IP review before any later authorization review.
