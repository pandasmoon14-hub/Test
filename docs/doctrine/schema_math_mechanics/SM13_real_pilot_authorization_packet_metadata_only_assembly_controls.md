# SM13 Real Pilot Authorization Packet Metadata-Only Assembly Controls

## 1. Purpose and status

SM13 is a real pilot authorization packet metadata-only assembly-control file only. It defines how a future metadata-only real pilot authorization packet shell may arrange required sections, metadata-only reference slots, owner slots, dependency slots, C-family pressure slots, legal/IP slots, review-routing slots, no-donor safety slots, and failure-routing slots before a future packet can be drafted or validated.

SM13 does not create a completed real authorization packet. SM13 does not make an actual authorization decision. SM13 does not authorize actual execution. SM13 does not approve execution. SM13 does not run conversion. SM13 does not create actual converted donor content or pilot outputs. SM13 is not pilot evidence, not legal/IP clearance, not source clearance, not runtime readiness, not canon readiness, not sourcebook readiness, not final-mechanics readiness, not live-play readiness, and not training readiness.

## 6. Metadata-only assembly definition

Metadata-only assembly means arranging required packet shell sections and reference slots for a future real authorization packet without copying donor material into the repository. It arranges labels, owner placeholders, routing references, source-access posture, legal/IP review posture, C-family pressure targets, lawful outcome routes, mapping ledger routes, rejected-import routes, source-local routes, `pending_schema` routes, confidence/review-routing notes, repair routes, quarantine routes, failure-report routes, and donor leakage hard-fail criteria.

Metadata-only assembly is not donor content, not legal/IP clearance, not source clearance, not conversion evidence, not canon evidence, and not execution approval. Metadata-only categories must not create final schema fields, not database fields, not runtime contracts, not final validators, and not implementation contracts.

## 7. Packet shell versus completed packet boundary

A packet shell is an ordered container of section requirements and metadata-only slots. A completed real authorization packet is a later owner-controlled artifact that would contain reviewed metadata-only entries, owner decisions, authorization status, and validation posture under downstream controls. SM13 creates only assembly controls for a possible shell; it does not create the shell as a completed packet and does not fill it with real source content.

A packet shell may show where packet identity metadata, source access posture, legal/IP owner, future review owner, C-family pressure, lawful outcome route, mapping ledger route, and failure routes belong. A completed packet boundary remains downstream and cannot be inferred from SM13.

## 8. Metadata-only assembly eligibility requirements

A future metadata-only shell is eligible for assembly review only if: SM00-SM12 are present and current; no completed real authorization packet is created by the assembly PR; no real donor content is embedded; the legal/IP owner slot exists; the future review owner slot exists; metadata is not treated as legal/IP clearance; external-reference readiness is not treated as source clearance; no-donor sentinel checks are not treated as legal/IP clearance; SM02 gates have no waived minimum gate; SM03-SM12 dependency slots are represented; C-family pressure remains bounded; lawful outcome, mapping, rejected-import, source-local, legal/IP, `pending_schema`, confidence, repair, quarantine, and failure-report routes exist; and donor leakage hard-fail criteria are explicit.

## 9. No-donor-in-repository assembly rule

SM13 assembly requires no donor content in the repository. Assembly material must not include raw donor excerpts, donor statblocks, donor tables, donor maps, donor setting prose, protected expression, donor proper nouns, converted donor content, actual pilot outputs, sourcebook-ready examples, benchmark corpora, evaluation corpora, training data, fine-tuning data, or model behavior data.

If embedded donor content appears, assembly must hard-fail and route to quarantine, legal/IP review, rejected import, source-local retention, repair, or failure report as appropriate.

## 10. Legal/IP and external-reference assembly posture

The shell must include legal/IP owner and legal/IP route slots, but those slots do not clear any source. Metadata can identify that a legal/IP review owner must assess a source-access posture, source-clearance boundary, source hash or hash-later policy, page/range reference posture, and external-reference pointer status later.

Metadata is not legal/IP clearance. No-donor sentinel checks are not legal/IP clearance. External-reference readiness is not source clearance.

## 11. Source access and source-clearance assembly boundary

Source access posture describes whether a future reviewer can locate source material outside the repository under permitted access rules. Source-clearance posture is separate and must not be assumed from access, hash, citation pointer, page/range posture, or source-family metadata. External-reference readiness is not source clearance, and source access is not permission to convert.

## 12. SM02 no-waiver assembly requirement

SM13 cannot waive SM02 gates. The shell must include an SM02 no-waiver posture slot stating that minimum pilot readiness gates cannot be waived and that absence of a satisfied SM02 posture blocks assembly rather than allowing execution.

## 13. SM03 dry-run assembly requirement

The shell must include an SM03 dry-run posture slot. The slot records whether dry-run review planning has been represented or whether failures route to repair, quarantine, deferred gap ledger, `pending_schema`, or another owner. It does not run a dry conversion.

## 14. SM04 evaluation-rubric assembly requirement

The shell must include an SM04 evaluation/rubric handoff slot. The slot records where future post-execution review would route if execution were separately authorized later. It does not create benchmark corpora, evaluation corpora, scoring data, or pilot evaluation success.

## 15. SM05 authorization-slot assembly requirement

The shell must include an SM05 authorization slot. The slot must remain empty or metadata-only until a later owner-controlled authorization decision. SM13 cannot bypass SM05 authorization and cannot treat metadata-only assembly as execution approval.

## 16. SM06 execution/output-capture assembly requirement

The shell must include an SM06 execution/output-capture slot so a future packet can point to output-capture posture if execution is separately authorized later. The slot does not approve execution, does not run conversion, and does not capture actual pilot outputs.

## 17. SM07 scaffold/review-harness assembly requirement

The shell must include an SM07 scaffold/review-harness slot referencing the scaffold and harness posture. It may point to placeholder/synthetic/non-donor Markdown fixtures only and must not create executable harnesses, runtime contracts, final validators, or converted outputs.

## 18. SM08 no-donor safety assembly requirement

The shell must include an SM08 no-donor safety slot and donor leakage hard-fail criteria. SM08 sentinel posture can support no-donor safety checks, but no-donor sentinel checks are not legal/IP clearance.

## 19. SM09 dry template assembly requirement

The shell must include an SM09 dry template reference to the dry authorization packet template. That template is placeholder/synthetic/non-donor and remains distinct from any future completed real authorization packet.

## 20. SM10 reviewer validation assembly requirement

The shell must include an SM10 reviewer validation slot for dry authorization validation and reviewer decision-gate posture. SM10 validation of dry packets does not validate a completed real authorization packet and does not approve execution.

## 21. SM11 preparation-control assembly requirement

The shell must include an SM11 preparation-control slot. SM11 preparation controls govern how a future real packet may be prepared, but SM13 assembly does not complete that packet and does not decide authorization.

## 22. SM12 external-reference review assembly requirement

The shell must include an SM12 external-reference review slot. SM12 owns external-reference review-gate controls; SM13 only arranges where the external-reference review posture appears in a metadata-only packet shell.

## 30. Donor leakage hard-fail assembly

Donor leakage hard-fail criteria must block assembly when the shell contains real copyrighted donor excerpts, donor statblocks, donor tables, donor maps, donor setting prose, protected expression, donor proper nouns, converted donor content, actual pilot outputs, sourcebook-ready examples, training data, benchmark corpora, evaluation corpora, or model behavior data.

A donor leakage hard-fail does not prove legal/IP clearance after removal; it proves only that the unsafe assembly attempt must be repaired, quarantined, rejected, or escalated.
