# SM09 Controlled Pilot Execution Dry Authorization Packet Template

## 1. Purpose and status

SM09 is a controlled pilot execution dry authorization packet template file only. It defines a dry, non-donor template for a future controlled pilot execution authorization packet, so reviewers can see the required sections, review slots, owner slots, evidence slots, authorization-decision slots, and downstream review handoff slots that a later real authorization packet would need.

SM09 is not an authorization decision, not execution approval, not conversion, and not pilot evidence. SM09 does not authorize actual execution. SM09 does not approve execution. SM09 does not run conversion. SM09 does not create actual converted donor content or pilot outputs. SM09 may create a placeholder/synthetic/non-donor Markdown fixture template under `tests/fixtures/pilot_conversion_scaffold/` for dry template review only.

SM09 does not select, embed, quote, paraphrase, transform, summarize, or convert real copyrighted donor excerpts, donor statblocks, donor tables, donor maps, donor setting prose, protected expression, or donor proper nouns. SM09 does not import donor proper nouns as Astra defaults.

SM09 does not create benchmark corpora, evaluation corpora, training data, fine-tuning data, model behavior data, model behavior policy, sourcebook-ready examples, final schemas, executable JSON Schema, Pydantic models, final validators, runtime schemas, backend schemas, database schemas, runtime/backend/database contracts, entity/component/event schemas, command lifecycle contracts, context packet contracts, save-state shapes, final mechanics, exact math, canon, sourcebook prose, live-play behavior, accepted lexicon, or registry promotion.

## 6. Dry authorization packet definition

A dry authorization packet is a placeholder/synthetic/non-donor packet template that demonstrates what a later real pilot authorization packet must contain before any separately controlled execution proposal can be reviewed. It is a representation of required slots, not a filled authorization record.

A dry authorization packet must be visibly labeled as dry, placeholder, synthetic, non-donor, not actual authorization, not execution approval, not converted content, not pilot output, not training data, not canon, not sourcebook prose, and not runtime import data.

## 7. Dry template fixture policy

The optional dry template fixture may exist only as Markdown at `tests/fixtures/pilot_conversion_scaffold/dry_authorization_packet_template.md`, as a direct child of `tests/fixtures/pilot_conversion_scaffold/`. It must contain the words `placeholder`, `synthetic`, and `non-donor`; it must state `no real donor content`, `not converted content`, `not training data`, `not canon`, `not sourcebook prose`, and `not runtime import data`.

## 8. Placeholder/non-donor content policy

## 10. SM02 readiness gate representation

The dry template must include a placeholder slot for a statement that SM02 gates were satisfied with no waived minimum gate. Missing this slot blocks the dry template. The slot may not assert that SM02 gates are actually satisfied for a real donor packet unless a later SM05-style authorization packet supplies evidence.

SM09 cannot waive SM02 gates, cannot imply a waiver through template readiness, and cannot convert a placeholder readiness statement into conversion execution readiness.

## 11. SM03 dry-run result representation

The dry template must include a placeholder slot for a statement that SM03 dry-run review passed or failures were routed. This slot records whether dry-run review evidence exists or whether unresolved failures were routed to repair, quarantine, or failure-report handling.

This representation is not pilot evidence and not conversion success. It is only a required future packet field.

## 12. SM04 rubric readiness representation

## 20. Rejected-import, source-local, legal/IP, and `pending_schema` representation

## 21. Confidence, review-routing, repair, quarantine, and failure-report representation
