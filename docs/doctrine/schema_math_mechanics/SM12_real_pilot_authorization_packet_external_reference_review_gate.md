# SM12 Real Pilot Authorization Packet External-Reference Review Gate

## 1. Purpose and status

SM12 is a real pilot authorization packet external-reference review gate only. It defines how reviewers check a future real pilot authorization packet that identifies candidate source material through metadata-only external references before any separate execution proposal can be considered.

SM12 does not create a real authorization packet. SM12 does not make an actual authorization decision. SM12 does not authorize actual execution. SM12 does not approve execution. SM12 does not run conversion. SM12 does not create actual converted donor content or pilot outputs.

SM12 is not a real authorization packet, not an authorization decision, not execution approval, not conversion, not pilot evidence, not legal/IP clearance, and not runtime/canon/sourcebook/final-mechanics readiness. SM12 is a review-gate control file only.

SM12 does not select, embed, quote, paraphrase, transform, summarize, or convert real copyrighted donor excerpts, donor statblocks, donor tables, donor maps, donor setting prose, protected expression, or donor proper nouns.

SM12 does not create benchmark corpora, evaluation corpora, training data, fine-tuning data, model behavior data, model behavior policy, sourcebook-ready examples, final schemas, final output schemas, executable JSON Schema, JSON Schema files, Pydantic models, final validators, runtime schemas, backend schemas, database schemas, runtime/backend/database schemas, runtime/backend/database contracts, entity/component/event schemas, command lifecycle contracts, context packet contracts, save-state shapes, final mechanics, exact math, resolution dice, damage formulas, resource formulas, progression math, canon, sourcebook prose, live-play behavior, accepted lexicon, registry promotion, or donor-statblock validators that treat donor statblocks as Astra defaults.

## 6. External-reference review gate definition

An external-reference review gate is a metadata-only review of whether a future real authorization packet can identify candidate source material without copying donor content into the repository. The gate asks whether references are reviewable, routed, bounded, and safe enough to move toward a later and separate execution proposal review.

The gate does not approve the references for use, does not clear the sources, does not verify copyright status, does not decide legal/IP outcomes, and does not authorize conversion.

## 7. Real packet, preparation, review gate, and execution boundary

SM12 distinguishes four boundaries:

- Real packet: a future metadata-only authorization packet shell that is not created by SM12.
- Preparation: SM11 controls how the future packet is prepared for review without creating converted content.
- Review gate: SM12 controls how reviewers assess the future packet's metadata-only external references before any execution proposal.
- Future execution PR: a later owner-controlled proposal, if ever allowed, that still must pass SM02, SM05, legal/IP, source-clearance, no-donor safety, and output-capture controls before any actual execution.

Passing SM12 review does not create a real packet, does not make an authorization decision, does not approve execution, and does not run conversion.

## 8. External-reference eligibility requirements

Any future packet using external references must show all of the following before it can be considered reviewable under SM12:

- legal/IP owner is named;
- future review owner is named;
- source access posture is reviewable;
- source-clearance posture is not assumed;
- reference metadata is sufficient for reviewers without copying protected expression;
- no raw donor excerpts, statblocks, tables, maps, setting prose, protected expression, or donor proper nouns are embedded;
- source-local and rejected-import routing are available;
- quarantine route is available for unsafe references;
- failure-report route is available;
- SM02 gates have no waived minimum gate;
- SM03-SM11 dependency posture is represented;
- C-family pressure stays bounded.

## 9. Metadata-only reference posture

SM12 may describe external-reference metadata categories, but it must not create final schema fields, JSON Schema, Pydantic models, database fields, runtime contracts, or final validators.

Allowed metadata categories may include internal packet candidate id, donor-family category, source access status, legal/IP review status, source hash or hash-later policy, page/range reference posture, external-reference pointer status, extraction-readiness route, review owner, legal/IP owner, C-family pressure target, lawful outcome route, source-local / rejected-import / quarantine route, and failure-report route.

Metadata is not donor content, not legal/IP clearance, not source clearance, not conversion evidence, not canon evidence, and not execution approval. Metadata must never be treated as donor content or conversion evidence.

## 10. No-donor-in-repository review rule

SM12 requires a no-donor-in-repository review rule for any future external-reference packet. The repository must not receive raw donor excerpts, donor statblocks, donor tables, donor maps, donor setting prose, protected expression, or donor proper nouns through the external-reference review process.

A donor leakage finding is a hard fail. The route is repair, quarantine, failure report, rejected import, legal/IP review, or source-local retention outside repository doctrine as appropriate.

## 11. Legal/IP owner review requirement

A future external-reference packet must name a legal/IP owner before the packet can move beyond SM12 review. The named owner must be able to decide whether a reference requires clearance, rejection, source-local handling, or further owner review.

SM12 does not provide legal/IP clearance. No-donor sentinel checks are not legal/IP clearance. Metadata is not legal/IP clearance.

## 12. Source access and source-clearance boundary

A future external-reference packet must describe source access posture in a reviewable metadata-only way. External-reference readiness is not source clearance. Source access status does not imply permission to copy, convert, summarize, transform, quote, paraphrase, or embed donor material.

If source clearance is missing, uncertain, or owner-unassigned, the reference is blocked, repaired, quarantined, routed to legal/IP review, retained source-local, or rejected.

## 13. SM02 no-waiver review requirement

Reviewers must confirm SM02 minimum pilot readiness gates have no waived minimum gate. SM12 cannot waive SM02 gates and cannot accept a future packet that treats external-reference review as a substitute for SM02 readiness.

## 14. SM03 dry-run review requirement

## 25. Evidence/provenance pointer review

Evidence/provenance pointer review checks whether future reviewers can understand where a candidate reference is intended to point without copying protected expression into the repository. Pointers may describe reviewability, hash posture, source-local retention, or hash-later policy, but they do not create conversion evidence, canon evidence, legal/IP clearance, source clearance, or execution approval.

## 26. Lawful outcome and mapping reference review

SM12 requires reviewers to verify that lawful outcome taxonomy and mapping reference posture are represented. A reference must have a route to lawful transformation, source-local retention, rejected import, legal/IP review, `pending_schema`, repair, quarantine, or failure report.

Mapping posture must not import donor expression, donor proper nouns, exact math, final mechanics, or sourcebook-ready examples.

## 27. Rejected-import, source-local, legal/IP, and `pending_schema` reference review

Reviewers must confirm that rejected-import, source-local, legal/IP, and `pending_schema` routes exist for unsafe, ambiguous, owner-missing, or schema-uncertain references. `pending_schema` is a routing posture, not a final schema field and not permission to implement runtime/backend/database schemas.

## 28. Confidence, review-routing, repair, quarantine, and failure-report review

Reviewers must check confidence posture, review-routing posture, repair route, quarantine route, and failure-report route before any future packet advances. Low confidence, missing owner, embedded donor content, scope expansion, legal/IP uncertainty, source clearance gap, runtime creep, mechanics creep, or C-family overreach must block or route the reference.

## 29. Donor leakage hard-fail review

Donor leakage is a hard fail. Embedded donor excerpts, donor statblocks, donor tables, donor maps, donor setting prose, protected expression, donor proper nouns, copied mechanical blocks, or converted donor content must stop review and route to quarantine, repair, rejected import, legal/IP review, failure report, or source-local retention as appropriate.
