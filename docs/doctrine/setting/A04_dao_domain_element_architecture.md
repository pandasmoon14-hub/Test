# A04 Dao / Domain / Element Architecture

## 1. Purpose and status
- File ID: A04
- Layer: 1_setting_spine
- Phase: 1A
- Status: draft
- Purpose: Define Astra doctrine-level architecture boundaries for Dao, domain, element, and conceptual law interaction grammar without importing donor metaphysics as Astra default law.

## 2. What this file owns
- Doctrine-level classification grammar for Dao constructs, domain constructs, element constructs, and conceptual-law constructs.
- Doctrine-level interaction grammar for resonance, antagonism, synthesis, expression constraints, and compatibility posture across Dao/domain/element constructs.
- Doctrine-level handling for multi-construct coexistence and contradiction triage when donor systems present conflicting metaphysical assumptions.
- Source-local construct, quarantine posture, and escalation posture for unresolved Dao/domain/element pressure.

## 3. What this file must not own
- Specific techniques, spells, powers, maneuvers, paths, classes, or subclasses.
- Specific Dao names, domain names, element names, deity portfolios, factions, planets, planes, or hidden truths.
- Lore canon declarations or world-history claims.
- Resource pools, costs, recharge, backlash, action economy, damage, conditions, cultivation stages, or progression.
- Runtime state, event commits, hidden-information behavior, or implementation logic.
- Accepted lexicon term governance owned by K01.
- Final Core Rulebook prose.
- Donor Dao/domain/element mechanics as Astra default law.

## 4. Required definitions
- **Dao construct**: doctrine-level classification for principle-path metaphysical framing used in source systems, without importing donor naming or progression law.
- **Domain construct**: doctrine-level classification for authority/scope metaphysical framing used in source systems, without importing donor taxonomies.
- **Element construct**: doctrine-level classification for elemental or substrate metaphysical framing used in source systems, without importing donor element sets.
- **Conceptual-law construct**: doctrine-level classification for abstract principle constraints that shape Dao/domain/element interaction posture.
- **Resonance relation**: interaction classification where constructs can align or reinforce without collapsing distinction.
- **Antagonism relation**: interaction classification where constructs conflict or inhibit and require explicit boundary handling.
- **Synthesis relation**: interaction classification where constructs can combine under doctrine constraints without erasing provenance.
- **Expression constraint**: doctrine-level boundary defining where a construct may be expressed, interpreted, or retained.
- **Compatibility posture**: doctrine-level statement of whether relation handling is direct, normalized, source-local, quarantined, or escalated.
- **Source-local construct**: donor construct retained with provenance and local scope only.
- **Quarantined construct**: unresolved construct held outside active Astra defaults pending later review.
- **Escalated doctrine problem**: contradiction requiring formal downstream doctrine resolution.

### Interaction outcome grammar
- **Direct compatible relation**: construct relation maps directly to existing doctrine grammar without semantic distortion.
- **Normalized compatible relation**: construct relation maps after explicit normalization preserving provenance and boundary notes.
- **Constrained compatibility relation**: relation is compatible only under explicit expression constraints and cannot be treated as universal law.
- **Source-local relation only**: relation retained as donor-local construct only.
- **Quarantined pending later doctrine**: relation held in quarantine until later doctrine resolution.
- **Escalated contradiction**: relation escalated due to unresolved contradiction.

## 5. Core doctrine rules
- A04 defines classification and interaction grammar boundaries only; it does not define runtime mechanics.
- A04 remains `A04_dao_domain_element_architecture.md` and is not renamed.
- K01 remains separate and lexicon governance is not moved into A04.
- A04 is doctrine draft guidance and is not current.
- A04 does not define lore canon and does not promote donor metaphysics into Astra default law.
- A04 may reference A01 only for cosmology/topology boundary context; it does not redefine A01 ownership.
- A04 may reference A02 only where source-field relation posture is required for interaction boundary handling.
- A04 may reference A03 where ontology boundaries constrain Dao/domain/element interaction interpretation.

## 6. Conversion mapping rules
- **Direct architecture mapping**: donor construct maps to existing A04 grammar without semantic distortion.
- **Normalized architecture mapping**: donor construct maps after explicit normalization with provenance and boundary notes.
- **Interaction outcome mapping**: donor interaction claims are classified using interaction outcome grammar before downstream usage.
- **Source-local retained construct**: donor construct retained for source-local continuity without Astra-canon authority.
- **Quarantined construct pending later doctrine**: unresolved construct isolated pending later doctrine/canon review.
- **Escalated doctrine contradiction**: contradiction or unclassifiable construct escalated for formal doctrine resolution.

## 7. Source-local handling
- Donor Dao/domain/element systems can be retained as source-local records with provenance.
- Source-local records do not become Astra architecture canon by extraction success alone.
- Repeated donor pressure may form canon candidates only through later K-layer/canon review.
- Unclassifiable interaction claims are quarantined or escalated, not normalized by invention.
- Donor proper nouns remain source-local unless later canon promotion accepts them.
- Source-local terms do not become accepted lexicon terms through A04.

## 8. Donor pressure absorbed
- Pressure to import donor Dao/domain/element taxonomies as default Astra metaphysical law.
- Pressure to collapse concept-layer and mechanics-layer claims into one unbounded doctrine surface.
- Pressure to treat donor deity-domain frameworks as universal architecture.
- Pressure to force contradictory element interaction models into a single unreviewed default.
- Pressure to equate extraction success with canon adoption.

## 9. Hard refusals / rejected imports
- Donor Dao names are not Astra default architecture law.
- Donor divine-domain systems are not Astra default domain architecture.
- Donor element sets are not automatically Astra default element architecture.
- Donor conceptual/metaphysical proper nouns are not accepted lexicon terms by A04.
- Contradictory donor interaction claims are not silently merged by invention.

## 10. Escalation triggers
- A proposed construct requires interaction types beyond resonance, antagonism, or synthesis.
- A proposed interaction handling conflicts with downstream scale/resource/advancement doctrine boundaries.
- A proposal attempts to move K01 lexicon governance into A04.
- A proposal requests runtime state/event ownership or hidden-information behavior ownership inside A04.
- A proposal requests specific techniques, powers, classes, domains, element sets, or lore-canon declarations inside A04.
