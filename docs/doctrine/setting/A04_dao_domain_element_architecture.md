# A04 Dao / Domain / Element Architecture

## 1. Purpose and status
- File ID: A04
- Layer: 1_setting_spine
- Phase: 1A
- Status: draft
- Purpose: Define Astra doctrine-level architecture boundaries for Dao, domain, element, and conceptual law interaction grammar without importing donor metaphysics as Astra default law.

## 2. What this file owns
- Doctrine-level conceptual classification grammar for Dao constructs, domain constructs, element constructs, concept/law constructs, and related construct families used in donor pressure handling.
- Doctrine-level interaction grammar for resonance, antagonism, synthesis, containment, adjacency, opposition, transformation, contamination, occlusion/masking, and unstable hybridization.
- Doctrine-level synthesis grammar for compatibility posture across mixed constructs, including constrained, unstable, antagonistic, dominant-subordinate, and balanced outcomes.
- Doctrine-level source-local retention, quarantine posture, and escalation posture for unresolved Dao/domain/element conceptual contradictions.

## 3. What this file must not own
- Specific techniques.
- Specific spells.
- Specific powers, maneuvers, paths, classes, or subclasses.
- Specific Dao names, domain names, element names, deity portfolios, factions, planets, planes, or hidden truths.
- Resource pools, costs, recharge, backlash, action economy, damage, conditions, cultivation stages, or progression.
- Faction control of domains.
- Donor Dao imports as Astra default law.
- Runtime state.
- Hidden information state.
- Event commits or runtime hidden-information behavior.
- Accepted lexicon terms.
- Lore canon declarations or world-history claims.
- Donor conceptual metaphysics as Astra default law.
- Final Core Rulebook prose.

## 4. Required definitions
- **Dao construct**: doctrine-level classification for principle-path metaphysical framing used in source systems without importing donor naming or progression law.
- **Domain construct**: doctrine-level classification for authority/scope metaphysical framing used in source systems without importing donor taxonomies.
- **Element construct**: doctrine-level classification for elemental or substrate metaphysical framing used in source systems without importing donor element sets.
- **Concept/law construct**: doctrine-level classification for abstract conceptual constraints that shape interaction posture.
- **Aspect/tag-like construct**: doctrine-level classification for label-style conceptual pressure that behaves as an attached interpretive marker rather than full doctrine law.
- **Sphere/paradigm-like construct**: doctrine-level classification for worldview or discipline framing pressure without adopting donor metaphysical frameworks as defaults.
- **Tradition/school-like construct**: doctrine-level classification for lineage-style or school-style metaphysical interpretation pressure without defining named schools.
- **Portfolio-like construct**: doctrine-level classification for authority bundles that may appear in donor systems without adopting donor portfolio defaults.
- **Elemental family**: doctrine-level grouping class for element constructs retained as classification containers only.
- **Conceptual family**: doctrine-level grouping class for concept/law constructs retained as classification containers only.
- **Source-local conceptual construct**: donor conceptual construct retained with provenance and local scope only.
- **Quarantined conceptual construct**: unresolved conceptual construct held outside active Astra defaults pending later review.
- **Escalated conceptual problem**: unresolved conceptual contradiction requiring formal downstream doctrine resolution.

### Interaction grammar
- **Resonance**: relation where constructs can align or reinforce without collapsing distinction.
- **Antagonism**: relation where constructs conflict or inhibit and require explicit boundary handling.
- **Synthesis**: relation where constructs can combine under doctrine constraints without erasing provenance.
- **Containment**: relation where one construct bounds expression scope of another without redefining either construct's core classification.
- **Adjacency**: relation where constructs remain near-compatible or co-present without implied synthesis.
- **Opposition**: relation where constructs stand in direct conceptual opposition and must not be silently merged.
- **Transformation**: relation where interpretation shifts from one construct posture to another and must preserve provenance notes.
- **Contamination**: relation where cross-construct leakage introduces instability risk requiring explicit handling posture.
- **Occlusion/masking**: relation where one construct obscures interpretability of another without proving elimination.
- **Unstable hybridization**: relation where mixed-construct handling remains unstable and cannot be treated as default architecture.
- **Source-local only**: interaction retained as donor-local relation only.
- **Quarantined pending later doctrine**: interaction held in quarantine until later doctrine resolution.
- **Escalated contradiction**: interaction escalated due to unresolved contradiction.

### Synthesis grammar
- **Compatible synthesis**: synthesis classification where involved constructs cohere without contradiction under doctrine boundaries.
- **Constrained synthesis**: synthesis classification valid only under explicit expression constraints and non-universal scope.
- **Unstable synthesis**: synthesis classification with unresolved instability that cannot become default architecture.
- **Antagonistic synthesis**: synthesis classification where combined posture retains active antagonism and must be explicitly bounded.
- **Dominant-subordinate synthesis**: synthesis classification where one construct is primary and another is subordinate for interpretation scope only.
- **Balanced synthesis**: synthesis classification where constructs are jointly weighted without implying permanent universal balance.
- **Source-local synthesis only**: synthesis retained as donor-local handling only.
- **Quarantined synthesis**: synthesis held outside active defaults pending later doctrine resolution.
- **Escalated synthesis contradiction**: synthesis contradiction escalated for formal doctrine resolution.

## 5. Core doctrine rules
- A04 defines conceptual classification, interaction grammar, and synthesis grammar boundaries only; it does not define runtime mechanics.
- A04 remains `A04_dao_domain_element_architecture.md` and is not renamed.
- K01 remains separate and lexicon governance is not moved into A04.
- A04 is doctrine draft guidance and is not current.
- A04 does not define lore canon and does not promote donor conceptual metaphysics into Astra default law.
- A04 may reference A01 only for cosmology/topology boundary context and does not redefine worlds, planes, or dimensional topology ownership.
- A04 may reference A02 only where source-field relation posture is required and does not redefine source fields or magic/technology relation ownership.
- A04 may reference A03 where ontology boundaries constrain interpretation and does not redefine identity continuity ownership.

## 6. Conversion mapping rules
- **Direct architecture mapping**: donor construct maps to existing A04 grammar without semantic distortion.
- **Normalized architecture mapping**: donor construct maps after explicit normalization with provenance and boundary notes.
- **Interaction outcome mapping**: donor interaction claims are classified using interaction grammar before downstream usage.
- **Synthesis outcome mapping**: donor synthesis claims are classified using synthesis grammar before downstream usage.
- **Source-local retained construct**: donor construct retained for source-local continuity without Astra-canon authority.
- **Quarantined construct pending later doctrine**: unresolved construct isolated pending later doctrine/canon review.
- **Escalated doctrine contradiction**: contradiction or unclassifiable construct escalated for formal doctrine resolution.

## 7. Source-local handling
- Donor Dao/domain/element conceptual systems can be retained as source-local records with provenance.
- Source-local conceptual records do not become Astra architecture canon by extraction success alone.
- Repeated donor pressure may form canon candidates only through later K-layer/canon review.
- Unclassifiable interaction or synthesis claims are quarantined or escalated, not normalized by invention.
- Donor proper nouns remain source-local unless later canon promotion accepts them.
- Source-local terms do not become accepted lexicon terms through A04.

## 8. Donor pressure absorbed
- Pressure to import donor Dao/domain/element taxonomies as default Astra conceptual architecture law.
- Pressure to treat aspect/tag, sphere/paradigm, or tradition/school metaphysical framings as automatic global defaults.
- Pressure to treat donor portfolio-like bundles as universal architecture.
- Pressure to force contradictory interaction or synthesis models into a single unreviewed default.
- Pressure to equate extraction success with canon adoption.

## 9. Hard refusals / rejected imports
- Donor Dao names are not Astra default architecture law.
- Donor divine-domain systems are not Astra default domain architecture.
- Donor element sets are not automatically Astra default element architecture.
- Donor conceptual metaphysical proper nouns are not accepted lexicon terms by A04.
- Contradictory donor interaction or synthesis claims are not silently merged by invention.

## 10. Escalation triggers
- A proposed construct requires classification or interaction types outside A04 required grammar.
- A proposed interaction or synthesis handling conflicts with downstream scale/resource/advancement doctrine boundaries.
- A proposal attempts to move K01 lexicon governance into A04.
- A proposal requests runtime state/event ownership or hidden-information behavior ownership inside A04.
- A proposal requests specific techniques, spells, powers, classes, named domains/elements/Dao names, or lore-canon declarations inside A04.
