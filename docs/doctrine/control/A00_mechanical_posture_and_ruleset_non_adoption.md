# A00 Mechanical Posture and Ruleset Non-Adoption

## 1. Purpose and status
- File ID: A00
- Layer: 0_control
- Phase: 0
- Status: draft
- Purpose: Establish the pre-A01 governance posture that extraction/conversion work does not equal mechanical adoption.

## 2. What this file owns
- Mechanical non-adoption posture during doctrine bootstrap.
- Guardrails for handling donor ruleset pressure before A01 drafting is promoted.
- Control-layer constraints that prevent implicit rule authority transfer from donor systems.

## 3. What this file must not own
- Cosmology and dimensional architecture doctrine content (owned by A01).
- Lexicon governance and reserved term policy (owned by K01).
- Runtime implementation state mutation controls and Gate B backend operations.

## 4. Required definitions
- **Mechanical adoption**: any claim that donor mechanics become executable Astra rule authority.
- **Conversion mapping**: routing donor constructs into Astra mapping outcomes without granting rule authority.
- **Source-local retained construct**: donor construct retained for provenance, not as Astra canon or mechanics doctrine.
- **Rejected import**: donor construct explicitly refused from Astra mechanics authority.

## 5. Core doctrine rules
- Astra doctrine does **not** adopt donor rulesets, donor mechanics, or donor balancing assumptions by default.
- Conversion permission is not mechanical adoption.
- Registry tracking records are work-order/control metadata and are not canon authority.
- A01 remains `A01_cosmology_and_dimensional_architecture.md` and is not renamed by this file.
- K01 remains `K01_lexicon_governance_and_reserved_terms.md`; lexicon governance is not moved into A01.

## 6. Conversion mapping rules
- Donor mechanics may map to: direct Astra mapping, normalized Astra mapping, source-local retained construct, quarantined construct, or escalated doctrine problem.
- Mapping outcomes must preserve provenance and must not imply term acceptance.
- Mapping tags must remain separated from reserved-term governance tags.

## 7. Source-local handling
- Source-local records preserve donor wording, context, and provenance.
- Source-local records are non-authoritative for Astra mechanics and canon.
- Source-local records may trigger K01 or pre-A01 work orders when lexicon conflicts are detected.

## 8. Donor pressure absorbed
- Pressure to rapidly import donor mechanics into active doctrine.
- Pressure to treat extraction completeness as ruleset readiness.
- Pressure to collapse lexicon gating into A01 drafting for speed.

## 9. Hard refusals / rejected imports
- No donor rules text is accepted as Astra rule authority by default.
- No donor subsystem labels are accepted as reserved Astra terms without K01 process.
- No donor balance logic is accepted as Astra balance doctrine without downstream review.

## 10. Escalation triggers
- Any file claims donor mechanics are authoritative absent explicit promotion.
- Any process renumbers or replaces A01 through control-layer tracking records.
- Any process routes Stage 5 lexicon-control issues directly into A01 renumbering instead of K01/pre-A01 ledgers and work orders.

## 11. Dependencies
- ROADMAP-001
- REGISTRY-001
- Active control files governing scope model and conversion IR precedence

## 12. Handoff to downstream layers
- A01 receives guardrails and cannot interpret this file as mechanics doctrine content.
- K01 receives lexicon conflict routes, reserved-term family constraints, and term lifecycle governance inputs.
- C00+ schema layers receive only mapping posture constraints, not mechanics authority.

## 13. Test cases / pressure examples
- Example: donor spellcasting subsystem extracted at high confidence still remains non-authoritative until doctrine promotion.
- Example: dense donor tables become source-local/rejected-import records when not aligned to accepted Astra mechanics doctrine.
- Example: Stage 5 lexicon-control collision creates K01 ledger entry and pre-A01 work order, not A01 renumbering.

## 14. Versioning and review protocol
- Substantive change requires version bump, registry update, and re-review.
- `review_status` should remain non-current until review + pressure tests complete.
- Control-layer amendments must be logged in `docs/decisions/current_decisions_log.md` and tracked by registry record(s).
