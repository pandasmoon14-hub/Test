## 1. Purpose and scope

This file creates the documentation/control owner scaffold for RT-012, the D-series/native-design source-pack promotion-boundary track identified by `REMEDIATION-PRIORITY-LEDGER-001`. It exists to name the future owner boundary that must separate draft D-series source material, design pressure, doctrine adoption decisions, authority-level updates, sourcebook inclusion, runtime dependency checks, training-use review, conflict review, registry tracking, and canon promotion.

This scaffold is planning/control only. It is owner scaffold only: no doctrine rewrite, no runtime implementation, no schema implementation, no command IR implementation, no math implementation, no validator implementation, no generator implementation, no generated-record creation, no persistence writer implementation, no retrieval index implementation, no context-packet compiler implementation, no source-pack promotion, no sourcebook inclusion, no canon promotion, no live-play/training authorization, and no donor-content audit.

It does not define a promotion workflow implementation, canon promotion, sourcebook inclusion, runtime authority, training authorization, validator implementation, doctrine rewrite, final mechanics, final math, generators, generated records, context-packet compilers, persistence writers, retrieval indexes, command IR, or live-play behavior.

## 10. Dependency relationships

D-series/native-design material may exert design pressure only until explicitly promoted. Promotion requires separate reviewer decision, canon/sourcebook inclusion policy, conflict review, authority-level update, and registry tracking. D-series material can inform later doctrine but cannot silently become Astra law.

## 11. LLM non-authority rules

The LLM is prohibited from:

- promoting D-series source-pack material to canon;
- treating D-series draft material as runtime authority;
- using D-series examples as live-play behavior;
- converting source-pack prose into mechanics;
- overriding canon hierarchy;
- resolving canon conflicts;
- authorizing sourcebook inclusion;
- authorizing training data;
- using draft material in context packets as authoritative truth;
- treating source-pack presence as adoption.

The LLM may summarize D-series material only as draft design pressure when explicitly asked and must keep source-pack status, authority level, and promotion state visible.

## 12. Context-packet and narration handoff expectations

Context packets must not present D-series/native-design draft material as authoritative truth, backend state, accepted mechanics, live-play behavior, canon, sourcebook inclusion, or training material. Narration must not use D-series examples as live-play behavior or imply that source-pack prose has been adopted. If draft material is mentioned for review, it must be labeled as design pressure, pending promotion review, rejected/quarantined, or explicitly promoted by a separate decision.
