## D06 core law

A Route is the technical umbrella for formal route identity and structured development direction. A Path is the player-facing identity of a Route. A Route is not a donor class table, not a profession, not a skill, not a feature list, not a spell list, not a faction rank, and not a donor package with renamed nouns.

D06 may create Route candidates, Route Authorization Packets, Route Facets, Route Feature Ledgers, Principle / Concept facets, Technique candidates, source-local retained structures, and owner-file handoffs. Route identity becomes active only through lawful authorization.

1. Route is not class.
2. Path is not donor class name import.
3. Archetype is not subclass.
4. Route Feature Ledger is not freeform class-feature generation.
5. Technique is not every named action.
6. Principle / Concept is not early reality override.
7. Profession is not automatic Path.

# D06-00 Route Architecture Overview

**File ID:** `D06-00`  
**Pack:** Astra Ascension D06 — Route, Path, Principle, Technique, and Donor Class Normalization Doctrine  
**Version:** v0_1  
**Status:** Design doctrine draft generated from accepted D06 decisions.  
**Purpose:** Define Route as the technical umbrella, Path as player-facing identity, and Route Kernel as the formal structure that replaces donor class-table assumptions.

## Standing D06 boundary

D06 owns formal route identity, Path identity, Route Kernels, route facets, Route Feature Ledgers, Techniques, Principle / Concept route facets, route development, multipath / confluence, profession-route formalization, source-local route conversion, and donor class normalization.

D06 does **not** own D03 Power Economy costs, carriers, Expressions, reservoirs, burdens, or confluence math; D04 proof sufficiency, level, tier, Awakening, Development Picks, breakthroughs, capstone, or Ascension Access; D05 skills, professions, knowledge, training, research, methods, and synthesis substrate; D07 harm, corruption, madness, mutation harm, overload, identity erosion, or recovery; D08 companions, summons, spirits, AI, bonds, or nonhuman actor-state mechanics; D09 relics, tools, implants, weapons, vehicles, ships, mechs, or platforms; D10 factions, rank, law, reputation, territory, settlement, economy, or world-state; final runtime/database schemas; canon sourcebook prose; or live-play GM behavior.

D06 records candidates, structure, permissions, limits, and owner-file handoffs. It does not use route language to bypass the owners above.


## 1. D06 ownership

D06 owns the formal identity layer. It answers: once D04 proof and D05 route pressure exist, what lawful structure does the actor now have? D06 turns authorization into bounded route identity. It does not create power, proof, skill, faction, item, companion, or harm systems by itself.

D06 owns:

- Route identity and Path identity.
- Route Authorization Packets.
- Route Kernel structure.
- Optional typed route facets.
- Route Feature Ledgers.
- Principle / Concept route facets.
- Technique architecture and Technique Sets.
- Profession-Route formalization.
- Route development, branch, drift, scar, mutation, severance, fusion, multipath, and confluence.
- Source-local route conversion and donor class-like package normalization.

D06 does not own the systems listed in the standing boundary. Every cross-system effect must carry an owner-file handoff.

## 2. Route, Path, and Class

**Route** is the technical umbrella for formal route identity and structured development direction. It is backend / doctrine-facing.

**Path** is the player-facing formal identity of a Route. It is how a character understands and names the route they are walking.

**Class** is donor-facing and conversion-facing. Astra does not use Class as the default player-facing core term in D06. Donor classes are decomposed into Astra structures by function.

A donor class may contain route identity, skills, resources, features, equipment assumptions, faction ties, level tables, metaphysics, and capstones. Astra separates those into D03, D04, D05, D06, D07, D08, D09, and D10 owners.

## 3. Route Kernel

A **Route Kernel** is the D06-owned structure that records the core of a formal Route. It is not a full character class. It is a bounded identity kernel that can contain only the facets it needs.

A Route Kernel should be able to record:

- route identity and Path name;
- source and authorization basis;
- current route state;
- route permissions and limits;
- typed facets such as Domain, Archetype, Principle / Concept, Technique Set, Profession-Route, External Anchor, Lineage / School / Faction, Source-local, and Drift / Scar / Constraint;
- Route Feature Ledger links;
- Technique links;
- D03 / D04 / D05 / D07 / D08 / D09 / D10 handoffs;
- source-local boundary and validation state.

## 4. Route Kernel is not a class table

A Route Kernel does not automatically provide a fixed list of benefits by level. It can have a Route Feature Ledger, but that ledger is personalized, route-instance-specific, and owner-validated. Fixed feature tables may exist only as source-local constructs, simplified templates, conversion artifacts, or later canon-designed route packages.

## 5. Route sources

A Route may originate from many sources:

- Awakening Package;
- profession transmutation;
- D05 route pressure;
- teacher, lineage, school, faction, guild, order, sect, crew, or institution;
- external anchor such as relic, tool, platform, companion, AI, territory, or faction office;
- Principle / Concept revelation;
- Domain contact;
- research formalization;
- forced, coerced, corrupted, or accidental route event;
- source-local donor conversion;
- breakthrough, capstone, or post-capstone transformation.

A source is not enough. Active route identity requires authorization, permissions, limits, and handoffs.

## 6. Design examples

### Example: Far Fisher

A fisher's D05 profession, Depth Reading Method, impossible-catch proof, and D04 Awakening Package may authorize a D06 Route. The player-facing Path may be **Far Fisher**. It might permit development around line, distance, lure, hidden aquatic movement, patient pursuit, and impossible catch logic. It does not automatically grant teleportation, all-water control, creature domination, planar travel, or tide authority.

### Example: Glass-Tomb Archivist

A dangerous archive researcher may develop a Route through D05 knowledge, D05 research, dead-record proof, and D07 risk. The Route may include Domain facets around Funeral Archives and Glass, a Principle facet around Memory or Preservation, Techniques for record stabilization, and risks of identity bleed. D07 owns identity harm; D09 may own tomb-lens relics; D10 may own archive institutions.

### Example: Mech-Synced Ash Cavalier

A platform-route actor may have a Route dependent on a mech. D06 owns route identity and Techniques; D09 owns the mech; D07 owns neural feedback harm; D03 validates any powered overdrive cost. If the mech is destroyed, the route may become dormant, scarred, externalized into a cockpit core, mutated, or severed.

## 7. Conversion stance

Donor classes, subclasses, archetypes, schools, sect paths, prestige classes, playbooks, spell lists, class features, and class resources are evidence. They are not authority. D06 preserves useful identity signals while routing mechanics to the correct owner file.

## 8. Core rule

A Route is a proof-shaped formal identity kernel. A Path is its player-facing identity. A Route has permissions and limits, may have typed facets, may support personalized features and Techniques, and must maintain owner-file handoffs. It is not a donor class with renamed nouns.

# D06-01 Route Authorization and Path Identity

**File ID:** `D06-01`  
**Pack:** Astra Ascension D06 — Route, Path, Principle, Technique, and Donor Class Normalization Doctrine  
**Version:** v0_1  
**Status:** Design doctrine draft generated from accepted D06 decisions.  
**Purpose:** Define the gate by which Route candidates become active Routes and Path identities.

## Standing D06 boundary

D06 owns formal route identity, Path identity, Route Kernels, route facets, Route Feature Ledgers, Techniques, Principle / Concept route facets, route development, multipath / confluence, profession-route formalization, source-local route conversion, and donor class normalization.

D06 does **not** own D03 Power Economy costs, carriers, Expressions, reservoirs, burdens, or confluence math; D04 proof sufficiency, level, tier, Awakening, Development Picks, breakthroughs, capstone, or Ascension Access; D05 skills, professions, knowledge, training, research, methods, and synthesis substrate; D07 harm, corruption, madness, mutation harm, overload, identity erosion, or recovery; D08 companions, summons, spirits, AI, bonds, or nonhuman actor-state mechanics; D09 relics, tools, implants, weapons, vehicles, ships, mechs, or platforms; D10 factions, rank, law, reputation, territory, settlement, economy, or world-state; final runtime/database schemas; canon sourcebook prose; or live-play GM behavior.

D06 records candidates, structure, permissions, limits, and owner-file handoffs. It does not use route language to bypass the owners above.


## 1. Core problem

Astra must prevent automatic identity inflation. A profession, title, research result, donor class, faction rank, or conceptual affinity cannot become a Route by itself. D06 therefore uses the **Route Authorization Packet**.

A Route Authorization Packet is the doctrine structure that records why a Route is lawful now.

## 2. Route Authorization Packet requirements

A Route may become active only when it has:

1. lawful route source;
2. authorization basis;
3. selected Path identity;
4. route function;
5. route permissions;
6. route limits;
7. route facets;
8. required owner-file handoffs;
9. route state;
10. validation state;
11. actor-state delta.

The packet may be lightweight in play, but doctrine requires these elements so route identity is auditable.

## 3. Authorization sources

Valid route sources include:

- D04 Awakening Package;
- D05 Profession-Route pressure;
- D05 mastery, research, method, or synthesis records;
- teacher, lineage, school, guild, faction, order, sect, crew, or institution;
- external anchor such as relic, ship, mech, implant, companion, AI, territory, shrine, archive, or office;
- Principle / Concept route event;
- Domain contact;
- forced, coerced, corrupted, cursed, accidental, or hostile route imposition;
- source-local converted class or path;
- D04 breakthrough, capstone, or Ascension Access transformation.

Invalid sources include naming a desired Path without basis, copying donor class packaging, high profession rank alone, faction rank alone, one impressive event without route-relevant proof, or a D05 method without D06 authorization.

## 4. Path identity

A **Path** is the player-facing identity of an authorized Route. A strong Path identity answers:

- What has the actor become authorized to develop?
- What proof, source, or event shaped this Route?
- What facets does it contain?
- What permissions does it create?
- What limits does it impose?
- What obligations, scars, anchors, or source-local boundaries apply?

Bad Path design: “Far Fisher: magical fisher.”

Better Path design: “Far Fisher: a profession-route formed from impossible pursuit, aquatic reading, long-distance prey logic, line-and-depth methods, and world-crossing desire. It authorizes development around distance, lure, hidden movement, patient pursuit, aquatic domains, and impossible catch logic. It does not automatically grant teleportation, planar travel, creature domination, or tide authority.”

## 5. Route states

D06 tracks route states so unusual cases can be handled without improvising new systems.

| State | Meaning |
|---|---|
| candidate | Possible route, not yet authorized. |
| latent | Route seed exists but is not active. |
| offered | Route appears as a bounded choice. |
| selected | Chosen but still resolving packet requirements. |
| active | Lawful route development can proceed. |
| dormant | Route exists but is inaccessible or inactive. |
| scarred | Route carries lasting wound, cost, trauma, or contradiction. |
| coerced | Route was forced or imposed. |
| corrupted | Route is contaminated, hostile, or unstable. |
| externalized | Route depends on an external anchor. |
| source-local | Route is bounded to a source or context. |
| unstable | Route works but risks drift, backlash, or contradiction. |
| rejected | Route was refused but may remain in the ledger. |
| severed | Route was cut away, lost, or intentionally removed. |
| fused | Route merged with another route structure. |
| mutated | Route changed into another structure. |
| retired | Route no longer develops but remains historical. |

UI may simplify these, but doctrine should preserve them.

## 6. Candidate handling

Multiple route candidates may exist internally. Player-facing choice should usually remain bounded, especially at Awakening. A character with many proof clusters should not receive dozens of choices. D06 may collapse candidates into strongest lawful options while recording latent, rejected, dormant, scarred, or future fusion candidates.

## 7. Low-tier Principle / Concept limit

A Principle or Concept route at low tier may shape perception, route pressure, method development, resonance signals, bounded technique candidates, and domain alignment. It does not grant reality override, immunity, conceptual damage, resurrection, universal truth, or cosmic rank by default. Those require D04 proof / tier authorization and D03 / D07 / D10 handoffs as relevant.

## 9. Varied examples

### Lawful: Oathbound Street Surgeon

A street medic repeatedly saves hostile faction members under oath-bound neutrality. D05 records medical skill, profession, and trauma methods. D04 records proof. D10 records faction recognition. D06 authorizes **Oathbound Street Surgeon** with permissions around triage, oath-protected neutrality, battlefield medical Techniques, and limits against universal healing or legal invulnerability.

### Unlawful: “I name myself Void Emperor”

A player declares a Path name without proof, source, authorization basis, facets, permissions, limits, or handoffs. D06 classifies this as unsupported route declaration. It may become a desired route, research project, or dangerous temptation, but it is not active.

### Source-local: Cloud Palm Disciple

A donor sect grants Cloud Palm Disciple rank. D06 records a source-local Route candidate. D10 owns sect rank. D03 reviews cloud qi resource if retained. D06 may normalize palm Techniques, but the donor sect path does not become canon Astra law.

## 10. Rule

A formal Path is proof-shaped, bounded, and authorized. It is not a donor class table, profession label, skill rank, faction rank, title, research output, or player-declared identity.

# D06-02 Route Facets, Domains, and Archetypes

**File ID:** `D06-02`  
**Pack:** Astra Ascension D06 — Route, Path, Principle, Technique, and Donor Class Normalization Doctrine  
**Version:** v0_1  
**Status:** Design doctrine draft generated from accepted D06 decisions.  
**Purpose:** Define optional typed facets inside Route Kernels and preserve boundaries among Domain, Archetype, anchors, lineage, source-local, and constraint facets.

## Standing D06 boundary

D06 owns formal route identity, Path identity, Route Kernels, route facets, Route Feature Ledgers, Techniques, Principle / Concept route facets, route development, multipath / confluence, profession-route formalization, source-local route conversion, and donor class normalization.

D06 does **not** own D03 Power Economy costs, carriers, Expressions, reservoirs, burdens, or confluence math; D04 proof sufficiency, level, tier, Awakening, Development Picks, breakthroughs, capstone, or Ascension Access; D05 skills, professions, knowledge, training, research, methods, and synthesis substrate; D07 harm, corruption, madness, mutation harm, overload, identity erosion, or recovery; D08 companions, summons, spirits, AI, bonds, or nonhuman actor-state mechanics; D09 relics, tools, implants, weapons, vehicles, ships, mechs, or platforms; D10 factions, rank, law, reputation, territory, settlement, economy, or world-state; final runtime/database schemas; canon sourcebook prose; or live-play GM behavior.

D06 records candidates, structure, permissions, limits, and owner-file handoffs. It does not use route language to bypass the owners above.


## 1. Optional typed facets

A Route Kernel may contain any lawful combination of typed facets. It does not require every facet. Facets are mechanically relevant routing containers, not flavor-only descriptors. Each facet must define what it contributes, what it does not authorize, and which owner files it touches.

Common facets:

- Path Identity.
- Domain.
- Archetype.
- Principle / Concept.
- Technique Set.
- Profession-Route.
- External Anchor.
- Lineage / School / Faction.
- Source-local.
- Drift / Scar / Constraint.

## 2. Path Identity facet

The Path Identity facet names and frames the player-facing Route. It is not the whole route. It summarizes the proof-shaped identity and should contain permissions and limits.

Examples:

- **Glass-Tomb Archivist**: shaped by burial archives, memory recovery, dead-language scholarship, and dangerous record preservation.
- **Red-Moon Skirmisher**: shaped by night raids, lunar fieldcraft, blood-risk tactics, and irregular combat.
- **Quiet Engine Cantor**: shaped by ship-system chant protocols, engine resonance, and platform-interface discipline.
- **Thorn-Court Mediator**: shaped by hostile court negotiation, oath interpretation, and social hazard survival.

## 3. Domain facet

A **Domain** is a bounded sphere of relevance, alignment, affinity, environmental interaction, or contextual resonance. It is not automatic territory control, divine portfolio, or area-of-effect power.

Domains may be material, environmental, social, symbolic, institutional, technological, ecological, or source-local.

Examples:

- Saltwater.
- Broken Machines.
- Oaths.
- Thresholds.
- Deep Roads.
- Battlefield Smoke.
- Glass.
- Hunger.
- Urban Rooftops.
- Winter Markets.
- Funeral Archives.
- Engine Choirs.
- Pack Predation.
- Dream Cartography.

Domain relations:

| Relation | Meaning |
|---|---|
| primary | Central to route function. |
| secondary | Relevant but not defining. |
| latent | Present but undeveloped. |
| opposed | Route struggles against it. |
| corrupted | Domain relation is unstable or contaminated. |
| externalized | Domain relation depends on anchor / place / tool. |
| source-local | Valid only in bounded source context. |

Examples:

- A **Desert Rainmaker** can have Domains Drought and Rain without contradiction if the route concerns scarcity and restoration.
- A **City Rat Saint** can have Domains Alleys, Disease, Survival, and Crowd Memory without automatically gaining disease control; D07 / D10 / D03 would validate any such mechanics.
- A **Ship-Born Cartographer** may have Domains Corridors, Pressure Doors, and Starless Navigation that function in ships but not open wilderness.

## 4. Archetype facet

An **Archetype** is a functional role-shape. It is not a class, subclass, or fixed feature table.

Archetypes organize route expectations, UI grouping, technique tags, and balance review. They do not grant features by themselves.

Possible archetypes:

- Striker, Guardian, Controller, Scout, Investigator, Crafter, Healer, Commander, Infiltrator, Summoner, Anchor, Negotiator, Ritualist, Platform Operator, Survivalist, Saboteur, Scholar, Duelist, Warden, Swarm Handler, Relic Bearer, Domain Specialist.

A route may have a primary archetype and secondary archetypes. The same Archetype can produce different Route Feature Ledgers depending on proof, Route facets, anchors, and owner-file handoffs.

Example:

- **Crow-Marked Rooftop Scout**, **Drone-Swarm Cartographer**, **Hollow-Depth Listener**, and **Fungal Courier** may all include Scout-like function, but their features, anchors, risks, and Techniques differ.

## 5. Technique Set facet

A **Technique Set** is a related family of formal Techniques associated with a Route, facet, teacher, lineage, source-local system, profession-route, or external anchor. Technique Sets can be fixed, branching, research-grown, method-grown, teacher-gated, source-local, route-ledger generated, or anchor-dependent. They are not class tables by default.

## 6. Profession-Route facet

A **Profession-Route** facet records that a D05 Profession has become formal Route structure. The profession remains a D05 record. D06 adds route identity and route permissions.

Examples:

- Fisher → Far Fisher.
- Baker → Hearth Wolfbane.
- Street Surgeon → Oathbound Street Surgeon.
- Cartographer → Drone-Swarm Cartographer.
- Miner → Hollow-Depth Listener.

A profession does not become a Route automatically. D05 basis, D04 proof, and D06 authorization are required.

## 7. External Anchor facet

An **External Anchor** facet means route function depends partly on something outside the actor: relic, weapon, tool, implant, ship, mech, vehicle, AI, companion, spirit, territory, faction office, domain site, workshop, shrine, archive, banner, map, or contract.

Owner handoffs:

- D08 for companion, spirit, AI, bond, nonhuman actor relation.
- D09 for relic, tool, implant, vehicle, ship, platform, weapon.
- D10 for territory, faction office, institution, shrine as world-state.
- D03 if externalization changes Power Economy or carrier behavior.

D06 owns route identity, not anchor mechanics.

## 8. Lineage / School / Faction facet

This facet records that a route is taught, granted, restricted, recognized, or shaped by a tradition or institution. D10 owns faction, rank, law, reputation, and institution state. D05 owns teaching and training. D06 owns the route facet.

## 9. Source-local facet

A **Source-local Facet** records route names, mechanics, or metaphysics bounded to a specific source, conversion, campaign, faction, realm, or donor-derived context. Source-local facets may be used inside their boundary but do not become Astra-wide law unless promoted through canon review.

## 10. Drift / Scar / Constraint facet

This facet records instability, cost, contradiction, coercion, mutation, source-local lock, external dependency, corruption, or route damage. D07 owns harm/corruption consequences; D06 records route impact.

Examples:

- Coerced pact route.
- Scarred healer route after battlefield failure.
- Corrupted memory route contaminated by hostile dead records.
- Contradictory mercy route carrying vengeance proof.
- Externalized route that collapses if a relic is destroyed.

## 11. Domain vs. Principle / Concept

Domain asks: what sphere, context, material, environment, or relevance field does the route resonate with?

Principle / Concept asks: what abstract logic, meaning, or alignment does the route understand and enact?

Examples:

| Route | Domain | Principle |
|---|---|---|
| Hollow-Depth Listener | Mines, Stone Stress, Echo | Pressure / Warning |
| Thorn-Court Mediator | Courts, Oaths, Social Debt | Reciprocity / Constraint |
| Mech-Synced Ash Cavalier | Ash Fields, Machines, Cavalry Shock | Momentum / Sacrifice |
| Fungal Courier | Wet Roads, Spores, Colony Paths | Continuity / Return |

```yaml
  facet_name: string
  facet_state: latent | active | dormant | corrupted | scarred | externalized | source_local | unstable | severed | mutated | retired
  permissions: []

    D03: []

```

## 13. Rule

Facets are optional, bounded, owner-aware route containers. They define permissions and limits. They cannot bypass owner-file validation and do not become universal mechanics just because one Route has them.

## 1. Principle / Concept definition

A **Principle / Concept** is a D06 Route facet representing structured alignment with an abstract, material, symbolic, social, procedural, elemental, existential, or metaphysical axis.

It may shape perception, methods, techniques, domains, proof interpretation, route identity, resonance, dissonance, contradiction, route development, and high-tier authority.

It does not grant reality overwrite, immunity, conceptual damage, resurrection, domain creation, cosmic rank, universal truth, or law-writing by default.

## 2. Pressure inventory status

Dao, Law, Truth, Aspect, Edict, Intent, Authority, Mandate, Essence, and similar terms are treated as pressure signals, source-local terms, lexicon candidates, or rejected donor terms until formally adopted. Function decides routing.

## 3. Principle categories

A Principle may be:

- material: Glass, Salt, Iron, Bone, Ash;
- elemental: Fire, Storm, Ice, Lightning, Stone;
- procedural: Cutting, Binding, Repair, Pursuit, Reflection;
- existential: Death, Hunger, Memory, Silence, Mercy;
- social: Debt, Oath, Authority, Hospitality, Betrayal;
- spatial / contextual: Threshold, Road, Harbor, Depth, Horizon;
- symbolic: Crown, Mask, Lantern, Bell, Chain;
- compound: Salt-Memory, Blood-Oath, Glass-Mercy, Ash-Repair.

A Principle may be actor-bound, route-bound, domain-bound, faction-bound, territory-bound, source-local, world-facing, or capstone-scale.

## 4. Principle source categories

| Source | Example |
|---|---|
| innate | Ember-born actor has latent Fire / Change pressure. |
| acquired | Surgeon develops Mercy through battlefield triage. |
| taught | Ledger guild teaches Debt through oath accounting. |
| researched | Scholar derives Reflection from mirror relics. |
| awakened | D04 Awakening offers a Principle-shaped route. |
| forced | Cursed relic imposes Hunger. |
| corrupted | Memory route becomes contaminated by Ruin. |
| domain contact | Threshold realm imprints Passage. |
| profession-route | Repair worker's proof becomes Repair. |
| source-local | Donor Dao of Sword remains bounded until normalized. |

## 5. Depth bands

| Depth | Function |
|---|---|
| Trace | Faint mark, sensitivity, latent pressure. |
| Orientation | Principle shapes interpretation and route direction. |
| Practice | Principle shapes methods and bounded techniques. |
| Embodiment | Principle significantly shapes route identity and proof. |
| Authority | Bounded conceptual effects under cost and context. |
| Dominion | High-tier or capstone-scale authority. |
| Mythic / Capstone | World-facing, legacy-facing, or final-form state. |

Low-tier Principles operate at Trace, Orientation, or Practice. Authority, Dominion, and Mythic / Capstone require D04 proof / tier authorization, route development, D03 cost validation if activated, and D07 / D10 handoffs when risk or world consequence exists.

## 6. Low-tier allowed effects

Low-tier Principles may produce:

- interpretive insight;
- relevance upgrade in narrow contexts;
- method shaping;
- Technique candidate creation;
- domain resonance signal;
- proof tagging;
- route pressure;
- minor cost efficiency when aligned;
- warning when dissonant;
- specialized perception.

Example: a **Hollow-Depth Listener** with Pressure may sense dangerous stone stress. They do not automatically crush enemies with gravity.

## 7. Resonance

A Route resonates when an actor acts in a way that coheres with the Principle. Resonance may produce lower cost, better relevance, better information yield, proof quality, technique stability, reduced backlash, route feature availability, or route pressure reinforcement.

Resonance must be bounded. A **Crow-Marked Rooftop Scout** with Watchfulness may gain resonance through patient protection of a district, not by charging into open duel and claiming all awareness benefits.

## 8. Dissonance

A Route enters dissonance when the actor meaningfully contradicts the Principle. Dissonance may produce increased cost, blocked features, reduced relevance, route strain, harmful synthesis, proof conflict, route drift, D07 identity pressure, mutation, or severance risk.

Dissonance should not punish trivial behavior. It should arise when contradiction matters.

Example: a **Truth-bound Diplomat** lies to save a hostage. This does not automatically destroy the Route. It creates contradiction that may strain, scar, reinterpret, mutate toward Mercy, or require reconciliation depending on proof and consequence.

## 9. Contradiction outcomes

## 10. Principle conflicts

Principle conflict types include opposition, overlap, asymmetry, domain suppression, identity conflict, fusion tension, corruption conflict, source-local conflict, and world-scale conflict.

Examples:

- **Salt-Saint of the Drowned Roads** with Return conflicts with **Void Cartographer** with Refusal over whether lost travelers can return.
- **Mech-Synced Ash Cavalier** with Momentum clashes tactically with a Stillness route; this affects technique interaction but does not automatically decide the outcome.
- A fungal courier with Continuity conflicts with sterile machine-domain where decay and spore memory cannot propagate.

Principles provide vulnerabilities as well as power.

## 12. Compound Principles

Compound Principles are allowed but not automatically stronger. They are more specific, more complex, more costly to stabilize, and more likely to generate contradiction or drift.

Examples:

Compound Principles may arise from D05 synthesis, route fusion, profession-route development, contradiction resolution, external anchors, source-local doctrine, or capstone transformation.

## 13. Route Feature Ledger integration

Principle development can create Route Feature Ledger entries such as resonance features, dissonance constraints, Technique candidates, domain interactions, contradiction scars, perception features, dangerous temptations, capstone candidates, or source-local features.

## 1. Purpose

Astra needs personalized and customized feature lists, but not universal class tables. D06 therefore uses the **Route Feature Ledger**.

A Route Feature Ledger is actor-specific or route-instance-specific. It records lawful feature options, active features, candidates, constraints, scars, permissions, refinements, Techniques, Expression candidates, route modifications, source-local entries, blocked entries, and future unlocks generated from that Route Kernel.

## 2. Feature Ledger is not a class table

A class table says: at level X, all members of this class gain Y.

A Route Feature Ledger says: this actor's Route has generated these lawful features or options because of their Path identity, proof, facets, competencies, Expressions, advancement state, scars, anchors, faction relations, and owner-file handoffs.

Fixed tables are allowed only as source-local structures, simplified templates, conversion artifacts, UI convenience layers, or later canon-designed route packages. They are not the default D06 model.

## 3. Sources for Route Features

A Route Feature may arise from:

- Route Kernel facets;
- D04 proof;
- D04 Awakening Package;
- D04 Development Pick;
- D05 competencies;
- D05 methods;
- D05 research tracks;
- D05 synthesis records;

- D07 scars, corruption, harm, mutation, constraints;

- D09 relics, tools, platforms, implants;
- D10 factions, territory, law, institutions;
- source-local grants.

Features are not generated from Archetype alone.

## 4. Required fields

Every Route Feature should define:

- feature name;
- source;
- unlock basis;
- owner file;
- permissions;
- limits;
- state;
- validation state;
- route and facet links;
- handoff refs if cross-system.

A feature cannot exist merely because it sounds thematic.

# D06-05 Technique Architecture and D05 Method Handoff

**Purpose:** Define Technique as a validated formal capability above D05 Method and below or adjacent to D03 Expression when powered.

## Standing D06 boundary

D06 owns formal route identity, Path identity, Route Kernels, route facets, Route Feature Ledgers, Techniques, Principle / Concept route facets, route development, multipath / confluence, profession-route formalization, source-local route conversion, and donor class normalization.

## 1. Core boundary

D05 owns Methods. D06 owns Techniques. D03 owns Expression mechanics when cost, carrier, Power Economy, inscription, burden, activation, confluence, overdraw, or supernatural resource behavior enters.

The progression is:

**D05 Method → Technique Candidate → D06 Formal Technique → D03 Expression-linked Technique if powered.**

Not every named repeatable action becomes a Technique.

## 2. Definitions

### Method

A Method is a repeatable application of competency below formal route technique. Examples: Hammer Rhythm, Depth Reading, Ash-Scent Lure, Breath-Step Pattern, Emergency Clamp Sequence, Hostage Listening Posture.

### Technique Candidate

A candidate is a Method, research output, route feature, donor ability, or source-local capability being evaluated for D06 status. It is not fully authorized.

### Formal Technique

A Technique is a validated formal capability with defined use case, trigger, effect boundary, route relation, training path, limits, and validation state.

### Expression-linked Technique

A Technique that uses or modifies D03 Expression mechanics. D06 owns formal structure. D03 owns cost, carrier, burden, inscription, re-authoring, confluence, and activation.

## 3. Technique categories

Techniques may be martial, craft, social / oath, exploration / survival, platform / interface, companion / bond, esoteric / Principle, source-local, or hybrid.

Examples:

- Martial: Gate-Hinge Shielding, Red-Moon Low Cut, Broken Spear Recovery.
- Craft: Stress-Light Reading, Cold-Seam Repair, Siege-Kettle Ration Cycle.
- Social: Thorn-Court Safe Address, Equal-Cost Bargain, Hostage Breath Pause.
- Exploration: Hollow-Depth Fault Listening, Whiteout Trail Split, Wet-Road Spore Marking.
- Platform: Ash Recoil Dump, Quiet Engine Cant, Drone-Swarm Grid Stitch.
- Companion: Split-Watch Signal, Pack Recall Cut, Colony Whisper Relay.
- Esoteric: Memory Echo Stabilization, Oath-Weight Reading, Glass-Mercy Reflection.
- Source-local: donor maneuver, region-specific ritual, sect-only step art.

## 4. Method-to-Technique validation procedure

When a D05 Method might become a Technique:

1. Identify Method and parent competencies.
2. Confirm repeated use, training, research, proof, route pressure, or teacher validation.
3. Identify Route or facet support.
4. Define trigger and effect boundary.
5. Define limits.
6. Determine if mundane, route-linked, source-local, or powered.
7. Determine D03, D04, D07, D08, D09, D10 handoffs.
8. Add as Technique Candidate.
9. Authorize, block, source-localize, or escalate.

## 5. Technique anatomy

Every formal Technique should define:

- name;
- source;
- Route / facet relation;
- category;
- trigger;
- action posture;
- effect boundary;
- limits;
- cost;

- training path;

## 8. Technique vs. Expression boundary

A Technique remains D06-only when it is formal structure, training, procedure, or route capability without D03 Power Economy mechanics.

A Technique requires D03 handoff when it uses Mistra / Zi / Volith / other Power Economies, cost partition, carrier burden, resource pool, recharge, Expression inscription, confluence, overdraw, backlash, supernatural activation, or re-authoring.

Examples:

- **Oath-Weight Reading** as D06 Technique reads social / oath tension through trained court procedure and route sense.

- **Quiet Engine Cant** as D06 Technique synchronizes engine crew rhythm.
- **Engine-Soul Overchant** as D03 / D09 Expression-linked Technique forces impossible engine output with cost and feedback risk.

## 9. Technique Sets

A Technique Set is a related family of Techniques associated with Route, facet, teacher, lineage, source-local system, profession-route, or external anchor. It can be fixed, branching, research-grown, method-grown, teacher-gated, route-ledger generated, source-local, or anchor-dependent. It is not a class table by default.

## 10. Failure modes

Technique failure may involve misfire, overstrain, backlash, misread, anchor fault, domain mismatch, dissonance, collateral, source-local failure, or corruption leak. D02 handles resolution when rolled. Owner files resolve consequences.

## 11. Donor ability normalization

Donor spells, maneuvers, feats, class features, rituals, powers, cultivation arts, protocols, and companion commands may map to D05 Method, D06 Technique, D03 Expression, Route Feature, source-local entry, or escalation. They are not imported as-is.

# D06-06 Route Development, Branches, Drift, and Mutation

**File ID:** `D06-06`  
**Pack:** Astra Ascension D06 — Route, Path, Principle, Technique, and Donor Class Normalization Doctrine  
**Version:** v0_1  
**Status:** Design doctrine draft generated from accepted D06 decisions.  
**Purpose:** Define governed route evolution after authorization.

## Standing D06 boundary

D06 owns formal route identity, Path identity, Route Kernels, route facets, Route Feature Ledgers, Techniques, Principle / Concept route facets, route development, multipath / confluence, profession-route formalization, source-local route conversion, and donor class normalization.

D06 does **not** own D03 Power Economy costs, carriers, Expressions, reservoirs, burdens, or confluence math; D04 proof sufficiency, level, tier, Awakening, Development Picks, breakthroughs, capstone, or Ascension Access; D05 skills, professions, knowledge, training, research, methods, and synthesis substrate; D07 harm, corruption, madness, mutation harm, overload, identity erosion, or recovery; D08 companions, summons, spirits, AI, bonds, or nonhuman actor-state mechanics; D09 relics, tools, implants, weapons, vehicles, ships, mechs, or platforms; D10 factions, rank, law, reputation, territory, settlement, economy, or world-state; final runtime/database schemas; canon sourcebook prose; or live-play GM behavior.

D06 records candidates, structure, permissions, limits, and owner-file handoffs. It does not use route language to bypass the owners above.


## 1. Core model

Routes are stable enough to matter, but not static. D06 uses **Governed Route Evolution**. Route change requires lawful pressure such as D04 proof, D04 Development Pick, D04 breakthrough, D05 training, D05 research, D05 method, D05 synthesis, D03 Expression / Power Economy change, D06 Principle contradiction, D07 harm / corruption, D08 bond event, D09 anchor event, D10 faction / world event, source-local rule, or capstone event.

## 2. Development vs. mutation

**Route development** means the Route grows while remaining itself. Examples: Far Fisher gains better lure Technique; Glass-Tomb Archivist deepens Memory resonance; Mech-Synced Ash Cavalier gains safer recoil dump.

**Route mutation** means the Route's identity or structure changes. Examples: Far Fisher becomes Abyssal Baitbearer after repeated self-sacrificial lure proof; Glass-Tomb Archivist becomes Dead-Record Host after identity erosion; Mech-Synced Ash Cavalier becomes Ash-Wreck Remnant after mech destruction.

Development is normal. Mutation is major and proof-heavy.

## 3. Route development states

| State | Meaning |
|---|---|
| stable | functioning as authorized. |
| developing | gaining features, techniques, domains, refinements. |
| branching | subordinate direction appears. |
| drifting | route pressure pulls away from current identity. |
| scarred | lasting wound, cost, trauma, contradiction, or mark. |
| mutating | route structure changes into something different. |
| stabilizing | actor repairs or grounds instability. |
| externalizing | route function moves into anchor. |
| dormant | route exists but is inactive. |
| blocked | route cannot progress due to missing basis. |
| severing | route is being cut away or lost. |
| fusing | route is merging with another. |
| source-localizing | route is being bounded. |
| retiring | route no longer develops but remains history. |
| escalating | missing doctrine requires review. |

## 4. Branches

A Route Branch is a subordinate development direction inside an existing Route. It may unlock features, Techniques, Domain relations, Principle depth, training projects, research tracks, or D04 Development Pick targets. A branch is not automatically a second Route.

Examples:

- Far Fisher → Impossible Bait, Deep Current Reading, Long-Line Pursuit.
- Glass-Tomb Archivist → Dead-Language Indexing, Memory Echo Stabilization.
- Siege-Kettle Quartermaster → Morale Rationing, Siege Logistics.
- Fungal Courier → Spore Memory, Humid Roadfinding, Colony Relay.

## 5. Drift

Route drift occurs when proof, actions, methods, contradictions, or external pressures pull the Route away from current identity. Drift may be value drift, Domain drift, Principle drift, Technique drift, profession drift, anchor drift, corruption drift, faction drift, survival drift, or source-local drift. Drift is not always bad, but it must be recorded when it affects permissions, limits, features, or proof.

## 6. Scars

A Route Scar is a lasting mark on route identity caused by failure, contradiction, breakthrough, harm, coercion, sacrifice, corruption, severance attempt, or world-state consequence. ## 7. Stabilization

Unstable Routes can stabilize through D05 training, D05 research, D04 proof, D07 recovery, D03 economy correction, D08 bond repair, D09 anchor repair, D10 faction reconciliation, Principle reinterpretation, feature retirement, or source-local containment.

## 8. Externalization, dormancy, blockage, severance

Externalization occurs when route function becomes dependent on an external anchor. Dormancy preserves inactive route history. Blockage prevents progress until missing basis resolves. Severance is consequential loss, rejection, cutting, removal, or destruction of route structure. It is not casual respec.

## 1. Core model

D06 uses **Governed Multipath and Route Confluence**. Actors may have multiple route pressures and, with authorization, multiple active Routes. Multiple Routes do not automatically stack features, Techniques, Domains, Principles, or Development Pick permissions.

## 2. Multipath vs. fusion

**Multipath** means the actor has more than one active or semi-active route structure. The Routes remain distinct.

**Fusion** means two or more Routes become a new combined identity or route structure. Fusion is heavier and requires proof, compatibility or contradiction resolution, rewritten permissions / limits, owner handoffs, and state delta.

## 3. Relationship states

Route relationships may be independent, primary / secondary, latent, dormant, subordinate, parallel, entangled, synergistic, conflicted, dominating, fusing, fused, splitting, severed, source-local, or escalated.

## 4. Compatibility bands

| Band | Meaning |
|---|---|
| compatible | coexist cleanly. |
| complementary | support one another in bounded ways. |
| tense | coexist with friction. |
| contradictory | pull against each other and require management. |
| incompatible | cannot remain active together without mutation, suppression, severance, or exception. |
| unknown | requires research, proof, teacher, or validation. |
| source-local | compatibility valid only inside bounded context. |

Compatibility is function-based, not name-based. Mercy and Severance may conflict or fuse as Mercy-Through-Severance depending on proof.

## 5. Route Confluence

A Route Confluence is a governed interaction between Routes or route facets. It may produce shared features, limited cross-route technique access, compound Principle, merged Domain relation, branch, scar, conflict, mutation, fusion candidate, severance pressure, source-local exception, or owner-file handoff.

Route Confluence is not automatic feature stacking.

## 6. Feature Ledger interaction

Rules:

1. Features do not automatically transfer between Routes.
2. A feature may become cross-route only if Route Confluence authorizes it.
3. A feature may be shared, duplicated, transformed, blocked, retired, or fused.
4. If two features overlap, the actor does not automatically get both benefits.
5. D04 validates Development Pick use.
6. D03 validates powered feature interactions.
7. D07 validates overload, contradiction, corruption, and identity damage.

## 7. Multipath load

Multiple Routes create opportunity and burden. Burdens may include feature conflict, Development Pick dilution, training strain, proof fragmentation, Principle dissonance, identity instability, anchor dependency, social obligation conflict, source-local lock, D03 carrier/resource incompatibility, or D07 overload.

## 8. Fusion gates

Fusion should require:

- at least two active or strong candidate Routes / facets;
- proof of meaningful interaction;
- compatibility or contradiction-resolution basis;
- defined new identity or compound structure;
- retained, transformed, retired, or blocked features identified;
- permissions and limits rewritten;
- owner-file handoffs resolved;
- D04 authorization if advancement-relevant;
- D07 review if identity harm or overload is involved;
- state delta recorded.

## 9. D03 confluence vs. D06 confluence

D03 confluence handles Power Economy interaction, cost partition, carrier tolerance, Expression re-authoring, energetic compatibility, overdraw, backlash, and resource burden.

D06 confluence handles route identity interaction, facet compatibility, Principle / Domain overlap, Technique Set interaction, Route Feature Ledger interaction, route fusion, mutation, and severance.

Both may apply, but they are not the same system.

## 10. Source-local multiclass handling

Donor multiclass systems normalize into source-local Routes, route facets, Technique Sets, Route Features, D05 competencies, D03 Expressions, D04 picks, or escalation. Donor multiclass rules are not imported directly.

## 11. Varied examples

# D06-11 Integration Checklists and DDR Register

**Pack:** Astra Ascension D06 — Route, Path, Principle, Technique, and Donor Class Normalization Doctrine

## Standing D06 boundary

D06 owns formal route identity, Path identity, Route Kernels, route facets, Route Feature Ledgers, Techniques, Principle / Concept route facets, route development, multipath / confluence, profession-route formalization, source-local route conversion, and donor class normalization.

D06 does **not** own D03 Power Economy costs, carriers, Expressions, reservoirs, burdens, or confluence math; D04 proof sufficiency, level, tier, Awakening, Development Picks, breakthroughs, capstone, or Ascension Access; D05 skills, professions, knowledge, training, research, methods, and synthesis substrate; D07 harm, corruption, madness, mutation harm, overload, identity erosion, or recovery; D08 companions, summons, spirits, AI, bonds, or nonhuman actor-state mechanics; D09 relics, tools, implants, weapons, vehicles, ships, mechs, or platforms; D10 factions, rank, law, reputation, territory, settlement, economy, or world-state; final runtime/database schemas; canon sourcebook prose; or live-play GM behavior.

D06 records candidates, structure, permissions, limits, and owner-file handoffs. It does not use route language to bypass the owners above.


## 1. Accepted decision register

D06 accepted these blocks:

1. **Route Kernel Opening** — Route is technical umbrella; Path is player-facing; Class remains donor-facing; Route Kernel uses typed facets; D05 pressure requires D04 / D06 authorization.
2. **Route Authorization and Path Identity** — Routes require Authorization Packets, permissions, limits, states, bounded candidate display, source-local donor handling, low-tier Principle limits.
3. **Route Facets, Domains, and Archetypes** — Optional typed facets; Domain is bounded resonance; Archetype is role-shape; External Anchors hand off; Source-local and Constraint facets are formal.
4. **Route Feature Ledgers** — Personalized feature ledgers, not class tables; features require source, unlock basis, owner, permissions, limits, state, validation.
5. **Principle / Concept Authority** — Principle is route facet, not universal law; depth bands; resonance / dissonance; contradiction; risks; compound rules; donor-term containment.
6. **Technique Architecture** — Technique is validated capability above Method; D03 handoff for powered Techniques; broad categories; Technique states; donor ability normalization.
7. **Route Evolution** — Governed development, branches, drift, scars, mutation, stabilization, externalization, dormancy, blockage, severance, fusion, retirement.
8. **Multipath / Confluence** — Governed multipath; fusion distinct; compatibility bands; no automatic feature stacking; D03 / D06 confluence distinction.
9. **Profession / School / Faction Sources** — Structured source model; Profession-Route requires D05 + D04 + D06; faction rank separate; D10 boundary preserved.
10. **Donor Normalization** — Functional decomposition and source-local normalization; donor packages are evidence; fixed tables contained; normalization records required.
11. **Runtime / UI Handoff** — Tiered route records; record families; durable thresholds; partial visibility; UI grouping; owner handoff; source-local durability.

## 2. Ownership checklist

D06 owns formal Route structure. If a route element touches Power Economy, proof, skill/training/research, harm, companion/AI, item/platform, faction/territory/world-state, lexicon promotion, final runtime schema, or live-play behavior, it must hand off.

## 3. Route Authorization Checklist

A Route may become active only if it has lawful source, authorization basis, Path identity, route function, permissions, limits, facets, owner handoffs, route state, validation state, actor-state delta, bounded player-facing choice if multiple candidates exist, and source-local boundary when needed.

Failure states include pending proof, pending choice, pending owner file, source-local, blocked, dangerous, quarantined, or escalated.

## 4. Route Facet Checklist

Every facet must define type, contribution, permissions, limits, state, owner handoffs, validation, and boundary from other facets. Domain is not territory control. Archetype is not subclass. Principle is not universal law. External Anchor is not owned entirely by D06. Source-local remains bounded. Constraint handoff to D07 where harm or corruption exists.

## 5. Route Feature Ledger Checklist

A feature is lawful only if it has name, source, unlock basis, owner file, permissions, limits, state, validation, route/facet link, and handoff refs. Red flags: feature exists only because thematic, grants late-tier effects early, bypasses D03/D04, stacks across routes, imports donor table behavior, or hides faction/item/companion/world mechanics.

## 6. Principle Checklist

Principle entries must define terminology status, source, depth, scope, resonance, dissonance, permissions, limits, contradiction state, risks, handoffs, route feature links, proof, and validation. Low-tier effects are perception, method shaping, route pressure, domain resonance, bounded technique, proof tagging, and warning. High-tier authority requires D04/D03/D07/D10 as relevant.

## 7. Technique Checklist

Technique must define source, category, state, route/facet relation, trigger, action posture, effect boundary, limits, cost profile, training path, risks, proof refs, handoffs, and validation. Repeatable competency use is D05 Method. Powered activation is D03 Expression-linked. Item, companion, and faction procedures hand off to D09, D08, and D10.

## 8. Route Development Checklist

Record trigger source, affected facets, features, Techniques, permissions added/removed, limits added/removed, proof, training/research, handoffs, and result state. Development preserves identity. Mutation changes identity. Severance is not casual respec.

## 9. Multipath / Confluence Checklist

Check interacting routes, trigger, relationship state, compatibility, feature transfer, authorization, D03 confluence, D07 overload/corruption, source-local multiclass leakage, and validation. Multiple Routes do not automatically stack.

## 10. Profession / School / Faction Checklist

Separate D05 profession/training, D10 institution/rank/law, and D06 route source. Profession alone is not a route. Teacher alone is not route authorization. Faction rank alone is not Faction Route. School curriculum alone is not a Path table.

## 11. Donor Normalization Checklist

For every donor class-like package: identify construct type, separate identity from mechanics, identify route-facing and non-route material, route each element to owner file, mark source-local status, reject donor assumptions, preserve useful candidate identity, create normalization record, quarantine unresolved constructs, escalate missing frameworks.

Rejected assumptions include donor class table progression, donor class resources as universal, spell slots as default, donor action economy, cosmology, level pacing, balance assumptions, equipment assumptions, alignment law, metaphysical origin, species/class dependency, and multiclass stacking.

## 12. Runtime / UI Checklist

UI groups: Path Summary, Facets, Features, Techniques, Growth, Risks / Constraints, Sources, Multipath, Conversion Notes. Durable records are required for route identity, authorization, facets, features, Techniques, Principle depth / contradiction / risk, development, confluence, source, donor normalization, source-local boundary, and handoff changes. Partial visibility must be fair and discoverable.
