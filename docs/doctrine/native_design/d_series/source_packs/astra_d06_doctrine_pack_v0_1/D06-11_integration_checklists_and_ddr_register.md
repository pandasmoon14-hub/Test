## Standing D06 boundary

D06 owns formal route identity, Path identity, Route Kernels, route facets, Route Feature Ledgers, Techniques, Principle / Concept route facets, route development, multipath / confluence, profession-route formalization, source-local route conversion, and donor class normalization.

D06 does **not** own D03 Power Economy costs, carriers, Expressions, reservoirs, burdens, or confluence math; D04 proof sufficiency, level, tier, Awakening, Development Picks, breakthroughs, capstone, or Ascension Access; D05 skills, professions, knowledge, training, research, methods, and synthesis substrate; D07 harm, corruption, madness, mutation harm, overload, identity erosion, or recovery; D08 companions, summons, spirits, AI, bonds, or nonhuman actor-state mechanics; D09 relics, tools, implants, weapons, vehicles, ships, mechs, or platforms; D10 factions, rank, law, reputation, territory, settlement, economy, or world-state; final runtime/database schemas; canon sourcebook prose; or live-play GM behavior.

D06 records candidates, structure, permissions, limits, and owner-file handoffs. It does not use route language to bypass the owners above.


## 1. Accepted decision register

D06 accepted these blocks:

1. **Route Kernel Opening** — Route is technical umbrella; Path is player-facing; Class remains donor-facing; Route Kernel uses typed facets; D05 pressure requires D04 / D06 authorization.
2. **Route Authorization and Path Identity** — Routes require Authorization Packets, permissions, limits, states, bounded candidate display, source-local donor handling, low-tier Principle limits.
3. **Route Facets, Domains, and Archetypes** — Optional typed facets; Domain is bounded resonance; Archetype is role-shape; External Anchors hand off; Source-local and Constraint facets are formal.
4. **Route Feature Ledgers** — Personalized feature ledgers, not class tables; features require source, unlock basis, owner, permissions, limits, state, validation.
5. **Principle / Concept Authority** — Principle is route facet, not universal law; depth bands; resonance / dissonance; contradiction; risks; compound rules; donor-term containment.
6. **Technique Architecture** — Technique is validated capability above Method; D03 handoff for powered Techniques; broad categories; Technique states; donor ability normalization.
7. **Route Evolution** — Governed development, branches, drift, scars, mutation, stabilization, externalization, dormancy, blockage, severance, fusion, retirement.
8. **Multipath / Confluence** — Governed multipath; fusion distinct; compatibility bands; no automatic feature stacking; D03 / D06 confluence distinction.
9. **Profession / School / Faction Sources** — Structured source model; Profession-Route requires D05 + D04 + D06; faction rank separate; D10 boundary preserved.
10. **Donor Normalization** — Functional decomposition and source-local normalization; donor packages are evidence; fixed tables contained; normalization records required.
11. **Runtime / UI Handoff** — Tiered route records; record families; durable thresholds; partial visibility; UI grouping; owner handoff; source-local durability.

## 2. Ownership checklist

D06 owns formal Route structure. If a route element touches Power Economy, proof, skill/training/research, harm, companion/AI, item/platform, faction/territory/world-state, lexicon promotion, final runtime schema, or live-play behavior, it must hand off.

## 3. Route Authorization Checklist

A Route may become active only if it has lawful source, authorization basis, Path identity, route function, permissions, limits, facets, owner handoffs, route state, validation state, actor-state delta, bounded player-facing choice if multiple candidates exist, and source-local boundary when needed.

Failure states include pending proof, pending choice, pending owner file, source-local, blocked, dangerous, quarantined, or escalated.

## 4. Route Facet Checklist

Every facet must define type, contribution, permissions, limits, state, owner handoffs, validation, and boundary from other facets. Domain is not territory control. Archetype is not subclass. Principle is not universal law. External Anchor is not owned entirely by D06. Source-local remains bounded. Constraint handoff to D07 where harm or corruption exists.

## 5. Route Feature Ledger Checklist

A feature is lawful only if it has name, source, unlock basis, owner file, permissions, limits, state, validation, route/facet link, and handoff refs. Red flags: feature exists only because thematic, grants late-tier effects early, bypasses D03/D04, stacks across routes, imports donor table behavior, or hides faction/item/companion/world mechanics.

## 6. Principle Checklist

Principle entries must define terminology status, source, depth, scope, resonance, dissonance, permissions, limits, contradiction state, risks, handoffs, route feature links, proof, and validation. Low-tier effects are perception, method shaping, route pressure, domain resonance, bounded technique, proof tagging, and warning. High-tier authority requires D04/D03/D07/D10 as relevant.

## 7. Technique Checklist

Technique must define source, category, state, route/facet relation, trigger, action posture, effect boundary, limits, cost profile, training path, risks, proof refs, handoffs, and validation. Repeatable competency use is D05 Method. Powered activation is D03 Expression-linked. Item, companion, and faction procedures hand off to D09, D08, and D10.

## 8. Route Development Checklist

Record trigger source, affected facets, features, Techniques, permissions added/removed, limits added/removed, proof, training/research, handoffs, and result state. Development preserves identity. Mutation changes identity. Severance is not casual respec.

## 9. Multipath / Confluence Checklist

Check interacting routes, trigger, relationship state, compatibility, feature transfer, authorization, D03 confluence, D07 overload/corruption, source-local multiclass leakage, and validation. Multiple Routes do not automatically stack.

## 10. Profession / School / Faction Checklist

Separate D05 profession/training, D10 institution/rank/law, and D06 route source. Profession alone is not a route. Teacher alone is not route authorization. Faction rank alone is not Faction Route. School curriculum alone is not a Path table.

## 11. Donor Normalization Checklist

For every donor class-like package: identify construct type, separate identity from mechanics, identify route-facing and non-route material, route each element to owner file, mark source-local status, reject donor assumptions, preserve useful candidate identity, create normalization record, quarantine unresolved constructs, escalate missing frameworks.

Rejected assumptions include donor class table progression, donor class resources as universal, spell slots as default, donor action economy, cosmology, level pacing, balance assumptions, equipment assumptions, alignment law, metaphysical origin, species/class dependency, and multiclass stacking.

## 12. Runtime / UI Checklist

UI groups: Path Summary, Facets, Features, Techniques, Growth, Risks / Constraints, Sources, Multipath, Conversion Notes. Durable records are required for route identity, authorization, facets, features, Techniques, Principle depth / contradiction / risk, development, confluence, source, donor normalization, source-local boundary, and handoff changes. Partial visibility must be fair and discoverable.
