## Standing D06 boundary

D06 owns formal route identity, Path identity, Route Kernels, route facets, Route Feature Ledgers, Techniques, Principle / Concept route facets, route development, multipath / confluence, profession-route formalization, source-local route conversion, and donor class normalization.

D06 does **not** own D03 Power Economy costs, carriers, Expressions, reservoirs, burdens, or confluence math; D04 proof sufficiency, level, tier, Awakening, Development Picks, breakthroughs, capstone, or Ascension Access; D05 skills, professions, knowledge, training, research, methods, and synthesis substrate; D07 harm, corruption, madness, mutation harm, overload, identity erosion, or recovery; D08 companions, summons, spirits, AI, bonds, or nonhuman actor-state mechanics; D09 relics, tools, implants, weapons, vehicles, ships, mechs, or platforms; D10 factions, rank, law, reputation, territory, settlement, economy, or world-state; final runtime/database schemas; canon sourcebook prose; or live-play GM behavior.

D06 records candidates, structure, permissions, limits, and owner-file handoffs. It does not use route language to bypass the owners above.


## Pack purpose

D06 defines Astra's formal route-identity layer. It converts route pressure into lawful Route structure, separates Path from donor class terminology, defines typed route facets, contains Principle / Concept authority, formalizes Techniques above D05 Methods, supports personalized Route Feature Ledgers, governs route development and multipath, and normalizes class-like donor packages without importing donor structure as hidden law.

This pack assumes D03, D04, and D05 have already been reconciled enough to provide the following upstream structures:

- D03: Power Economies, Expressions, carrier / burden / cost / confluence concepts, and anti-unauthorized-expression boundaries.
- D04: proof, Awakening Packages, Development Picks, tier transitions, breakthrough payloads, capstone, and Ascension Access.
- D05: competencies, professions, knowledge, training, research, methods, synthesis, route pressure, records, and advancement-track routing.

D06 feeds later D07, D08, D09, and D10. It deliberately does not solve harm systems, actor / companion systems, item / platform systems, or faction / territory / world-state systems.

## D06 core law

A Route is the technical umbrella for formal route identity and structured development direction. A Path is the player-facing identity of a Route. A Route is not a donor class table, not a profession, not a skill, not a feature list, not a spell list, not a faction rank, and not a donor package with renamed nouns.

D06 may create Route candidates, Route Authorization Packets, Route Facets, Route Feature Ledgers, Principle / Concept facets, Technique candidates, source-local retained structures, and owner-file handoffs. Route identity becomes active only through lawful authorization.

## Required anti-collapse rules

1. Route is not class.
2. Path is not donor class name import.
3. Archetype is not subclass.
4. Route Feature Ledger is not freeform class-feature generation.
5. Technique is not every named action.
6. Principle / Concept is not early reality override.
7. Profession is not automatic Path.
8. Faction rank is not Faction Route.
9. Multipath is not multiclass stacking.
10. Donor package is evidence, not authority.
11. Source-local boundaries are durable and explicit.
12. Record shapes are doctrine handoff structures, not final schemas.

## Post-D06 expectations

After D06 generation, perform a short D03 / D04 / D05 patch-note check. A full amendment pass is required only if D06 creates contradictions or new required owner interfaces. D07 should follow because D06 defers route scars, corruption, compulsion, overload, shattering, identity erosion, and route harm to D07.
