# D01-05 Conversion Mapping and Source-local Boundaries

## Purpose

This file defines how donor attributes and attribute-adjacent systems map into D01 without importing donor math.

## Standard donor mapping signals

Common donor mappings:

| Donor pressure | Astra candidate |
|---|---|
| Strength / might / physical force | Vessel |
| Dexterity / agility / speed / coordination | Finesse |
| Intelligence / logic / technical thought | Reason |
| Wisdom / perception / intuition | Insight |
| Constitution / endurance / toughness | Integrity |
| Charisma / presence / charm / command | Glamour |
| Soul / spirit / ki / qi / essence / will as inner force | Animus |
| Luck / fate / karma / fortune | Not automatically Pneuma |

These are mapping candidates, not automatic imports.

## Luck/fate handling

Donor luck and fate systems must be routed by function.

Possible outcomes:
- D03 spendable resource;
- D06 fate/fortune Technique or Route feature;
- D10 public myth/reputation/world-state;
- hidden Pneuma pressure only if doctrine-authorized;
- source-local retention;
- quarantine/escalation.

Do not map donor luck to Pneuma by default.

## Non-six-stat donors

Donor systems may use:
- body/mind/soul;
- approaches;
- attributes plus skills;
- profession-only systems;
- narrative tags;
- cultivation roots;
- cyberware capacities;
- sanity/stress axes;
- psionic disciplines;
- vehicle interface stats;
- companion stats;
- source-local traits.

D01 maps by function, not label.

## Source-local retention

Retain donor attribute systems source-locally when:
- donor math is inseparable from source-local procedure;
- the attribute is local genre machinery;
- mapping would erase important subsystem behavior;
- later doctrine is missing;
- donor construct is not meant to generalize.

## Quarantine and escalation

Quarantine if:
- attribute combines actor, resource, route, harm, and world-state in one bundle;
- donor math would overwrite Astra formula boundaries;
- fate/luck/cosmic pressure threatens Pneuma boundary;
- cyberware/humanity/sanity systems are actually D07/D08/D10 pressure;
- construct cannot be mapped without new doctrine.

Escalate if repeated donor pressure shows a missing Astra framework.

## Acceptance criteria

A conversion mapping is valid when it:

1. maps by function rather than donor label;
2. rejects donor math by default;
3. protects Pneuma;
4. preserves source-local systems when needed;
5. assigns owner handoffs for bundled constructs.
