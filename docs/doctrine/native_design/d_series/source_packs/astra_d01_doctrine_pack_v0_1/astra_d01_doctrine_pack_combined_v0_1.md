# D01-00 Attribute Architecture and Design Boundary

## Purpose

This file defines the purpose and boundary of Astra’s attribute architecture.

## Core rule

Attributes are durable character axes used to interpret capability, pressure, breakthrough, conversion, and derived-system handoffs. They are not donor ability scores imported with donor assumptions.

## Attribute accessibility

D01 deliberately uses familiar functional territory so players and converters can orient quickly, but donor labels do not import donor math.

Astra may map donor STR/DEX/INT/WIS/CON/CHA-like constructs into D01 axes, but donor saving throws, skill lists, class scaling, modifiers, point-buy assumptions, proficiency math, and derived-stat formulas are not imported.

## Visible and hidden axes

D01 uses:

- seven visible player-facing attributes;
- one hidden backend pressure axis.

Visible attributes are part of character-facing identity and capability.

Pneuma is not visible as a score, pool, stat, spendable currency, or ordinary build axis.

## Attribute vs. skill vs. route

Attributes describe broad capability and pressure.

Skills/methods belong to D05.

Routes, Techniques, and Principles belong to D06.

Advancement and breakthroughs belong to D04, but D01 defines the attribute lattice substrate that D04 can use.

## Corpus-scale requirement

The attribute system must absorb donor families including fantasy ability scores, sci-fi attributes, cultivation body/spirit/mind axes, narrative approaches, professions, cyberware effects, psionic stats, horror sanity-adjacent pressure, vehicle/mech interfaces, companion systems, and source-local advancement tracks.

## Acceptance criteria

A D01 attribute rule is valid when it:

1. preserves the seven visible axes and hidden Pneuma axis;
2. avoids donor math leakage;
3. distinguishes capability from skill/method and route/power;
4. supports multi-axis breakthroughs;
5. routes derived pools and consequences to owner files.

# D01-01 Visible Attribute Set

## Purpose

This file defines the seven visible attributes.

## Vessel

Vessel is body as container, force, load-bearing, physical capacity, corporeal expression, and form-change relevance.

Vessel may inform:
- lifting, carrying, striking, grappling, physical exertion;
- bodily capacity;
- form-shift tolerance;
- body cultivation;
- corruption resistance by physical substrate when owner files validate it;
- platform/body interface where D08 applies.

Vessel is not donor Strength imported wholesale.

## Finesse

Finesse is speed, precision, reflex, kinetic control, dexterity, coordination, balance, and fine action.

Finesse may inform:
- precise movement;
- initiative/tempo pressure where D02/D07 later define it;
- stealthy or delicate methods through D05;
- ranged or controlled action;
- fine tool handling;
- vehicle/control interfaces where D05/D09 apply.

Finesse is not donor Dexterity imported wholesale.

## Reason

Reason is cognitive processing, planning, analysis, technical comprehension, memory organization, formal interpretation, strategy, and abstract problem solving.

Reason may inform:
- interpretation after Insight detects signal;
- research structure through D05;
- technical tasks;
- planning;
- focus pool handoffs if later authorized;
- inference from records and archives.

Reason does not replace Insight.

## Insight

Insight is perception, pattern recognition, intuition, hidden-state reading, emotional/environmental sensitivity, and assessment initiation.

Insight may inform:
- assessment action;
- reading opposition or place-state;
- detecting false information;
- sensing hidden pressure;
- noticing route/environment cues;
- social or spiritual perception where owner files validate it.

Insight probes. Reason interprets.

## Integrity

Integrity is resilience, structural coherence, identity stability, endurance-like pressure, continuity under strain, and resistance to breakdown.

Integrity may inform:
- vitality threshold;
- injury absorption;
- recovery pressure;
- corruption/backlash tolerance where D07 validates;
- form-state stability where D08 validates;
- identity stability;
- body/soul/structure coherence.

Integrity is not donor Constitution imported wholesale.

## Glamour

Glamour is social force, presence, projection, performance, charm, command aura, symbolic authority, mask, illusion-adjacent expression, and reality-facing presence.

Glamour may inform:
- persuasion and social pressure through D05;
- public identity and reputation pressure through D10;
- illusion/performance if routed to D06;
- faction presence;
- command reach;
- symbolic legitimacy.

Glamour is not donor Charisma imported wholesale.

## Animus

Animus is inner force, soul/spirit/will, cultivation pressure, personal power source axis, spiritual drive, and sustained self-projection.

Animus may inform:
- power pool handoffs;
- corruption resistance through D07 if validated;
- sustained focus;
- soul/spirit style action;
- route compatibility;
- breakthrough pressure;
- cultivation-style source models.

Animus is visible and player-facing.

## Attribute relation principles

1. Attributes overlap in application but remain distinct in doctrine.
2. Skill/method determines how an attribute is applied.
3. Route/Technique determines extraordinary expression.
4. Owner files validate derived pools, harm, advancement, actor changes, and world consequences.
5. No donor attribute label imports donor math.

## Acceptance criteria

The visible attribute set is valid when each attribute:

1. has a clear functional territory;
2. avoids donor ability-score leakage;
3. supports conversion from multiple genres;
4. can hand off to D03–D10 without owning their mechanics.

# D01-02 Animus and Pneuma Doctrine

## Purpose

This file distinguishes Animus from Pneuma and protects Pneuma from becoming a visible luck stat.

## Animus

Animus is visible, character-facing, and build-relevant. It represents inner force, soul/spirit/will, self-projection, cultivation source, and power-bearing pressure.

Animus may connect to:

- Power Economy handoffs to D03;
- route compatibility through D06;
- spiritual or soul-facing pressures through D07/D08;
- sustained focus;
- breakthrough lattice pressure;
- source-local spirit/ki/qi/essence/will constructs.

Animus is not automatically the same as every donor soul, mana, ki, faith, psionic, or willpower system. Donor constructs must be mapped by function.

## Pneuma

Pneuma is hidden, backend-only, and world-facing. It represents fate/luck/cosmic vital breath pressure, but not as player-visible luck score or spendable resource.

Pneuma may influence rare cosmic/world-facing modifiers, hidden alignment pressure, relic/cosmic resonance, or major fate-marked events.

Pneuma should not be displayed as a score, optimized as a build stat, spent as currency, leveled directly, or used as ordinary luck.

## Pneuma shift triggers

Pneuma may shift only through rare and significant events, such as:

- cosmically significant moments;
- major breakthroughs;
- rare relics/quests with Pneuma tag;
- large-scale fate/cosmic pressure;
- rare narrative events validated by owner doctrine;
- source-local cosmic/fate systems retained under boundary.

Pneuma has no ordinary pool.

## Pneuma visibility

Pneuma should surface only through cryptic, indirect, or consequence-facing signals when appropriate.

Possible signals:
- omen;
- strange coincidence;
- impossible timing;
- relic resonance;
- public myth;
- hidden modifier;
- cosmic attention;
- source-local sign.

D10 may record world-facing consequences if Pneuma pressure becomes public, rumored, mythologized, or faction-relevant.

## Relic and method interaction

When a Pneuma-tagged relic or method is used, the backend may evaluate:

- declared use method;
- current PCR;
- source-local boundary;
- user compatibility;
- world-state pressure.

The same relic and method may produce different Pneuma-adjacent outcomes in different places.

## Conversion boundary

Donor luck, karma, fate points, hero points, destiny, fortune, divine favor, prophecy points, and similar systems do not automatically map to Pneuma.

Possible outcomes:
- map to D03 spendable resource;
- map to D10 reputation/myth/world-state;
- map to D06 route/fate Technique;
- retain source-locally;
- quarantine;
- escalate.

## Acceptance criteria

Animus/Pneuma doctrine is valid when it:

1. keeps Animus visible and build-relevant;
2. keeps Pneuma hidden and backend-only;
3. prevents Pneuma from becoming luck currency;
4. supports rare cosmic pressure;
5. routes donor fate/luck systems by function rather than label.

# D01-03 Multi-Axis Breakthrough Lattices

## Purpose

This file defines D01’s attribute breakthrough substrate.

## Core rule

Each visible attribute may have an independent threshold lattice. Breakthroughs are not automatically whole-character level-ups.

The seven visible breakthrough lattices are:

- Vessel lattice;
- Finesse lattice;
- Reason lattice;
- Insight lattice;
- Integrity lattice;
- Glamour lattice;
- Animus lattice.

Pneuma has no ordinary breakthrough lattice.

## Breakthrough lattice fields

Each visible attribute may track:

```yaml
attribute_breakthrough_lattice:
  attribute_ref: string
  current_value: string
  threshold_state: string
  breakthrough_eligibility: string
  breakthrough_history: []
  post_breakthrough_altered_state: string
  owner_handoffs:
    D03: []
    D04: []
    D06: []
    D07: []
    D08: []
    D10: []
```

D01 defines the existence of these fields. D04 owns advancement gates, proof, and breakthrough procedure.

## Threshold state

Possible threshold states:

- stable;
- pressured;
- saturated;
- unstable;
- eligible;
- blocked;
- divergent;
- corrupted;
- transformed;
- source-local.

D07 owns corruption/harm outcomes. D08 owns actor/form changes. D04 owns advancement logic.

## Breakthrough history

Breakthrough history matters because long campaigns need path dependency.

Record:
- attribute;
- trigger;
- cost;
- consequence;
- owner handoffs;
- altered state;
- source-local boundary;
- historical retention.

D10 may record public breakthrough consequence if socially visible.

## Independent but interacting axes

Breakthrough axes are independent, but may interact.

Examples:
- Vessel breakthrough affects body capacity but not Reason automatically.
- Animus breakthrough may enable power carrier expansion if D03/D04 validate.
- Integrity breakthrough may stabilize a transformation if D07/D08 validate.
- Insight breakthrough may improve assessment potential if D02/D05 validate.
- Glamour breakthrough may affect faction presence or social legitimacy if D10 validates.

## Multi-axis breakthroughs

Rare breakthroughs may involve multiple attributes at once.

Examples:
- Vessel + Integrity body remolding;
- Reason + Insight revelation;
- Glamour + Animus command aura;
- Vessel + Animus cultivation body;
- Integrity + Pneuma-adjacent cosmic survival event, with Pneuma still hidden.

D04 owns whether this is allowed and how it is gated.

## Acceptance criteria

A breakthrough-lattice ruling is valid when it:

1. preserves independent visible attribute lattices;
2. excludes Pneuma from ordinary breakthrough;
3. records history and altered state;
4. routes advancement procedure to D04;
5. routes harm/transformation/world consequence to D07/D08/D10;
6. prevents one donor level system from replacing Astra thresholds.

# D01-04 Pool, Reserve, and Derived Pressure Handoffs

## Purpose

This file records provisional derived-pool and reserve relationships while preserving owner boundaries.

## Core rule

D01 may identify which attributes can inform derived pools or reserves. D01 does not define final formulas.

D03 owns Power Economy pools, carriers, recharge, overdraw, costs, reserves, and resource behavior. D07 owns injury and harm. D08 owns actor body-state. D02 owns resolution usage.

## Provisional derived pressure map

| Attribute | Possible derived pressure |
|---|---|
| Vessel | physical capacity, load, body exertion, form-bearing capacity. |
| Finesse | tempo, initiative pressure, precision reserve, movement control. |
| Reason | focus pool, analysis reserve, planning capacity. |
| Insight | assessment capacity, perception pressure, hidden-state reading. |
| Integrity | vitality threshold, injury absorption, recovery pressure, identity stability. |
| Glamour | influence pool, command reach, social projection pressure. |
| Animus | power pool, spiritual reserve, corruption resistance signal, sustained focus. |

These are handoff signals, not formulas.

## Health/power-economy relevance

D01 must not detach attributes from health or power economy when relevant.

Vessel, Integrity, and Animus may matter to survivability, resource carrying, corruption resistance, or power expression, but only through owner files.

Finesse, Reason, Insight, and Glamour may also affect reserves or access, but no derived pool is automatic.

## Derived-stat risks

D01 blocks:

- donor HP formula import;
- donor saving throw import;
- donor ability modifier import;
- donor armor class import;
- donor spell slot import;
- donor sanity import;
- donor initiative formula import;
- donor point-buy import;
- donor skill list import.

## Handoffs

|---|---|

| Skill/assessment/research methods | D05 |

| Vitality, injury, corruption, backlash, conditions | D07 |
| Body form, kinform, transformations, actor state | D08 |
| Social consequence of visible attributes | D10 |

## Acceptance criteria

A derived pressure ruling is valid when it:

1. treats attribute-to-pool relationships as provisional signals;
2. routes formulas to D03/D07/other owners;
3. preserves Pneuma as no-pool hidden axis;
4. avoids donor derived-stat leakage;
5. preserves health/power relevance without solving it inside D01.

# D01-05 Conversion Mapping and Source-local Boundaries

## Purpose

This file defines how donor attributes and attribute-adjacent systems map into D01 without importing donor math.

## Standard donor mapping signals

Common donor mappings:

| Donor pressure | Astra candidate |
|---|---|
| Strength / might / physical force | Vessel |
| Dexterity / agility / speed / coordination | Finesse |
| Intelligence / logic / technical thought | Reason |
| Wisdom / perception / intuition | Insight |
| Constitution / endurance / toughness | Integrity |
| Charisma / presence / charm / command | Glamour |
| Soul / spirit / ki / qi / essence / will as inner force | Animus |
| Luck / fate / karma / fortune | Not automatically Pneuma |

These are mapping candidates, not automatic imports.

## Luck/fate handling

Donor luck and fate systems must be routed by function.

Possible outcomes:
- D03 spendable resource;
- D06 fate/fortune Technique or Route feature;
- D10 public myth/reputation/world-state;
- hidden Pneuma pressure only if doctrine-authorized;
- source-local retention;
- quarantine/escalation.

Do not map donor luck to Pneuma by default.

## Non-six-stat donors

Donor systems may use:
- body/mind/soul;
- approaches;
- attributes plus skills;
- profession-only systems;
- narrative tags;
- cultivation roots;
- cyberware capacities;
- sanity/stress axes;
- psionic disciplines;
- vehicle interface stats;
- companion stats;
- source-local traits.

D01 maps by function, not label.

## Source-local retention

Retain donor attribute systems source-locally when:
- donor math is inseparable from source-local procedure;
- the attribute is local genre machinery;
- mapping would erase important subsystem behavior;
- later doctrine is missing;
- donor construct is not meant to generalize.

## Quarantine and escalation

Quarantine if:
- attribute combines actor, resource, route, harm, and world-state in one bundle;
- donor math would overwrite Astra formula boundaries;
- fate/luck/cosmic pressure threatens Pneuma boundary;
- cyberware/humanity/sanity systems are actually D07/D08/D10 pressure;
- construct cannot be mapped without new doctrine.

Escalate if repeated donor pressure shows a missing Astra framework.

## Acceptance criteria

A conversion mapping is valid when it:

1. maps by function rather than donor label;
2. rejects donor math by default;
3. protects Pneuma;
4. preserves source-local systems when needed;
5. assigns owner handoffs for bundled constructs.

D01 owns:
- attribute meanings;
- visible/hidden axis distinction;
- Animus/Pneuma boundary;
- attribute conversion candidates;
- breakthrough lattice substrate;
- derived pressure handoff signals.

D01 does not own:
- final derived-stat formulas;
- Power Economy pools;
- advancement gate procedure;
- skill lists or methods;
- route powers;

5. Does it import donor math?
6. Does it require D03, D04, D05, D06, D07, D08, D09, or D10 handoff?
