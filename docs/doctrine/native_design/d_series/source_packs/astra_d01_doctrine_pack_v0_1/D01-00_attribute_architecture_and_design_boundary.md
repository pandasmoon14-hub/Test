# D01-00 Attribute Architecture and Design Boundary

## Purpose

This file defines the purpose and boundary of Astra’s attribute architecture.

## Core rule

Attributes are durable character axes used to interpret capability, pressure, breakthrough, conversion, and derived-system handoffs. They are not donor ability scores imported with donor assumptions.

## Attribute accessibility

D01 deliberately uses familiar functional territory so players and converters can orient quickly, but donor labels do not import donor math.

Astra may map donor STR/DEX/INT/WIS/CON/CHA-like constructs into D01 axes, but donor saving throws, skill lists, class scaling, modifiers, point-buy assumptions, proficiency math, and derived-stat formulas are not imported.

## Visible and hidden axes

D01 uses:

- seven visible player-facing attributes;
- one hidden backend pressure axis.

Visible attributes are part of character-facing identity and capability.

Pneuma is not visible as a score, pool, stat, spendable currency, or ordinary build axis.

## Attribute vs. skill vs. route

Attributes describe broad capability and pressure.

Skills/methods belong to D05.

Routes, Techniques, and Principles belong to D06.

Advancement and breakthroughs belong to D04, but D01 defines the attribute lattice substrate that D04 can use.

## Corpus-scale requirement

The attribute system must absorb donor families including fantasy ability scores, sci-fi attributes, cultivation body/spirit/mind axes, narrative approaches, professions, cyberware effects, psionic stats, horror sanity-adjacent pressure, vehicle/mech interfaces, companion systems, and source-local advancement tracks.

## Acceptance criteria

A D01 attribute rule is valid when it:

1. preserves the seven visible axes and hidden Pneuma axis;
2. avoids donor math leakage;
3. distinguishes capability from skill/method and route/power;
4. supports multi-axis breakthroughs;
5. routes derived pools and consequences to owner files.
