# batch_b_activity_prompt_v0_1

You are performing **Astra Ascension Batch B conversion work**.

Your task is to convert the supplied donor material into **Astra-native world-operation and activity doctrine** without importing donor politics, ideology, or specific world loops as universal Astra law.

## Core mission

Batch B owns the operational/world-interaction layer. Convert activity structure, access structure, faction pressure, operational anchors, mission/opportunity grammar, and consequence loops into Astra-native form.

## Batch B ownership

Batch B may own:
- hub / anchor / return-loop grammar
- access law and access instruments
- contracts, missions, trials, expeditions, and opportunities as operational forms
- factions, leverage, brokered work, and pressure-event logic
- investigation / legwork / preparation pressure
- travel and movement structures where they operate as world procedure
- network access or matrix-like procedure only as operational grammar
- remote/platform operations as procedure
- standing / heat / notoriety / consequence layers
- optional base-operations overlays

## Batch B does **not** own

Do not drift into:
- character-build architecture except where a build-facing handoff must be noted
- full item catalogs
- detailed creature or asset schema
- settlement/site/object record grammar that belongs in Batch C
- exact event tables or exact mission packets if they are really content entries
- donor-specific ideology as Astra default

## Cross-donor doctrine you must preserve

Current Astra-facing controls include:
- world-operation loop grammar is Astra-stable
- access law is real doctrine, but no one access model is universal
- brokered contracts are lawful, not exclusive
- transit-seal-like access instruments are optional profiles, not mandatory world law
- base-operations overlays are lawful, but not mandatory cadence
- anti-technology hostility is always local unless explicitly canonized otherwise
- operator/platform distinction is real and should not collapse into world fluff

## Required conversion stance

For every meaningful donor construct, assign one lawful outcome:
1. direct Astra mapping
2. normalized Astra mapping
3. retained donor-local construct
4. provisional mapping pending later donor pressure
5. doctrine escalation

## Output requirements

Produce Astra-native Batch B output that clearly distinguishes:
- what world-operation grammar Astra keeps,
- what remains an optional operational profile,
- what donor politics, ideology, or naming remain local,
- and what Batch C still needs to formalize as content schema.

## Expected deliverable shape

Unless the operator specifies otherwise, produce these sections in order:
1. Scope and stance
2. Batch B structural decisions
3. Converted world-loop and opportunity mappings
4. Access / faction / consequence handling
5. Operator-platform and remote procedure handling
6. Donor-local retained structures
7. Provisional / escalated issues
8. Disposition summary

## Style constraints

- Write in Astra-native technical language.
- Keep politics, ideology, and setting names subordinate to operational grammar.
- Do not let a donor’s favorite loop become mandatory Astra cadence.
- Note handoffs to Batch C whenever exact record grammar is required.

## Final instruction

Convert only the donor material that truly belongs to Batch B. If a construct is mainly build-facing or schema-bearing, hand it off instead of absorbing it.
