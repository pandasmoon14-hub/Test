# batch_a_chassis_prompt_strict_v0_1

You are performing **strict Astra Ascension Batch A conversion**.

This is a high-discipline conversion pass. Favor correctness, boundaries, and explicit unresolved status over completeness theater.

## Non-negotiable rules

- Stay entirely inside Batch A ownership.
- Do not universalize donor assumptions.
- Do not invent missing Astra law.
- Do not smooth over contradictions with vague prose.
- Do not silently rename donor constructs unless the Astra mapping is explicit.
- If a construct is not safely promotable, mark it donor-local or provisional.
- If the donor pressures a missing framework, escalate instead of improvising.

## Batch A ownership

Own only:
- character creation and layered build structure
- attributes, derived values, skill/rating architecture
- path / archetype / profession / package mapping
- techniques / powers / active expression mapping
- access channels, permissions, ratings, and build-facing reserves
- advancement structure and build-facing progression pressure
- contacts/resources/build capital when they are part of chargen or chassis identity

## Forbidden drift

Do not absorb:
- world loop procedure
- mission, legwork, or infiltration structure
- scene composition or creature records
- full item catalogs
- platform operation procedure
- settlement / stronghold cadence
- reward tables unless they directly constrain Batch A build structure

## Cross-donor controls

Current promoted or accepted Astra-facing controls include:
- layered build grammar
- access law as real doctrine
- optional profiles are allowed, but do not become baseline automatically
- exact donor slot models, exact priority tables, exact ideology, and exact donor reserve semantics are not baseline by default
- competence is not identical to codified active expression

## Required decision model

For each meaningful donor construct, explicitly label one of:
- Baseline Astra mapping
- Optional Astra profile
- Donor-local retained structure
- Provisional unresolved mapping
- Doctrine escalation

## Required output format

Use exactly these headings:
1. Scope
2. Accepted Astra mappings
3. Optional Astra profiles
4. Donor-local retained structures
5. Provisional unresolved mappings
6. Doctrine escalations
7. Batch B / C handoffs
8. Final disposition

Under each heading, be concrete and concise. Prefer bulletless prose blocks or short labeled subsections.

## Strict handling instructions

- If donor chargen is a unique package system, keep the package math donor-local unless Astra has explicitly promoted a broader grammar.
- If donor special currencies resemble Fate, Edge, Essence, Resonance, or similar systems, do not merge them by vibe alone.
- If donor techniques/powers do not fit Astra Techniques cleanly, preserve the distinction instead of forcing a false match.
- If contacts/resources are materially part of the build layer, keep them in Batch A with a clear ownership note.

## Final instruction

Produce a strict Batch A conversion document that can survive later cross-donor comparison without requiring the reader to guess what was promoted, what was retained, and what remains unresolved.
