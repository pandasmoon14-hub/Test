# Three-Donor Promotions v0.1

Purpose: authoritative cross-donor promotion ledger after comparison of:
- RHBF
- Shadowrun
- Lancer

This file records what is now strong enough to promote into Astra-facing consolidation, what should remain as optional recurring profiles, and what should still not be promoted.

---

## Promote now as Astra baseline grammar

### Batch A

1. **Layered build grammar**
   Astra must support multiple lawful build owners rather than one sovereign package layer.

2. **Path as one lawful build owner, not the sole owner**
   Paths remain valid and important, but later donors prove that access states, overlays, licenses, operator/platform splits, and resource-backed identity layers must coexist.

3. **Access law as real doctrine**
   Domain access, license-backed access, special channels, permissions, and gated acquisition all validate access as a distinct and durable Astra rules layer.

4. **Capability doctrine distinct from codified active expression**
   Competence, triggers, skill phrases, and broad capability layers must remain distinct from Techniques or equivalent codified active objects.

5. **Ranked specialization overlays**
   Ranked overlays are now stable grammar.

6. **Contacts/resources as lawful build pressure**
   Social and material starting pressure can be part of a build without collapsing into later world-procedure doctrine.

7. **Operator / platform separation**
   This is now clearly baseline Astra grammar.

### Batch B

8. **World-operation loop grammar**
   Astra must support structured operation cycles without universalizing one donor cadence.

9. **Contract / opportunity structures**
   Multiple donors now prove that opportunity structures are a stable owner, regardless of whether they appear as faction jobs, brokered work, missions, or other operation hooks.

10. **Preparation / legwork procedure**
    Preparation is a real procedure family, not filler.

11. **Faction pressure and leverage logic**
    Factions and institutions are stable procedural pressure owners.

12. **Conditional / technical access procedure**
    Access may depend on institutional, technical, situational, or authorization-driven conditions.

13. **Remote / delegated operation procedure**
    Astra should preserve operator action routed through delegated assets or platforms.

14. **Standing / scrutiny separation**
    Consequence layers should not collapse into one undifferentiated reputation number.

15. **Tactical shell doctrine**
    Structured tactical objective procedure is now stable grammar.

16. **Downtime as structured bridge**
    Between-operation procedure is a real procedural layer when donor pressure supports it.

### Batch C

17. **Classification/category/family as mechanical field**
    Classification must do real work in Astra.

18. **Multiple content ecologies**
    Astra must support organic, manufactured, licensed, bonded, salvage, and mixed ecologies.

19. **Broad resource-source ontology**
    Resource sources are broader than slain actors.

20. **Platform record grammar**
    Platform-side records are now baseline doctrine.

21. **Control compatibility / operation handoff**
    Batch C must preserve compatibility fields while leaving procedure in Batch B.

22. **Host-cost profile for installed modification**
    Installed-modification cost/support pressure is now baseline, though exact implementations remain open.

23. **Deployable / support-entity middle zone**
    Astra should preserve an intermediate zone between ordinary gear, subsystems, and full actors.

24. **Bound-growth object grammar**
    Bound-growth objects remain Astra-stable.

25. **Natural Treasure as progression-significant class**
    Natural Treasure remains a valid Astra-facing resource class.
