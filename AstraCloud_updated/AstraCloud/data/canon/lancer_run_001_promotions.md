# Lancer Run 001 Promotions

Run ID: `lancer_prod_run_001`  
Purpose: `Authoritative promotion ledger derived from the revised Lancer run after merge review and integration.`

This file records what Lancer pressures Astra to promote, what should remain optional-profile support, and what should not be promoted at this stage.

---

## Baseline grammar

### 1. Operator / platform separation
Astra should explicitly preserve separate ownership for:
- operator record,
- platform record,
- and operation procedure.

Lancer is the strongest donor proof so far that these layers must not collapse.

### 2. Linked-but-non-collapsed dual build
Astra should support donor families where a player-grade actor is assembled through:
- a personal or operator-side build,
- a platform-side build,
- and a progression bridge that advances both without merging them into one object.

This is now a promoted grammar family.

### 3. Access licensing as real progression law
Lancer strongly validates that access to advanced packages, chassis, systems, or equivalent capability bands may be governed by explicit authorization / licensing progression rather than only by class package or purchase.

Astra should preserve licensing / authorization-backed access as lawful grammar.

### 4. Triggers as competence-facing doctrine
Lancer strongly validates competence phrases / trigger-style capability expressions as a lawful owner inside Astra.

This reinforces that not all high-impact play expression belongs in Technique space.

### 5. Ranked specialization overlays
Talents strongly validate a distinct ranked specialization layer that sits above ordinary competence and below donor-local class monocultures.

Astra should preserve ranked specialization overlays as a lawful build owner.

### 6. Tactical shell doctrine
Lancer strongly validates tactical-shell structure as a real procedural family for operation scenes.
The reusable grammar is:
- declared tactical objective,
- structured pressure field,
- role-sensitive spatial stakes,
- and formal win / loss / extraction conditions.

### 7. Downtime as structured operational bridge
Downtime is promoted as a lawful operational bridge between major engagements, not mere narration filler.

Astra should preserve meaningful between-operation procedure where donor structure supports it.

### 8. Platform chassis identity
Astra should preserve chassis / frame identity as a real schema owner for platform-side records.

The platform is not merely a weapon list or armor value holder.

### 9. Mount capacity distinct from install capacity
Lancer strongly validates that weapon-bearing capacity and system / install-bearing capacity are not the same field and should not be collapsed by default.

Astra should preserve mount-family grammar separately from install-capacity grammar.

### 10. Deployable / support-entity middle zone
Astra should preserve a lawful middle zone between:
- ordinary gear,
- platform subsystems,
- and full independent actors.

Deployables, drones, and similar support entities validate this strongly.

### 11. Hostile class-plus-template platform grammar
Lancer strongly validates modular hostile platform construction where a hostile chassis/class owner can be modified by templates, roles, or loadout overlays.

Astra should preserve this modular hostile-entity grammar.

## Optional profiles

### 1. Dual-build assembly as a recurring profile
The linked pilot/mech build is strongly supported, but should remain one major lawful profile rather than universal player construction law.

### 2. Authorization-backed chassis access
Licensing is strong enough to retain as an optional recurring profile family even when exact donor rank math is not promoted.

### 3. Platform-tuning matrices
HULL / AGILITY / SYSTEMS / ENGINEERING style platform-tuning matrices are useful and coherent, but remain optional profiles rather than core Astra attributes.

### 4. Mission-style operation cadence
Briefing -> preparation -> deployment -> objective execution -> debrief -> downtime is a strong recurring loop, but not universal Astra cadence.

### 5. Structure / stress durability profiles
These are strong donor durability expressions, but remain optional profile support rather than promoted baseline durability law.

### 6. Platform intelligence as optional profile
AI / NHP-like intelligent-platform logic is lawful as an optional profile, but not promoted as baseline Astra assumption.

### 7. Repair-capacity and reserve declarations
These are strong recurring procedural profiles, but not yet universal Astra law.

## Explicit refusals

### 1. Do not let platform-tuning matrices overwrite Astra core attributes
Lancer’s platform-side tuning matrix is useful donor logic, not a replacement for Astra’s core attribute architecture.

### 2. Do not collapse triggers into Techniques
Trigger-style competence must remain competence-facing unless a donor clearly produces codified active objects.

### 3. Do not collapse mounts and installs into one generic slot family
They are distinct capacity owners.

### 4. Do not let exact mission cadence become mandatory Astra structure
The mission loop is strong, but must remain one lawful cadence among several.

### 5. Do not universalize intelligent-platform metaphysics
AI / NHP logic is valuable donor pressure, but not baseline Astra cosmology.
