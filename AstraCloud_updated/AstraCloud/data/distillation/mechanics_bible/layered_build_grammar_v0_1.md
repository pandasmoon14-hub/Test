# Astra Layered Build Grammar v0.1

Derived from cross-donor consolidation after RHBF, Shadowrun, and Lancer.

## Scope of this file

This file owns Astra's **layered build grammar**.

It defines:
- what a build layer is,
- which layer families Astra currently recognizes,
- what each layer may and may not own,
- how multiple layers coexist without collapsing into one donor-shaped package chain,
- how linked builds are handled,
- where access, specialization, resources, contacts, bonded objects, augmentations, and platform-side structures belong,
- and what remains donor-local or optional-profile support.

It does **not** own:
- final Path rosters,
- exact attribute lists,
- exact donor priority tables,
- exact license charts,
- exact reserve-economy math,
- exact frame/platform numbers,
- exact donor metaphysics,
- or full object-side schema for platforms, relics, augmentations, or credential assets.

This is a grammar owner, not a catalog owner.

---

## Why this file exists

The first three donors establish that Astra can no longer behave as though one sovereign package chain explains all player construction.

RHBF proves that Astra must support:
- Path-facing growth,
- domain-gated access,
- codified active expression,
- and non-exclusive development.

Shadowrun proves that Astra must also support:
- stacked package layers,
- resource and contact pressure,
- access states,
- augmentation pressure,
- and capability doctrine that does not reduce to Technique logic.

Lancer proves that Astra must further support:
- operator/platform separation,
- linked-but-non-collapsed dual builds,
- licensing as real access law,
- trigger-style competence phrases,
- and ranked specialization overlays.

The stable conclusion is:

**Astra builds are layered.**  
No single donor layer becomes universal baseline law.

---

## Core build law

A lawful Astra build may contain multiple simultaneous layers.

Astra does not assume:
- one mandatory donor class chain,
- one mandatory Path-only architecture,
- one universal priority ladder,
- one universal license table,
- one universal operator/platform split,
- or one universal reserve/resource package.

Instead, Astra recognizes build layers as distinct owners that may coexist.

The current cross-donor-stable rule is:

> A build layer may shape identity, access, growth, capability, or object attachment without automatically absorbing the responsibilities of the other layers.

---

## Recognized layer families

These are the current Astra-facing layer families.

### 1. Body / origin layer
This layer owns foundational embodied facts such as:
- biology, kinform, ancestry, lineage, substrate, or comparable donor-local embodiment,
- body-facing constraints or potentials,
- any stable inherited pressures that belong to the actor rather than to training or role.

This layer should **not** automatically own:
- full role identity,
- access permissions,
- social standing,
- licenses,
- or codified active expression catalogs.

---

### 2. Culture / background layer
This layer owns:
- upbringing,
- socialization,
- institutional starting context,
- background-facing training pressure,
- starting affiliations,
- and similar pre-current-life structuring context.

It may influence:
- competencies,
- social capital,
- starting resources,
- ordinary permissions,
- or introductory contacts.

It should **not** absorb:
- all role doctrine,
- all access law,
- or all progression.

---

### 3. Role / Path / profession layer
This layer owns:
- major role-facing identity,
- opening doctrine of growth,
- profession/package logic,
- and coherent starting direction.

It may influence:
- Attribute priorities,
- access tendencies,
- technique or equivalent active-expression entry,
- defensive tendencies,
- progression hooks,
- and identity-facing role expectations.

It does **not** automatically own:
- all meaningful differentiation,
- all capabilities,
- all access states,
- all social capital,
- all special ratings,
- or all platform-side build facts.

---

### 4. Capability / competence layer
This layer owns:
- broad trained capabilities,
- competencies,
- trigger phrases,
- skill groups,
- ordinary proficiencies,
- and general performance-facing capability statements.

This layer is distinct from codified active expression.

The key rule is:

> Capability is not the same thing as a Technique, power, spell, maneuver, exploit, or other codified active object.

This distinction is now non-negotiable after Shadowrun and Lancer.

---

### 5. Codified active-expression layer
This layer owns:
- Techniques,
- powers,
- spells,
- maneuvers,
- explicitly defined active workings,
- and other object-like active expressions with discrete activation and effect structure.

This layer may be strongly present in some donors and weakly present in others.

It should **not** swallow the entire capability layer.

---

### 6. Access / authorization layer
This layer owns:
- Domains,
- licenses,
- permissions,
- ratings tied to lawful usage,
- restricted access states,
- credential-backed authorization,
- and comparable gate structures.

This layer exists because multiple donors prove that access law is real doctrine.

This layer should remain distinct from:
- raw competence,
- raw resources,
- and role identity.

---

### 7. Specialization overlay layer
This layer owns:
- ranked specialization overlays,
- talent-style tiers,
- focused growth tracks,
- and narrow but meaningful build differentiation above general competence.

This layer is neither ordinary skill nor full Path doctrine.

It may coexist with:
- Paths,
- capabilities,
- licenses,
- and operator/platform structures.

---

### 8. Social-capital / resource layer
This layer owns:
- contacts,
- starting networks,
- sponsorship,
- resource-backed entry pressure,
- and comparable social/material setup that materially shapes a build before ordinary play begins.

This layer is lawful and should not be forced entirely into later Batch B procedure.

It does **not** replace world-facing faction procedure, but it does shape the starting build.

---

### 9. Bonded / attached-object layer
This layer owns:
- bonded relics,
- attuned implements,
- growth-linked objects,
- companion-linked objects,
- and similar attached-object structures that materially shape identity or growth.

This layer should remain distinct from:
- augmentations,
- ordinary gear,
- and full platform-side builds.

---

### 10. Installed / augmentation layer
This layer owns:
- augmentations,
- installable modifications,
- host-cost-bearing body changes,
- and similar installed identity or capability changes.

This layer is lawful and should not be collapsed into Path, skill, or ordinary equipment doctrine.

---

### 11. Operator / platform layer
This layer exists when a donor supports linked but non-collapsed builds where:
- one side is the operator or personal actor,
- one side is the platform, vehicle, frame, mech, drone-host, or equivalent externalized build object,
- and progression or authorization may bridge them without collapsing them into one record.

Lancer makes this layer unavoidable as a stable Astra-hosted family.

This layer should remain distinct from:
- role/Path doctrine,
- ordinary equipment,
- and world-operation procedure.

---

## Ownership boundaries

Astra build layers must not silently absorb one another.

The most important boundary rules are:

1. **Path is one lawful build owner, not the sovereign owner of all differentiation.**
2. **Capability is not the same as codified active expression.**
3. **Access law is real and distinct from competence.**
4. **Specialization overlays remain distinct from both Path and ordinary skill.**
5. **Contacts/resources can be real build pressure without becoming the entire social system.**
6. **Bonded-object logic must not collapse into augmentation logic.**
7. **Installed modification must not collapse into ordinary gear.**
8. **Operator and platform must remain distinct records when the donor requires it.**

---

## Linked builds

Some donors produce **single-record builds**.  
Some produce **stacked builds**.  
Some produce **linked dual builds**.

Astra must support all three.

### Single-record build
One actor record carries nearly all meaningful build identity.

### Stacked build
One actor record contains several distinct layer owners within it:
- background,
- role,
- access state,
- resources,
- competencies,
- and similar pressures.

### Linked dual build
Two linked records coexist:
- operator-side record,
- platform-side record.

The stable Astra rule is:

> When a donor produces linked dual builds, Astra preserves both records and the bridge between them rather than collapsing them into one simplified package.

This is now cross-donor-stable because Shadowrun pressures remote/platform operation and Lancer makes operator/platform doctrine explicit.

## Layer interaction rules

### Paths with overlays
A Path may coexist with ranked specialization overlays without being replaced by them.

### Paths with access law
A Path may influence access, but access remains a distinct owner.

### Capabilities with Techniques
A character may have broad capabilities without a large codified active-expression catalog.
A character may also have a codified active-expression layer without those objects replacing the capability layer.

### Resources with identity
Resources, contacts, and sponsorship can materially shape identity and opening play pressure without becoming the sole owner of character doctrine.

### Bonded objects with Paths
A bonded-object layer may strongly affect a character without becoming identical to their Path.

### Installed changes with body/origin
Installed modifications may change a character materially, but they should not retroactively overwrite body/origin doctrine.

### Operator/platform with Paths
A Path may shape an operator.
A donor-local or Astra-facing platform record may shape the platform.
Neither should erase the other.

## What remains optional-profile support

The following are lawful recurring profiles, but not baseline universal law:

- tri-axis advancement / breakthrough gating,
- linked dual-build assembly as a recurring donor profile,
- priority / sacrifice ladders,
- heavy resource-first starting identity,
- special access ratings distinct from primary attributes,
- platform-tuning matrices,
- explicit access instruments,
- mission-style cadence as the only operation loop,
- structure/stress durability profiles,
- platform intelligence profiles.

This file should acknowledge them without presenting them as mandatory Astra build law.

## What remains donor-local

The following remain outside live Astra baseline build grammar:

### RHBF donor-local
- exact `1/4/1/3` technique topology,
- exact body-slot / tattoo logic,
- exact reserve-triplet semantics,
- exact cultivation-shaped growth assumptions.

### Shadowrun donor-local
- exact priority A-E ladders,
- exact Edge menu,
- exact technomancer resonance terminology,
- exact Essence decimal implementation,
- exact permit/SIN/legal stack semantics.

### Lancer donor-local
- exact LL progression math,
- exact HULL / AGILITY / SYSTEMS / ENGINEERING values,
- exact frame and manufacturer tables,
- exact sitrep names as build-law defaults.

## Current Astra-facing summary

A lawful Astra build may therefore include, in whole or in part:

- body/origin structure,
- culture/background structure,
- Path/profession/role structure,
- capability/competence structure,
- codified active-expression structure,
- access/authorization structure,
- specialization overlay structure,
- contacts/resources/social-capital structure,
- bonded-object structure,
- installed-modification structure,
- and, where required, linked operator/platform structure.

This is the current stable layered build law after RHBF, Shadowrun, and Lancer.

The final rule is simple:

> Astra must host multiple lawful build grammars without forcing them into one donor-shaped monoculture.
